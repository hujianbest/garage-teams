# MDC 任务拆解

创建任务计划，把已批准设计转化为可执行工作。

## 硬性门禁

在任务规划完成并准备好评审之前，不得开始实现。

如果当前请求尚未经过 `mdc-workflow-starter` 的阶段判断，不直接开始任务规划；应先回到 `mdc-workflow-starter` 完成路由。

## 前置条件

仅在以下条件满足时使用本 skill：

- 需求规格已批准
- 设计已批准

如果设计仍是草稿或还在修订中，请回到 `mdc-design-review` 或 `mdc-design`。

## 工件与状态要求

默认情况下，任务计划应保存到：

- `docs/tasks/YYYY-MM-DD-<topic>-tasks.md`

若 `AGENTS.md` 为当前项目声明了任务计划路径映射，优先使用映射路径；否则使用默认路径。

在任务计划交评审前，文档中至少应体现：

- 状态字段，例如 `状态: 草稿`
- 主题或范围标识
- 当前活跃任务选择规则
- 可供评审定位的章节结构

若项目尚未形成固定进度记录格式，默认使用：

- `skills/mdc-workflow/templates/task-progress-template.md`

任务计划交给 `mdc-tasks-review` 后，还应同步：

- `task-progress.md` 中的 Current Stage
- `task-progress.md` 中的 Current Active Task
- `task-progress.md` 中的 Next Skill

## 目标

任务计划必须让实现过程变得可预期。

它应明确以下内容：

- 里程碑
- 有序任务
- 依赖关系
- 任务级完成条件
- 验证预期

## 红旗信号

出现以下想法时，先停下。这通常说明你正在为“跳过有效任务拆解”找理由：

| 想法 | 实际要求 |
|---|---|
| “设计差不多了，先开始实现再补任务” | 任务计划未准备好评审前，不进入实现。 |
| “先列几个大任务，后面再细化” | 任务必须小到可以明确执行和验证。 |
| “依赖关系实现时自然会处理” | 依赖和顺序必须提前写清。 |
| “完成条件先不写，做完就知道了” | 每个任务都要有明确完成定义和验证方式。 |
| “把整块功能放成一个任务更省事” | 大到无法独立验证的任务不能进入计划。 |
| “验证放到功能全部做完再一起测” | 验证预期必须体现在任务级，而不是拖到最后。 |

## 工作流

### 1. 阅读已批准输入

阅读：

- 已批准需求规格
- 已批准设计
- 当前项目上下文（如相关）

优先读取：

- `AGENTS.md` 中与任务计划相关的路径映射、模板覆盖和任务拆解约束
- `task-progress.md` 中的当前阶段记录（如果存在）

提取：

- 主要工作流
- 依赖与顺序
- 测试影响
- 风险或不确定区域

### 2. 定义里程碑

把工作分组为能产生明确阶段性成果的里程碑。

每个里程碑应包含：

- 目标
- 包含的任务
- 退出标准

### 3. 拆解为任务

任务必须足够小，能够被明确执行和验证。

优先使用这类任务形态：

- 编写失败测试
- 运行并确认失败
- 实现最小行为
- 运行并确认通过
- 重构并重新运行
- 更新状态与记录

避免使用这类模糊任务项：

- 实现某模块
- 完成功能
- 后面再优化

### 4. 记录依赖与完成条件

对每个任务明确：

- 前置条件
- 会触碰哪些工件
- 验证方式
- 完成条件

并至少为计划补一条规则：

- 如何从任务计划中选定唯一 `Current Active Task`

### 5. 编写任务计划

若 `AGENTS.md` 为当前项目声明了任务计划模板、章节骨架或当前活跃任务规则，优先遵循这些约定。

若 `AGENTS.md` 未提供任务计划模板覆盖，则使用以下最小结构：

```markdown
# <主题> 任务计划

## 概述

## 里程碑

## 任务拆解

## 依赖关系

## 完成定义

## 验证策略

## 风险与顺序说明

## 当前活跃任务选择规则
```

### 6. 交给评审

任务计划准备好后，交给 `mdc-tasks-review`。

若项目尚未形成固定 review 记录格式，默认由 `mdc-tasks-review` 使用：

- `skills/mdc-workflow/templates/review-record-template.md`

交接前请同步：

- 将任务计划保存到约定路径
- 在 `task-progress.md` 中写入 Current Stage
- 在 `task-progress.md` 中写入 Current Active Task
- 在 `task-progress.md` 中将 Next Skill 更新为 `mdc-tasks-review`

输出应使用：

```markdown
任务计划草稿已起草完成，请提交 `mdc-tasks-review` 进行评审。

下一步 skill: `mdc-tasks-review`
```

## 反模式

- 把任务计划写成设计文档副本
- 使用大到无法验证的任务
- 漏掉依赖关系
- 漏掉完成条件
- 把验证拖到整个功能结束时才做

## 完成条件

只有在存在一份可评审任务计划，并且已准备好交给 `mdc-tasks-review` 时，这个 skill 才算完成。
