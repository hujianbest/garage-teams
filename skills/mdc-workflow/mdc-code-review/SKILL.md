# MDC 代码评审

评审当前任务的实现代码。

## 目的

这个 skill 用于在测试质量完成评审后，判断实现本身是否可靠。

它不能替代需求评审或设计评审。

## 适用时机

优先用于以下场景：

- 当前任务已经通过测试评审，准备进入实现质量判断
- 当前改动涉及核心逻辑、错误处理或设计约束
- 需要决定当前实现是否可以进入追溯性评审

## 输入

阅读以下最少必要信息：

- 当前任务对应的规格、设计和任务片段
- 当前任务实现改动与相关测试
- 已有测试评审结论

## 记录要求

评审完成后，默认将本次结论写入：

- `docs/reviews/code-review-<task>.md`

如 `AGENTS.md` 为当前项目声明了等价路径，按其中映射路径保存。

若项目尚未形成固定 review 记录格式，默认使用：

- `skills/mdc-workflow/templates/review-record-template.md`

如果结论为 `通过`，应同步更新：

- `task-progress.md` 中的代码评审状态
- `task-progress.md` 的 Next Skill 为 `mdc-traceability-review`

如果结论为 `需修改`，应同步更新：

- 当前任务需要回到实现修订
- `task-progress.md` 的 Next Skill 为 `mdc-implement`

如果结论为 `阻塞`，应同步更新：

- `task-progress.md` 中的代码评审阻塞状态与原因
- `task-progress.md` 的 Next Skill 暂时保持为 `mdc-code-review`

若项目尚未形成固定进度记录格式，默认使用：

- `skills/mdc-workflow/templates/task-progress-template.md`

## 检查清单

### 1. 正确性

- 代码是否看起来满足当前任务目标？
- 是否明显漏掉了边界情况？

### 2. 设计一致性

- 实现是否仍然符合已批准设计？
- 当前任务是否引入了意外的架构漂移？

### 3. 可读性与可维护性

- 命名是否清晰易懂？
- 逻辑是否存在不必要的复杂度？
- 是否有单处承担过多职责的情况？

### 4. 错误处理

- 常见失败路径是否处理得当？
- 实现是否引入了脆弱假设？

## 输出格式

请严格使用以下结构：

```markdown
## 结论

通过 | 需修改 | 阻塞

## 发现项

- [严重级别] 问题

## 代码风险

- 条目

## 下一步

`mdc-traceability-review` | `mdc-implement` | `补齐阻塞条件后重试 mdc-code-review`

## 记录位置

- `docs/reviews/code-review-<task>.md` 或映射路径
```

## 判定规则

只有当当前任务实现足够合理、可以进入下一步验证时，才返回 `通过`。

如果在正确性、可维护性或设计一致性方面仍有问题需要先修正，再返回 `需修改`。

如果缺少关键输入、无法定位实现范围，或代码上下文本身不可读，则返回 `阻塞`。

## 反模式

- 在这里重复需求评审意见
- 只因为测试是绿的就直接通过代码
- 因为任务小就忽略设计漂移

## 完成条件

只有在给出明确结论、记录位置和唯一下一步之后，这个 skill 才算完成。
