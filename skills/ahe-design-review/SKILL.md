# AHE 设计评审

评审设计文档，并判断它是否可以成为任务规划阶段的候选已批准输入。

在完整 AHE workflow 中，它用于设计完成后的质量门禁，而不是普通意见收集。

## 硬性门禁

在设计通过评审并经过真人确认之前，不得进入 `ahe-tasks`。

如果当前请求尚未经过 `ahe-workflow-starter` 的阶段判断，不直接开始设计评审；应先回到 `ahe-workflow-starter` 完成路由。

## 目的

这个 skill 用来防止过早拆解任务。

如果设计不清晰，任务规划就会变成猜测，或者过早下沉到实现细节。

高质量设计评审不只是判断“文档看起来完整”，而是判断：

- 设计是否真正锚定已批准规格
- 关键设计决策是否站得住
- 接口、边界和风险是否已经清楚到足以进入任务规划

## 评审范围

请基于以下内容评审设计：

- 已批准的需求规格
- 被评审的设计文档
- 主要约束与非功能需求
- 项目现有技术上下文（如果已知）
- `AGENTS.md` 中与设计评审、路径映射和状态命名有关的约定

评审时不要开始写任务或代码。

如果缺少已批准规格、设计文档状态不清，或批准证据彼此冲突，不要靠聊天记录补脑；应按证据不足处理，并返回 `阻塞` 或要求回到上游重新厘清。

## 高质量评审基线

交付给真人确认前，高质量评审结果至少应做到：

- 明确指出设计是否可追溯到已批准规格
- 明确指出关键风险、隐藏假设和任务规划阻塞点
- 区分“可修补的不完整”与“不能继续规划的阻塞问题”
- 不把评审退化成设计重写、任务拆解或编码建议
- 给出唯一下一步，并保持与 `ahe-workflow-starter` 的暂停点契约一致

## 真人确认规则

`ahe-design-review` 给出的“通过”，只表示设计已经达到可确认状态，不等于自动批准。

只有结论为 `通过` 时，才进入真人确认。

本 skill 默认运行在独立 reviewer subagent 中。若在该模式下运行，reviewer 只负责评审、落盘和回传结构化摘要；真正向真人展示结论并发起批准确认的动作，仍由父会话负责。

如果结论为 `需修改` 或 `阻塞`，不要发起真人批准确认；应基于评审结果直接回到 `ahe-design`，与用户确认必要修订后继续完善设计。

在进入 `ahe-tasks` 之前，必须完成以下动作：

1. 向真人展示评审结论、主要设计风险和需要重点确认的决策
2. 邀请真人确认“该设计是否批准进入任务规划阶段”
3. 如果真人提出修改意见，继续和真人对话，回到设计内容上迭代修订
4. 只有在真人明确确认通过后，设计才算已批准

## Reviewer Subagent 运行约定

当本 skill 由 reviewer subagent 执行时：

- reviewer 负责读取设计与最小必要辅助上下文
- reviewer 负责把评审记录写入约定路径
- reviewer 负责回传结构化摘要
- reviewer 不负责代替父会话推进主链
- reviewer 不负责代替父会话发起真人确认

## 记录要求

评审完成后，必须将本次结论写入：

- `docs/reviews/design-review-<topic>.md`

如 `AGENTS.md` 为当前项目声明了等价路径，按其中映射路径保存。

若项目尚未形成固定 review 记录格式，默认使用：

- `templates/review-record-template.md`

如果使用通用模板中的英文结论字段，请按以下方式映射：

- `通过` -> `pass`
- `需修改` -> `revise`
- `阻塞` -> `blocked`

如果当前任务正在使用 `task-progress.md` 驱动 workflow，且真人确认通过，还应同步更新：

- 设计文档中的状态字段
- `task-progress.md` 中的当前阶段与推荐下一步动作或 skill

若项目尚未形成固定进度记录格式，默认使用：

- `templates/task-progress-template.md`

## 评审方法

### 1. 先建立证据基线

在给出结论前，先读取并固定以下证据来源：

- 已批准规格
- 当前设计文档
- `AGENTS.md` 中的路径、状态词和模板约定
- `task-progress.md` 中的 Current Stage 与 Workflow Profile（如果存在）
- 必要的技术上下文

不要只根据会话记忆、用户口头确认或零散聊天内容判断“已批准”或“设计已经讲清楚”。

如果 `task-progress.md` 或其他阶段证据显示当前 profile 不是 `full`，或当前阶段并不支持进入 `ahe-design-review`，应回到 `ahe-workflow-starter` 重新编排，而不是继续给出评审结论。

### 2. 按多维标尺校准评审深度

在形成最终结论前，至少对以下维度做内部评分，评分范围 `0-10`：

- 需求覆盖与追溯
- 架构一致性
- 决策质量
- 接口与任务规划准备度
- 测试准备度与风险可控性

评分不是最终结论本身，但它能帮助区分：

- 哪些只是轻微缺口
- 哪些已经足以拉低到 `需修改`
- 哪些会让任务规划建立在错误前提上，从而升级到 `阻塞`

这些分数默认只用于内部校准严重程度，不强制写入通用 review 记录。

判定辅助规则：

- 任一关键维度低于 `6/10` 时，不得返回 `通过`
- 任一维度低于 `8/10` 时，通常至少应对应一条具体发现项或薄弱点

### 3. 用挑战式问题找隐藏风险

正式给 verdict 前，至少主动检查这些问题：

- 哪个需求、行为、接口或数据持久化无法追溯到已批准规格
- 哪个模块或流程看起来像“已设计”，其实仍然依赖实现阶段补脑
- 哪个假设一旦不成立，当前设计就会失效
- 哪个风险如果现在不指出，最可能在 `ahe-tasks` 或实现阶段爆炸

## 检查清单

### 1. 需求覆盖与追溯

- 设计是否覆盖了规格中的关键需求？
- 主要行为是否映射到了组件、模块、流程或接口？
- 关键需求能否回指到稳定需求条目、需求编号或等价规格锚点？
- 是否存在新增行为、接口、数据持久化或运营流程，却无法追溯到已批准规格？

### 2. 架构一致性

- 职责和边界是否清晰？
- 模块之间的交互是否容易理解？
- 设计是否是连贯架构，而不是组件清单？
- 关键视图、关键交互或关键流程是否已经清楚到足以支撑评审？

### 3. 决策质量

- 是否真的比较了至少两个可行方案？
- 选定方案是否被解释清楚，而不仅仅是提了名字？
- 是否说明了为什么选这个方案，而不是其他方案？
- 是否记录了主要收益、代价、风险与缓解思路？

### 4. 约束与非功能需求适配

- 设计是否体现了已声明约束？
- 是否考虑了非功能需求和集成点？
- 这些要求是否真正影响了设计，而不是只在概述里被提到？

### 5. 接口与任务规划准备度

- 关键契约是否足够明确，可以支撑任务规划？
- 主要数据流和控制流是否清晰？
- 模块边界是否已经稳定到可以继续拆任务？
- 是否还存在会直接破坏任务拆解顺序的设计空洞？

### 6. 测试准备度与隐藏假设

- 设计是否包含设计层面的测试策略？
- 该设计是否可测试，而不依赖隐藏假设？
- 是否指出了需要在后续任务规划或实现阶段优先验证的高风险点？

## 输出格式

请严格使用以下结构：

```markdown
## 结论

通过 | 需修改 | 阻塞

## 发现项

- [严重级别] 问题

## 薄弱或缺失的设计点

- 条目

## 下一步

`等待真人确认` | `回到设计修订`

## 记录位置

- `docs/reviews/design-review-<topic>.md` 或映射路径

## 交接说明

- `等待真人确认`：仅当结论为 `通过`
- `回到设计修订`：用于所有未通过场景；如果问题属于设计内容本身，直接回到 `ahe-design` 修订；如果问题属于需求范围漂移、批准证据冲突、profile / 阶段不匹配或路由错误，则仍回到 `ahe-design`，但第一步必须先经 `ahe-workflow-starter` 重新编排，再决定是否继续设计修订或回流上游
```

## 判定规则

只有在以下条件全部满足时，才返回 `通过`：

- 可以追溯到需求规格
- 没有无法追溯到已批准规格的关键新增设计内容
- 清楚说明了连贯的实现方案
- 关键决策、边界和接口足够清晰，可以进入任务规划
- 覆盖了主要约束、非功能需求和测试影响
- 不存在足以阻塞任务规划的关键设计空洞

以下情况返回 `需修改`：

- 核心设计可用，但还不完整
- 某些模块、流程或接口仍然描述不足
- 存在需求追溯、决策说明或测试准备度方面的明显缺口，但可以通过一轮定向修订补齐

此时下一步通常为 `回到设计修订`。

以下情况返回 `阻塞`：

- 设计无法清晰支撑需求规格
- 存在关键新增内容却无法追溯到已批准规格
- 关键架构决策仍然缺失，或只有结论没有支撑理由
- 存在矛盾、证据冲突或未解决的技术风险，无法继续规划

当 `阻塞` 来源于设计内容本身时，下一步为 `回到设计修订` 并补齐设计缺口。
当 `阻塞` 来源于需求级漂移、批准证据冲突、profile 不匹配或路由错误时，本 skill 的直接返回节点仍为 `ahe-design`；但 `ahe-design` 在继续修订前必须先经 `ahe-workflow-starter` 重新编排，再决定是否继续设计修订或回流上游。

## 批准处理

如果设计通过，请明确说明它已经可以提交给真人确认。

只有在真人明确确认后，它才能成为任务规划阶段的已批准输入。

真人确认完成后，请显式同步：

- 设计文档状态为已批准
- `task-progress.md` 的 Current Stage
- `task-progress.md` 中记录的推荐下一步动作或 skill
- review 记录中的 Human Confirmation 与 Evidence Location

如果未通过，请把它退回设计修订，并附上具体修订点。

退回时请显式做到：

- 不等待真人批准确认
- 把发现项转换为用户可确认的修订行动
- 如果问题属于设计层缺口，让下一轮设计修订直接基于这些修订点继续，而不是从零重新推导完整方案
- 如果问题属于需求范围漂移、批准证据冲突或当前节点路由错误，明确要求先回到 `ahe-workflow-starter` 重新编排，而不是继续在设计评审里硬推进

## 反模式

- 因为“实现时再说”就直接通过
- 把设计评审变成底层编码建议
- 接受只是复述需求、没有设计决策的文档
- 忽略缺失的接口或模块边界
- 忽略无法追溯到已批准规格的新增设计内容
- 设计评审刚返回“通过”就直接进入 `ahe-tasks`
- 不和真人确认，就把设计当成已批准输入

## 回传给父会话

若本 skill 运行在 reviewer subagent 中，至少回传：

- `conclusion`
- `next_action`
- `record_path`
- `key_findings`
- `needs_human_confirmation`：对本 skill 通常为 `true`
- 若发现当前需要重新编排而不是直接回修，补充 `reroute_via_starter=true`

## 完成条件

只有在评审记录已经落盘，且给出明确结论、支持真人确认或退回修订的说明，并且选定唯一下一步之后，这个 skill 才算完成。

若本 skill 由 reviewer subagent 执行，还应完成对父会话的结构化结果回传。
