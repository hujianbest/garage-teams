# AHE 需求评审

评审需求规格，并判断它是否已经可以批准进入设计阶段。

在完整 AHE workflow 中，它用于设计前的规格质量门禁，而不是普通文档意见收集。

## 目的

这个 skill 是需求规格冻结门禁。

如果规格本身不扎实，设计就会漂移；如果批准得过早，实现阶段就会继承本可避免的歧义。

高质量规格评审不只是判断“文档写得像规格”，而是判断：

- 范围、问题和成功标准是否已经站稳
- 关键需求是否真的可观察、可判断、可验收
- 规格是否已经足以成为 `ahe-design` 的稳定输入

## 硬性门禁

在规格通过评审并经过真人确认之前，不得进入 `ahe-design`。

如果当前请求尚未经过 `ahe-workflow-starter` 的阶段判断，不直接开始规格评审；应先回到 `ahe-workflow-starter` 完成路由。

## 评审方法

只评审需求规格本身，以及做判断所需的最少辅助上下文。

评审过程中不要顺手重新设计系统。

### 1. 先建立证据基线

在给出结论前，先读取并固定：

- 当前需求规格
- `AGENTS.md` 中的路径、模板、术语和状态约定
- `task-progress.md`（如果存在）
- 必要时回查少量上下文，用于确认规格状态、批准证据和关键锚点

不要只根据聊天记忆判断“这个规格已经差不多可以过了”。

### 2. 用冷读者视角检查规格

评审时，把自己当成一个马上要进入设计阶段、但没有聊天上下文的读者，重点问：

- 这个规格到底服务谁、解决什么问题、怎么判断成功？
- 哪些需求真的可验收，哪些只是听起来像要求？
- 如果把规格交给设计人员，哪里还会逼着对方补脑？

## 真人确认规则

`ahe-spec-review` 给出的“通过”，只表示规格已经达到可确认状态，不等于自动批准。

只有结论为 `通过` 时，才进入真人确认。

本 skill 默认运行在独立 reviewer subagent 中。若在该模式下运行，reviewer 只负责评审、落盘和回传结构化摘要；真正向真人展示结论并发起批准确认的动作，仍由父会话负责。

如果结论为 `需修改` 或 `阻塞`，不要发起真人批准确认；应基于评审结果直接回到 `ahe-specify`，与用户确认必要修订后继续完善规格。

在进入 `ahe-design` 之前，必须完成以下动作：

1. 向真人明确展示评审结论、关键风险和建议修改点
2. 邀请真人确认“该规格是否批准进入设计阶段”
3. 如果真人提出修改意见，继续和真人对话，回到规格内容上迭代修订
4. 只有在真人明确确认通过后，规格才算已批准

## Reviewer Subagent 运行约定

当本 skill 由 reviewer subagent 执行时：

- reviewer 负责读取规格与最小必要辅助上下文
- reviewer 负责把评审记录写入约定路径
- reviewer 负责回传结构化摘要
- reviewer 不负责代替父会话推进主链
- reviewer 不负责代替父会话发起真人确认

## 记录要求

评审完成后，必须将本次结论写入：

- `docs/reviews/spec-review-<topic>.md`

如 `AGENTS.md` 为当前项目声明了等价路径，按其中映射路径保存。

若项目尚未形成固定 review 记录格式，默认使用：

- `templates/review-record-template.md`

如果使用通用模板中的英文结论字段，请按以下方式映射：

- `通过` -> `pass`
- `需修改` -> `revise`
- `阻塞` -> `blocked`

如果当前任务正在使用 `task-progress.md` 驱动 workflow，且真人确认通过，还应同步更新：

- 需求规格文档中的状态字段
- `task-progress.md` 中的当前阶段与推荐下一步动作或 skill

若项目尚未形成固定进度记录格式，默认使用：

- `templates/task-progress-template.md`

## 高质量评审基线

交付给真人确认前，高质量规格评审结果至少应做到：

- 对齐 `ahe-specify` 的交付契约，而不是只看文档表面完整性
- 明确指出规格中的歧义、冲突、模糊词和未闭合开放问题
- 明确指出哪些问题会直接阻塞设计
- 不把评审退化成重新澄清一切，或顺手开始做设计

## 检查清单

请从以下维度检查需求规格：

### 1. 问题、范围与成功标准

- 问题陈述是否清晰？
- 规格是否清楚说明服务对象、目标与成功标准？
- 当前规格是否代表一个可被评审、可被设计的本轮增量，而不是把多个独立阶段或子系统混在同一轮里？
- 范围内内容是否明确？
- 范围外内容是否显式写出？

### 2. 需求质量与结构完整性

- 核心需求是否可观察、可测试？
- 关键功能需求是否带有对应验收标准？
- 是否存在重复、冲突、孤立或无法判断的需求条目？
- 需求描述中是否避免了隐藏设计决策？

### 3. 非功能需求与模糊词审计

- 模糊表达是否被移除或量化？
- 相关的非功能需求与关键约束是否写成可判断、可验收的条件？如果当前确实无相关项，是否明确写出 `无` / `不适用` 及其理由？
- 是否存在“快速 / 稳定 / 安全 / 友好”这类没有判断标准的表述？

### 4. 边界、依赖、接口与例外

- 主要用户角色是否已识别？
- 边界情况或负路径是否至少被识别出来？
- 约束、依赖、外部接口和假设是否写明？
- 范围例外、边界条件和失败路径是否足以支撑后续设计？

### 5. 开放问题与设计准备度

- 阻塞性开放问题是否被清楚标出？
- 若规格准备进入真人确认，阻塞性开放问题是否已经为空或明确写为“无”？
- 剩余非阻塞开放问题是否真的不会改变范围、成功标准、核心行为、接口、关键约束或关键非功能需求？
- 设计人员能否在不猜测基础信息的情况下继续？
- 需求编号或稳定需求条目是否足以支撑后续设计追溯？

## 输出格式

请严格使用以下结构：

```markdown
## 结论

通过 | 需修改 | 阻塞

## 发现项

- [严重级别] 问题

## 缺失或薄弱项

- 条目

## 下一步

`等待真人确认` | `回到规格修订`

## 记录位置

- `docs/reviews/spec-review-<topic>.md` 或映射路径

## 交接说明

- `等待真人确认`：仅当结论为 `通过`
- `回到规格修订`：仅当结论为 `需修改` 或 `阻塞`，并携带评审修订清单继续修订
```

严重级别说明：

- `critical`: 阻塞设计，或会直接导致错误设计输入
- `important`: 应在批准前修复
- `minor`: 不阻塞，但建议改进

## 判定规则

只有在以下条件全部满足时，才返回 `通过`：

- 范围清晰
- 问题、目标和成功标准足够清楚
- 当前规格代表一个可被评审、可被设计的单轮增量
- 核心需求可直接使用
- 核心范围具备验收标准
- 相关的非功能需求和关键约束具备可判断性，或已明确说明当前无相关项
- 不存在会阻塞设计的开放问题
- 剩余非阻塞开放问题不会改变设计主干判断
- 规格足以成为 `ahe-design` 的稳定输入

以下情况返回 `需修改`：

- 规格有用，但还不完整
- 重要区域仍存在歧义
- 验收标准偏弱或不完整
- 非功能需求、边界条件或负路径仍然偏弱，但可以通过一轮定向修订补齐

以下情况返回 `阻塞`：

- 规格过于模糊，无法支撑设计
- 核心问题或范围仍不清楚
- 关键矛盾尚未解决
- 阻塞性开放问题仍未闭合
- 需求结构、一致性或可追溯性缺口已经使下游设计无法安全启动

## 批准处理

如果规格通过，请明确说明它已经可以提交给真人确认。

只有在真人明确确认后，它才能成为设计阶段的已批准输入。

真人确认完成后，请显式同步：

- 规格文档状态为已批准
- `task-progress.md` 的 Current Stage
- `task-progress.md` 中记录的推荐下一步动作或 skill
- review 记录中的 Human Confirmation 与 Evidence Location

如果未通过，请把它退回规格修订，并附上具体修订点。

退回时请显式做到：

- 不等待真人批准确认
- 把发现项转换为用户可确认的修订行动
- 让下一轮规格修订直接基于这些修订点继续，而不是从零重新澄清

## 反模式

- 把评审变成重新设计
- 因为“后面再想”就直接批准
- 把隐含范围当成可以接受
- 忽略缺失的验收标准
- 忽略仍未闭合的阻塞性开放问题
- 评审刚返回“通过”就直接进入 `ahe-design`
- 不和真人确认，就把规格当成已批准输入

## 回传给父会话

若本 skill 运行在 reviewer subagent 中，至少回传：

- `conclusion`
- `next_action`
- `record_path`
- `key_findings`
- `needs_human_confirmation`：对本 skill 通常为 `true`
- 若发现当前需要重新编排而不是直接回修，补充 `reroute_via_starter=true`

## 完成条件

只有在评审记录已经落盘，且给出明确结论、支持真人确认或退回修订的说明，并且选定唯一下一步后，这个 skill 才算完成。

若本 skill 由 reviewer subagent 执行，还应完成对父会话的结构化结果回传。
