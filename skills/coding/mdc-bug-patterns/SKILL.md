# MDC 缺陷模式排查

基于已知错误模式，对当前改动做专项风险排查。

## 目的

这个 skill 用于把团队已经吃过的亏，转成当前任务可以执行的检查动作。

它不是通用代码评审的替代品，而是对高频错误类型的定向补强。

## 适用时机

优先用于以下场景：

- 当前任务刚完成最小实现，准备进入正式评审链
- 当前改动涉及边界、状态、时序、幂等、资源释放、配置读取等高风险区域
- 当前任务属于热修复或缺陷复发区域
- 团队已有类似错误案例，且本次改动命中相同风险类别

## 输入

阅读以下最少必要信息：

- 当前任务涉及的实现与测试变更
- 当前任务对应的规格、设计、任务片段
- 团队已知错误案例、事故复盘或缺陷模式清单（如有）

## 记录要求

排查完成后，默认将本次结论写入：

- `docs/reviews/bug-patterns-<task>.md`

如项目已有等价路径，可按 `mdc-contract` 映射路径保存。

若项目尚未形成固定 review 记录格式，默认使用：

- `skills/coding/templates/review-record-template.md`

如果结论为 `通过`，应同步更新：

- `task-progress.md` 中的缺陷模式排查状态
- `task-progress.md` 的 Next Skill 为 `mdc-test-review`

如果结论为 `需修改`，应同步更新：

- 当前任务需要回到实现修订
- `task-progress.md` 的 Next Skill 为 `mdc-implement`

如果结论为 `阻塞`，应同步更新：

- `task-progress.md` 中的缺陷模式排查阻塞状态与原因
- `task-progress.md` 的 Next Skill 暂时保持为 `mdc-bug-patterns`

若项目尚未形成固定进度记录格式，默认使用：

- `skills/coding/templates/task-progress-template.md`

## 参考资料

如果团队还没有成型的缺陷模式库，可先使用以下模板初始化：

- `references/bug-pattern-catalog-template.md`

## 检查清单

### 1. 历史重复缺陷

- 当前改动是否命中团队历史高频错误模式？
- 是否只是修掉症状，而没有处理同类缺陷机制？

### 2. 通用编码风险

- 是否存在边界值、空值、默认值处理缺口？
- 是否存在状态切换、时序依赖、重入、幂等等风险？
- 是否存在资源释放、异常恢复、配置误用等问题？

### 3. 防护是否足够

- 这些风险是否被测试覆盖？
- 是否有必要补充保护性判断、断言、降级逻辑或约束说明？

### 4. 扩散面

- 当前缺陷模式是否可能在相邻模块中重复出现？
- 是否需要把同类检查扩展到相关位置？

## 输出格式

请严格使用以下结构：

```markdown
## 结论

通过 | 需修改 | 阻塞

## 命中的缺陷模式

- 模式

## 缺失的防护

- 条目

## 下一步

`mdc-test-review` | `mdc-implement` | `补齐阻塞条件后重试 mdc-bug-patterns`

## 记录位置

- `docs/reviews/bug-patterns-<task>.md` 或映射路径
```

## 判定规则

只有当当前改动涉及的主要缺陷模式已经被识别，且风险有相应测试、保护措施或合理说明时，才返回 `通过`。

如果命中了已知高风险模式，但缺少必要防护、测试或修正，则返回 `需修改`。

如果缺少历史缺陷上下文、无法识别关键风险类别，或当前改动范围本身不可判定，则返回 `阻塞`。

## 反模式

- 把这个 skill 变成泛泛而谈的代码评审
- 只列风险，不说明是否已被覆盖
- 明知命中历史缺陷模式，却不补测试或防护
- 只检查单点，不考虑同类问题是否扩散

## 完成条件

只有在给出明确结论、命中模式、缺失防护、记录位置和唯一下一步之后，这个 skill 才算完成。
