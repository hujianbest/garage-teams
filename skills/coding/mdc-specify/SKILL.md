# MDC 需求澄清

创建一份定义“要做什么”的需求规格说明。

## 硬性门禁

在需求规格通过评审并获批之前，不得开始架构设计、任务拆解、脚手架或实现代码编写。

在 `mdc-spec-review` 给出“通过”结论之前，不发起真人批准确认。

如果当前请求尚未经过 `mdc-workflow-starter` 的阶段判断，不直接开始规格工作；应先回到 `mdc-workflow-starter` 完成路由。

## 目标

需求规格必须足以让后续设计和实现不靠猜测推进。

它应回答以下问题：

- 解决的是什么问题
- 用户是谁
- 哪些内容在范围内
- 哪些内容在范围外
- 存在哪些约束
- 如何验证成功

## 工件与状态要求

默认情况下，需求规格应保存到：

- `docs/specs/YYYY-MM-DD-<topic>-srs.md`

若项目已通过 `mdc-contract` 映射到其他路径，可使用映射路径。

在规格草稿交评审前，文档中至少应体现：

- 状态字段，例如 `状态: 草稿`
- 主题或范围标识
- 可供评审定位的清晰章节结构

若项目尚未形成固定进度记录格式，默认使用：

- `skills/coding/templates/task-progress-template.md`

规格草稿交给 `mdc-spec-review` 后，后续真人确认完成时还应同步：

- 规格文档状态
- `task-progress.md` 中的 Current Stage
- `task-progress.md` 中的 Next Skill

## 红旗信号

出现以下想法时，先停下。这通常说明你正在为“跳过需求澄清”找理由：

| 想法 | 实际要求 |
|---|---|
| “需求已经差不多清楚了，直接做设计吧” | 规格未通过评审前，不进入设计。 |
| “这些细节实现时自然会知道” | 范围、约束、验收标准必须先在规格里写清楚。 |
| “先写个大概，范围外内容不用专门说明” | 范围外内容必须显式写出。 |
| “用户没提验收标准，就先按常识理解” | 成功标准必须明确，不能靠默认理解。 |
| “我先顺手提个架构方案更高效” | 规格阶段不提前做设计决策。 |
| “这些开放问题以后再说，不影响继续写” | 会阻塞评审的问题必须先澄清。 |

## 工作流

按以下顺序执行。

### 1. 了解上下文

只阅读理解请求所需的材料：

- 用户提供的需求说明
- 与当前范围相关的现有项目文档
- 如果是增强需求，则阅读现有系统行为

优先读取：

- `mdc-contract` 中的规格路径映射
- `task-progress.md` 中的当前阶段记录（如果存在）

提炼初始假设、约束和未知项。

### 2. 先澄清，再写文档

在起草规格前，先提出聚焦问题。

规则如下：

- 尽量一次只问一个主题
- 只有在能减少来回沟通时，才合并相关问题
- 把模糊表述替换成可衡量描述
- 如果用户未给出范围外内容，要主动询问

至少要澄清这些方面：

1. purpose and users
2. functional scope
3. exclusions
4. constraints and dependencies
5. acceptance criteria

如果当前是因为 `mdc-spec-review` 返回“需修改”或“阻塞”而重新进入本 skill：

- 先读取评审结论中的发现项、缺失项和修订建议
- 优先修复 `critical` 与 `important` 问题
- 只针对仍然阻塞修订的问题向用户补充确认
- 不要重新发起整轮无关澄清

### 3. 起草规格

编写一份把“需求”与“设计决策”分开的规格文档。

除非项目已有固定模板，否则使用以下结构：

```markdown
# <主题> 需求规格说明

## 目标

## 范围

## 用户角色

## 功能需求

## 非功能需求

## 约束

## 范围外内容

## 验收标准

## 开放问题
```

默认应结合 `mdc-contract` 和项目模板决定最终文件位置；若项目无现成模板，至少应保留状态字段与上述章节骨架。

编写建议：

- 功能需求描述可观察行为
- 非功能需求描述可衡量质量目标
- 约束描述硬性限制
- 范围外内容必须显式写出，而不是隐含存在
- 开放问题只能保留那些不会阻塞评审的问题

### 4. 检查规格质量

交付前请确认：

- 需求可测试
- 模糊词已量化或删除
- 需求陈述中没有混入设计选择
- 范围外内容已显式说明
- 核心行为具备验收标准

### 5. 交给评审

草稿准备好后，直接交给 `mdc-spec-review`。

不要在 `mdc-specify` 阶段请求真人批准规格；真人确认只发生在 `mdc-spec-review` 给出“通过”之后。

若项目尚未形成固定 review 记录格式，默认由 `mdc-spec-review` 使用：

- `skills/coding/templates/review-record-template.md`

交接前请同步：

- 将草稿保存到约定路径
- 在 `task-progress.md` 中将 Current Stage 更新为需求评审前状态
- 在 `task-progress.md` 中将 Next Skill 更新为 `mdc-spec-review`

输出应写成：

```markdown
需求规格草稿已起草完成，请提交 `mdc-spec-review` 进行评审。

下一步 skill: `mdc-spec-review`
```

如果这是基于评审意见修订后的版本，也使用同样交接方式重新进入 `mdc-spec-review`。

注意：只有 `mdc-spec-review` 返回“通过”后，才进入真人确认；真人确认通过后才能进入 `mdc-design`。

## 反模式

- 从用户想法直接跳到架构设计
- 把头脑风暴笔记当成已批准规格
- 在规格里直接写任务
- 除非是硬性外部约束，否则不要使用 class、endpoint、table、framework 这类设计语言
- 把成功标准留成隐含信息

## 完成条件

只有在存在一份可评审的需求规格草稿，并且它已经准备好交给 `mdc-spec-review` 时，这个 skill 才算完成。
