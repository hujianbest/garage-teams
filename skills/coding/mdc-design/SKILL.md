# MDC 设计

创建设计文档，说明已批准规格将“如何”被实现。

## 硬性门禁

在设计经过评审并获批之前，不得拆解任务或编写实现代码。

在 `mdc-design-review` 给出“通过”结论之前，不发起真人批准确认。

如果当前请求尚未经过 `mdc-workflow-starter` 的阶段判断，不直接开始设计工作；应先回到 `mdc-workflow-starter` 完成路由。

## 前置条件

仅在以下条件满足时使用本 skill：

- 已存在需求规格
- 该规格已通过评审，或被明确批准

如果规格仍是草稿或还在修订中，请回到 `mdc-spec-review` 或 `mdc-specify`。

## 工件与状态要求

默认情况下，设计文档应保存到：

- `docs/designs/YYYY-MM-DD-<topic>-design.md`

若项目已通过 `mdc-contract` 映射到其他路径，可使用映射路径。

在设计草稿交评审前，文档中至少应体现：

- 状态字段，例如 `状态: 草稿`
- 主题或范围标识
- 可供评审定位的章节结构

若项目尚未形成固定进度记录格式，默认使用：

- `skills/coding/templates/task-progress-template.md`

设计草稿交给 `mdc-design-review` 后，后续真人确认完成时还应同步：

- 设计文档状态
- `task-progress.md` 中的 Current Stage
- `task-progress.md` 中的 Next Skill

## 目标

设计文档应在保持设计层抽象的前提下，尽量消除后续实现中的主要猜测空间。

它应回答以下问题：

- 存在哪些主要组件或模块
- 职责如何划分
- 数据和控制流如何在系统中流动
- 需要哪些接口
- 哪些约束会影响实现
- 设计如何支撑测试

## 红旗信号

出现以下想法时，先停下。这通常说明你正在为“跳过设计推敲”找理由：

| 想法 | 实际要求 |
|---|---|
| “规格已经有了，直接拆任务就行” | 设计未评审前，不进入任务拆解。 |
| “这个方案看起来最顺手，不用比较备选方案” | 至少比较两个可行方案后再定稿。 |
| “先把实现细节写进去，后面省事” | 设计要保持设计层抽象，不写成逐行实现说明。 |
| “非功能需求影响不大，可以先忽略” | 约束和非功能需求必须体现在设计决策里。 |
| “接口先模糊写一下，后面再补” | 关键接口与契约要足够明确，才能支撑任务规划。 |
| “测试以后再考虑” | 设计阶段就要说明测试策略。 |

## 工作流

### 1. 阅读已批准规格

提取：

- 核心范围
- 验收标准
- 约束
- 非功能需求
- 集成点

优先读取：

- `mdc-contract` 中的设计路径映射
- `task-progress.md` 中的当前阶段记录（如果存在）

### 2. 了解技术上下文

阅读最少但必要的技术上下文：

- 现有架构或项目布局
- 当前框架和依赖
- 已知的部署或运行约束

此时不要提前进入实现规划。

### 3. 提出 2 到 3 个候选方案

在最终选定设计前，至少比较两个可行方案。

对每个方案，简要说明：

- 它如何工作
- 主要优点
- 主要缺点
- 为什么适合或不适合当前规格

最后推荐一个方案。

如果当前是因为 `mdc-design-review` 返回“需修改”或“阻塞”而重新进入本 skill：

- 先读取评审结论中的发现项、薄弱点和修订建议
- 优先修复阻塞任务规划的问题
- 只针对仍然影响设计定稿的问题向用户补充确认
- 不要把已确认且未受影响的设计内容全部重做

### 4. 编写设计文档

除非项目已有固定模板，否则使用以下结构：

```markdown
# <主题> 实现设计
## 概述
## 设计驱动因素
## 候选方案
## 选定方案
## 架构设计
## 模块职责
## 数据流与控制流
## 接口与契约
## 测试策略
## 风险与待定问题
```

默认应结合 `mdc-contract` 和项目模板决定最终文件位置；若项目无现成模板，至少应保留状态字段与上述章节骨架。

编写建议：

- 区分需求与设计
- 清晰定义边界和职责
- 解释为什么所选方案适合该规格
- 提供足够支撑任务拆解的细节，但不要写成逐行编码说明

### 5. 评审前自检

确认以下内容：

- 每个主要需求都被设计覆盖
- 约束和非功能需求确实影响了设计决策
- 架构是连贯的，而不是组件清单
- 接口定义足够明确，能支撑后续任务规划
- 存在设计层面的测试策略

### 6. 交给评审

准备好之后，交给 `mdc-design-review`。

不要在 `mdc-design` 阶段请求真人批准设计；真人确认只发生在 `mdc-design-review` 给出“通过”之后。

若项目尚未形成固定 review 记录格式，默认由 `mdc-design-review` 使用：

- `skills/coding/templates/review-record-template.md`

交接前请同步：

- 将设计草稿保存到约定路径
- 在 `task-progress.md` 中将 Current Stage 更新为设计评审前状态
- 在 `task-progress.md` 中将 Next Skill 更新为 `mdc-design-review`

输出应使用：

```markdown
设计文档草稿已起草完成，请提交 `mdc-design-review` 进行评审。

下一步 skill: `mdc-design-review`
```

如果这是基于评审意见修订后的版本，也使用同样交接方式重新进入 `mdc-design-review`。

注意：只有 `mdc-design-review` 返回“通过”后，才进入真人确认；真人确认通过后才能进入 `mdc-tasks`。

## 反模式

- 把设计文档写成实现伪代码
- 只是复制需求规格，没有加入设计决策
- 只给一个方案，不讨论权衡
- 在设计文档里直接开始拆任务
- 忽略非功能需求和约束

## 完成条件

只有在存在一份可评审的设计文档，并且已准备好交给 `mdc-design-review` 时，这个 skill 才算完成。
