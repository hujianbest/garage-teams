# AHE 任务评审

评审任务计划，并判断是否可以开始实现。

在完整 AHE workflow 中，它用于实现前的执行准备度门禁，而不是普通意见收集。

## 硬性门禁

在任务计划通过评审并将结论写入仓库评审工件之前，不得进入 `ahe-test-driven-dev`。

如果当前请求尚未经过 `ahe-workflow-starter` 的阶段判断，不直接开始任务评审；应先回到 `ahe-workflow-starter` 完成路由。

## 目的

这个 skill 是执行准备度门禁。

如果任务计划过于粗糙，实现阶段就容易漂移。

## Reviewer Subagent 运行约定

本 skill 默认运行在独立 reviewer subagent 中。

当以该模式运行时：

- reviewer 负责读取任务计划与最小必要辅助上下文
- reviewer 负责把评审记录写入约定路径
- reviewer 负责回传结构化摘要
- reviewer 不负责代替父会话推进整个 workflow
- reviewer 不需要代替父会话再做额外编排，只返回结论与推荐下一步

高质量任务评审不只是判断“计划看起来能做”，而是判断：

- 任务是否真的能被单任务推进
- 依赖和顺序是否真的足以安全执行
- 验证方式和证据要求是否足以支持冷启动实现

## 高质量评审基线

交付给 `ahe-test-driven-dev` 前，高质量任务评审结果至少应做到：

- 明确指出任务计划是否真正对齐 `ahe-tasks` 产物契约
- 明确指出会阻塞实现的粒度、依赖、验证或追溯缺口
- 区分“可定向修补的任务计划缺口”与“不能继续推进的阻塞问题”
- 不把任务评审退化成重新写计划或提前进入实现建议

## 记录要求

评审完成后，必须将本次结论写入：

- `docs/reviews/tasks-review-<topic>.md`

如 `AGENTS.md` 为当前项目声明了等价路径，按其中映射路径保存。

若项目尚未形成固定 review 记录格式，默认使用：

- `templates/review-record-template.md`

如果使用通用模板中的英文结论字段，请按以下方式映射：

- `通过` -> `pass`
- `需修改` -> `revise`
- `阻塞` -> `blocked`

如果当前任务正在使用 `task-progress.md` 驱动 workflow，且结论为 `通过`，还应同步更新：

- 任务计划中的状态字段或批准记录
- `task-progress.md` 的 Current Stage
- `task-progress.md` 的 Current Active Task
- `task-progress.md` 中记录的推荐下一步动作或 skill

若项目尚未形成固定进度记录格式，默认使用：

- `templates/task-progress-template.md`

如果项目仍沿用该通用模板，评审完成前还必须确认实际进度工件已经补齐 canonical AHE 字段名，或至少能明确映射到 `Current Stage`、`Current Active Task`、`Next Action Or Recommended Skill`；不要假定 generic 字段会被下游自动兼容。

## 评审方法

### 1. 先建立证据基线

在给出结论前，先读取并固定以下证据来源：

- `AGENTS.md` 中的路径、模板和任务拆解约定
- 当前任务计划
- `task-progress.md`（如果存在）
- 必要时回查规格 / 设计中的锚点，用于 spot-check 追溯关系与批准新鲜度

不要只根据聊天记忆判断“这个计划应该够用了”。

### 2. 用执行者视角检查计划

评审时，不要把自己当成已经知道上下文的人，而要问：

- 如果另一个 agent 或下一次会话按这份计划执行，会不会在关键任务上补脑？
- 如果严格只推进一个活跃任务，这份计划是否还能顺利工作？
- 如果实现者照字面执行，最容易在哪些任务上误解顺序、证据或验证要求？

## 检查清单

### 1. 粒度与任务单元质量

- 任务是否足够小，可以直接执行和验证？
- 是否仍存在一个任务里混入多个行为的情况？
- 关键任务是否具有稳定任务 ID、目标、依赖、工件和完成条件？

### 2. 顺序、依赖与关键路径

- 任务顺序是否合理？
- 前置条件是否被正确遵守？
- 是否存在遗漏依赖、错误顺序或潜在并行冲突？
- 风险区域是否已经反映到顺序或关键路径里？

### 3. 验证准备度与新鲜证据

- 每个关键任务是否都有清晰验证方法？
- 完成条件是否明确？
- 关键任务是否说明了验证入口、预期证据或预期结果？
- 是否仍存在“做完再看”式的模糊任务？

### 4. 可追溯性与工件覆盖

- 主要任务能否追溯回设计和规格？
- 被 spot-check 的规格 / 设计锚点是否仍然是已批准且当前有效的上游依据？
- 文件 / 工件影响图是否足以支撑任务边界判断？
- 是否存在任务无法追溯到规格 / 设计依据，却又会进入实现？

### 5. 单任务推进与 TDD 入口准备度

- 是否给出了唯一 `Current Active Task` 选择规则？
- 推荐的首个活跃任务是否已经足够明确，能在 review 通过后安全写入状态工件？
- 对首个活跃任务或高风险任务，是否提供了可用的测试设计种子？
- 这些种子是否只是帮助 `ahe-test-driven-dev` 准备真人测试设计确认，而没有越权替代该门禁？

## 输出格式

请严格使用以下结构：

```markdown
## 结论

通过 | 需修改 | 阻塞

## 发现项

- [严重级别] 问题

## 计划薄弱点

- 条目

## 下一步

`进入 ahe-test-driven-dev` | `回到任务修订`

## 记录位置

- `docs/reviews/tasks-review-<topic>.md` 或映射路径
```

严重级别至少区分：

- `HIGH`：会显著影响执行安全、顺序正确性、验证有效性或单任务推进
- `MEDIUM`：不会立即阻塞实现，但会提高返工或误解风险
- `LOW`：清晰度或可维护性问题，可在后续修补

`回到任务修订` 是唯一的非通过下一步表达。若阻塞来源于上游规格 / 设计不稳定、批准状态不清或路由错误，仍先回到 `ahe-tasks`，再由 `ahe-tasks` 按既有契约先经 `ahe-workflow-starter` 重新编排。

## 判定规则

只有当计划满足以下条件时，才返回 `通过`：

- 顺序连贯
- 粒度可执行
- 关键依赖清楚
- 完成条件明确
- 关键任务具备验证方式与预期证据
- 能够追溯回规格和设计
- 被抽查的规格 / 设计锚点仍然处于已批准且当前有效的状态
- 已准备好唯一活跃任务选择规则，并足以安全进入 `ahe-test-driven-dev`

以下情况返回 `需修改`：

- 计划整体可用，但部分任务过大、描述不足、验证不清或追溯偏弱
- 首个活跃任务或高风险任务的测试设计种子仍偏弱，但可通过一轮定向修订补齐

以下情况返回 `阻塞`：

- 基于当前任务计划，无法安全推进实现
- 关键依赖、顺序或验证缺失，导致实现者必须大量猜测
- 任务无法可靠追溯到规格 / 设计，或首个活跃任务仍不具备安全进入实现的条件

## 批准处理

如果任务计划通过，请明确说明它已经可以进入 `ahe-test-driven-dev`。

如果未通过，请把它退回任务修订，并附上具体修订点。

退回时请显式做到：

- 不进入实现
- 把发现项转换为任务计划可执行的修订方向
- 让下一轮 `ahe-tasks` 直接基于这些修订点继续，而不是从零重写
- 如果阻塞来源于上游规格 / 设计不稳定，仍回到 `ahe-tasks`，但先经 `ahe-workflow-starter` 重新编排，再决定是否继续任务修订或回流上游

## 反模式

- 因为“实现的人会自己补齐”就直接通过
- 忽略缺失的验证步骤
- 用里程碑标题代替真正的任务项
- 接受没有活跃任务选择规则的任务计划
- 接受没有追溯依据却会直接进入实现的任务
- 评审通过后却不更新当前活跃任务和推荐下一步状态

## 回传给父会话

若本 skill 运行在 reviewer subagent 中，至少回传：

- `conclusion`
- `next_action`
- `record_path`
- `key_findings`
- `needs_human_confirmation`：对本 skill 通常为 `false`
- 若发现当前需要重新编排而不是直接回到任务修订，补充 `reroute_via_starter=true`

## 完成条件

评审结论必须写入仓库中的评审记录路径。可以在对话中摘要结论，但对话不能替代记录工件。

只有在评审记录已经落盘，且给出明确结论、计划薄弱点和唯一下一步之后，这个 skill 才算完成。

若本 skill 由 reviewer subagent 执行，还应完成对父会话的结构化结果回传。
