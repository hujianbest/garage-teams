# AHE 完成门禁

在宣告完成之前，必须先确认有足够证据。

在完整 AHE workflow 中，它通常作为末端完成判断。

## 目的

这个 skill 用于在更新状态、切换任务或输出完成说明之前，确认“当前任务已完成”这件事有最新证据支撑。

## 适用时机

优先用于以下场景：

- 需要确认当前任务是否真的可以宣告完成
- 当前任务已通过测试评审、代码评审、追溯性评审和回归检查
- 准备更新 `task-progress.md`、`RELEASE_NOTES.md` 或其他完成状态记录
- 准备宣告“当前任务完成”并进入 `ahe-finalize`

## 输入

阅读以下最少必要信息：

- 当前任务标识与其完成声明
- 最新测试、构建、类型检查或集成验证命令
- 已有评审与回归记录

如果当前上下文无法提供完整评审链记录，至少应提供当前任务的完成声明、支持该声明的最新验证命令，以及可用的结果证据。

## 铁律

没有针对当前最新代码状态的验证证据，就不能宣称任务完成。

如果本轮流程里没有亲自运行验证命令，就不能诚实地宣称完成。

## 非目标

本 skill 不负责：

- 重新做规格、设计或代码评审
- 用聊天印象替代验证命令
- 因为“差不多完成了”就放松完成门禁

## 在以下动作前必须使用本 Skill

- 说任务已经完成
- 更新进度或状态为已完成
- 切换到下一个任务
- 编写带有“已完成”含义的提交或交付说明

## 记录要求

完成门禁结束后，必须将本次验证写入：

- `docs/verification/completion-<task>.md`

如 `AGENTS.md` 为当前项目声明了等价路径，按其中映射路径保存。

若项目尚未形成固定 verification 记录格式，默认使用：

- `templates/verification-record-template.md`

如默认模板未覆盖本 skill 所需字段，应在该模板基础上补充，而不是省略这些信息。

当本 skill 与共享 verification 模板存在字段或标题差异时，以本 skill 的输出结构为准。

如果当前任务正在使用 `task-progress.md` 驱动 workflow，应同步更新：

- `task-progress.md` 中当前任务的完成状态
- 当前任务是可以进入 `ahe-finalize`，还是仍需回到实现修订
- 当前阻塞原因或证据缺口（如有）

若项目尚未形成固定进度记录格式，默认使用：

- `templates/task-progress-template.md`

## 工作流

### 1. 明确你要宣告的结论

明确写出你准备宣告什么，例如：

- 测试通过
- 功能行为正常
- 缺陷已修复
- 任务已完成

### 2. 确定证明该结论的命令

选择真正能证明该结论的命令。

不要用更弱的证据替代。

可接受的证据通常包括：

- 测试命令与结果
- 构建 / 类型检查 / lint 结果
- 与当前任务直接相关的集成验证

已落盘的 review / regression 记录可以作为辅助上下文，但不能替代当前轮直接支撑完成结论的验证命令结果。

### 3. 执行最新验证

立即运行完整验证命令。

### 4. 阅读完整结果

检查：

- 退出码
- 失败数量
- 输出是否真的支持该结论

### 5. 对结论做门禁判断

- 如果证据支持结论，则允许完成
- 如果不支持，则说明真实状态并回到实现修订

## 输出格式

请严格使用以下结构：

```markdown
## 结论

通过 | 需修改 | 阻塞

## 已验证结论

- 结论项

## 证据

- 命令与结果摘要

## 下一步

从以下值中选择一个：`ahe-finalize` | `ahe-test-driven-dev`

## 记录位置

- `docs/verification/completion-<task>.md` 或映射路径
```

## 判定规则

只有当准备宣告的“完成”结论被最新证据直接支持时，才返回 `通过`。

如果证据不足以支持该结论，或者仍需继续实现，则返回 `需修改`。

如果由于环境或工具链问题无法运行正确的验证命令，则返回 `阻塞`。

## 反模式

- 说“现在应该算完成了”
- 依赖旧输出
- 把主观感觉当成证据
- 认为评审通过就等于运行成功
- 因为做烦了就直接进入下一步

## 完成条件

门禁结论必须写入仓库中的验证记录路径。可以在对话中摘要结论，但对话不能替代验证工件。

只有在验证记录已经落盘，且用最新证据批准或拒绝完成结论，并给出唯一下一步后，这个 skill 才算完成。
