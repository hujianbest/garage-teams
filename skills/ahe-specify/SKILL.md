# AHE 需求澄清

创建一份定义“要做什么、为什么做、做到什么程度算完成”的需求规格说明。

## 硬性门禁

在需求规格通过评审并获批之前，不得开始架构设计、任务拆解、脚手架或实现代码编写。

在 `ahe-spec-review` 给出“通过”结论之前，不发起真人批准确认。

如果当前请求尚未经过 `ahe-workflow-starter` 的阶段判断，不直接开始规格工作；应先回到 `ahe-workflow-starter` 完成路由。

## 目标

需求规格必须足以让后续设计和实现不靠猜测推进。

它至少应回答以下问题：

- 解决的是什么问题，为什么值得做
- 用户是谁，关键使用场景是什么
- 哪些内容在范围内
- 哪些内容明确不做
- 核心行为是什么
- 成功与失败如何判断
- 存在哪些约束、依赖、接口和假设
- 哪些问题已经确认，哪些仍待补充

## 高质量规格基线

高质量规格不是写得越多越好，而是让后续设计在关键问题上不用猜。

交给 `ahe-spec-review` 之前，规格至少应具备这些特征：

- 范围清晰，且范围外内容显式写出
- 核心功能需求是可观察、可验证的
- 关键边界、失败路径或权限差异至少被识别出来
- 非功能需求和约束不是口号，而是可落地判断的条件
- 需求与设计决策分离，不提前做架构选型
- 阻塞性开放问题已经被解决，或仍被错误地留在文档中的情况被消除

## 工件与状态要求

默认情况下，需求规格应保存到：

- `docs/specs/YYYY-MM-DD-<topic>-srs.md`

若 `AGENTS.md` 为当前项目声明了规格路径映射，优先使用映射路径；否则使用默认路径。

在规格草稿交评审前，文档中至少应体现：

- 状态字段，例如 `状态: 草稿`
- 主题或范围标识
- 可供评审定位的清晰章节结构

若项目尚未形成固定进度记录格式，默认使用：

- `templates/task-progress-template.md`

规格草稿交给 `ahe-spec-review` 后，后续真人确认完成时还应同步：

- 规格文档状态
- `task-progress.md` 中的 Current Stage
- `task-progress.md` 中的 Next Action Or Recommended Skill

## 红旗信号

出现以下想法时，先停下。这通常说明你正在为“跳过需求澄清”找理由：

| 想法 | 实际要求 |
|---|---|
| “需求已经差不多清楚了，直接做设计吧” | 规格未通过评审前，不进入设计。 |
| “这些细节实现时自然会知道” | 范围、约束、验收标准必须先在规格里写清楚。 |
| “先写个大概，范围外内容不用专门说明” | 范围外内容必须显式写出。 |
| “用户没提验收标准，就先按常识理解” | 成功标准必须明确，不能靠默认理解。 |
| “我先顺手提个架构方案更高效” | 规格阶段不提前做设计决策。 |
| “这些开放问题以后再说，不影响继续写” | 会阻塞评审的问题必须先澄清。 |
| “这个需求很大，先全部塞进一份规格” | 先拆清当前轮次范围，再写规格。 |

## 工作流

按以下顺序执行。

### 1. 了解上下文

只阅读理解请求所需的材料：

- 用户提供的需求说明
- 与当前范围相关的现有项目文档
- 如果是增强需求，则阅读现有系统行为
- 如果已有规格草稿或评审记录，则读取与当前修订直接相关的部分

优先读取：

- `AGENTS.md` 中与规格相关的路径映射、模板覆盖和团队约束
- `task-progress.md` 中的当前阶段记录（如果存在）

先提炼：

- 已确认事实
- 初始目标和成功标准
- 约束、依赖与兼容性线索
- 明显未知项、矛盾点和潜在误解

### 2. 先判断是否需要缩小本轮范围

如果当前请求同时包含多个相对独立的子系统、阶段或产品能力，不要直接为整个大而散的目标起草一份规格。

先帮助用户收敛当前轮次：

- 哪个问题最值得优先解决
- 哪些能力必须进入当前版本
- 哪些能力属于后续增量

规格要服务于当前这轮可被评审、可被设计的范围，而不是把未来所有想法一次性装进去。

### 3. 分轮澄清需求

在起草规格前，先提出聚焦问题。澄清时遵循 `Capture -> Challenge -> Clarify`：

- `Capture`：先复述你当前理解的目标、对象和范围
- `Challenge`：主动指出模糊词、打包需求、隐藏假设或缺失边界
- `Clarify`：把想法收敛成可观察、可验证、可评审的表述

提问规则：

- 尽量一次只推进一个主题
- 只有在能减少来回沟通时，才合并 2 到 4 个紧密相关的问题
- 优先用具体场景、反例和失败路径来问
- 把模糊表述替换成可衡量描述
- 如果用户未给出范围外内容，要主动询问
- 如果用户已经提供足够完整的信息，不机械地把所有轮次都问一遍

默认至少检查这些澄清轮次是否需要：

1. **问题、用户、目标与非目标**
   - 解决的核心问题是什么
   - 主要用户是谁
   - 这一轮最重要的成功标准是什么
   - 明确不打算解决什么
2. **核心行为与关键流程**
   - 用户触发什么动作
   - 系统应表现出什么可观察结果
   - 核心流程有哪些关键步骤
3. **边界、异常与失败路径**
   - 输入无效、依赖失败、权限不足、资源缺失时会怎样
   - 是否存在不同角色、状态或阶段下的差异行为
4. **约束、依赖、接口与兼容性**
   - 既有系统、外部接口、数据约束、预算、法规、许可、部署环境
   - 是否要兼容旧流程、旧数据或旧接口
   - 哪些内容只是当前假设，若假设不成立会影响什么
5. **非功能需求与验收口径**
   - 性能、可靠性、安全、隐私、可观察性、可访问性、国际化、可维护性等是否相关
   - 每个相关非功能需求是否能写成可判断的条件，而不是“快”“稳定”“好用”
6. **术语与待确认项**
   - 是否有容易歧义的业务词汇
   - 是否还存在必须落盘说明的问题

如果当前是因为 `ahe-spec-review` 返回“需修改”或“阻塞”而重新进入本 skill：

- 先读取评审结论中的发现项、缺失项和修订建议
- 优先修复 `critical` 与 `important` 问题
- 只针对仍然阻塞修订的问题向用户补充确认
- 不要重新发起整轮无关澄清

### 4. 整理需求类别，再写文档

在写规格前，先把已经确认的内容按角色整理清楚。默认至少区分：

- `FR`：功能需求，描述系统必须表现出的行为
- `NFR`：非功能需求，描述可衡量的质量目标或运行要求
- `CON`：约束，描述对方案空间的硬性限制
- `IFR`：接口与依赖，描述外部系统、数据契约或兼容边界
- `ASM`：假设，描述当前暂时成立、但失效后会影响判断的前提
- `EXC`：范围外内容，描述本轮明确不做什么

对于非 trivial 的规格，核心功能需求默认使用稳定编号，例如 `FR-001`、`FR-002`。非功能需求可使用 `NFR-001` 形式。

要求如下：

- 每条核心功能需求要能单独理解
- 不要把多个彼此独立的行为硬写成一条需求
- 每条核心功能需求都应配至少一个可验证验收标准
- 验收标准优先用场景化表达，例如 Given / When / Then 风格，但不要为了格式牺牲清晰度
- 对关键失败路径、边界条件、权限冲突，也至少写出对应验收口径

### 5. 起草规格

编写一份把“需求”与“设计决策”分开的规格文档。

若 `AGENTS.md` 为当前项目声明了规格模板、章节骨架或命名要求，优先遵循这些约定。

若 `AGENTS.md` 未提供规格模板覆盖，则使用以下默认结构：

```markdown
# <主题> 需求规格说明

- 状态: 草稿
- 主题: <主题>

## 1. 背景与问题陈述

## 2. 目标与成功标准

## 3. 用户角色与关键场景

## 4. 范围

## 5. 范围外内容

## 6. 术语与定义（按需）

## 7. 功能需求

### FR-001 <需求标题>

#### 需求说明

#### 验收标准

## 8. 非功能需求

### NFR-001 <需求标题>

## 9. 外部接口与依赖（按需）

## 10. 约束

## 11. 假设（按需）

## 12. 开放问题

### 阻塞性开放问题（交评审前必须为空或写明 `无`）

### 非阻塞开放问题
```

默认应先遵循 `AGENTS.md` 中的路径映射与模板覆盖；若未提供，再使用本 skill 的默认路径和上述章节骨架。

编写要求：

- 背景描述为什么要做，不写成方案介绍
- 目标描述这一轮要达成什么结果
- 范围描述本轮必须覆盖的内容
- 范围外内容必须显式写出，而不是隐含存在
- 功能需求描述可观察行为，而不是实现手段
- 非功能需求描述可判断条件，而不是空泛形容词
- 约束描述硬性限制
- 假设要写明失效风险或影响
- 开放问题只能保留那些不会阻塞评审的问题

### 6. 区分开放问题

写规格时，明确区分以下四类事项：

- 已确认事项：可直接写入规格正文
- 需用户确认事项：必须先问清再写
- 非阻塞开放问题：可以保留在文档中，但不能影响后续设计主干判断
- 阻塞性开放问题：草稿期可以显式跟踪，但交给 `ahe-spec-review` 前必须已经解决；评审前该小节应为空或明确写 `无`

以下情况通常应视为阻塞性开放问题：

- 核心范围仍然摇摆
- 主要用户或主要成功标准未定
- 核心流程的关键结果无法判断
- 关键约束、依赖或兼容要求未知
- 关键非功能要求完全缺失，且会显著影响设计

保留“阻塞性开放问题”小节的目的，是为了在草稿收敛过程中暴露风险，而不是允许把这些问题带进评审。

### 7. 交付前自检

交给 `ahe-spec-review` 前，请确认：

- 问题陈述、目标和主要用户清楚
- 范围内与范围外内容都已显式说明
- 核心功能需求逐条可观察、可验证
- 核心功能需求与验收标准的粒度基本对应
- 关键异常、边界、失败路径或权限差异至少已识别
- 模糊词已量化、替换或删除
- 需求陈述中没有混入设计选择
- 非功能需求不是“快/稳/好用”式口号，而是可判断条件
- 约束、依赖、接口与假设已经写到足以支撑后续设计
- 没有把阻塞性开放问题留到评审之后
- 规格已经足以让设计阶段不靠猜测继续

### 8. 交给 reviewer subagent 执行规格评审

草稿准备好后，不要在父会话里直接按 `ahe-spec-review` 的检查清单内联评审。

正确做法是：

1. 将规格草稿保存到约定路径
2. 组装最小 spec review request，只包含：
   - 规格文档路径
   - 主题 / topic
   - 做判断所需的最小辅助上下文路径
   - 预期 review 记录路径
3. 启动独立 reviewer subagent，并在该 subagent 中调用 `ahe-spec-review`
4. 由 reviewer subagent 负责：
   - 读取 `ahe-spec-review`
   - 写 review 记录
   - 回传结构化摘要
5. 父会话读取 reviewer 返回结果后再继续：
   - 若结论为 `通过`，由父会话向真人展示评审结论并发起批准确认
   - 若结论为 `需修改` 或 `阻塞`，携带 reviewer 的关键发现项回到本 skill 修订

不要在 `ahe-specify` 阶段请求真人批准规格；真人确认只发生在 `ahe-spec-review` 返回“通过”，且该结果已经由 reviewer subagent 回传给父会话之后。

若项目尚未形成固定 review 记录格式，默认由 `ahe-spec-review` 使用：

- `templates/review-record-template.md`

交接前请同步：

- 将草稿保存到约定路径
- 在 `task-progress.md` 中将 Current Stage 更新为能唯一映射到“规格草稿已完成、下一步进入 `ahe-spec-review`”的规范阶段名；不要写成含糊的自然语言状态
- 在 `task-progress.md` 中将 Next Action Or Recommended Skill 更新为 `ahe-spec-review`

输出应写成：

```markdown
需求规格草稿已起草完成，下一步应派发独立 reviewer subagent 执行 `ahe-spec-review`。

推荐下一步动作或 skill: `ahe-spec-review`
```

如果这是基于评审意见修订后的版本，也使用同样交接方式重新派发 reviewer subagent 执行 `ahe-spec-review`。

注意：只有 `ahe-spec-review` 返回“通过”后，才进入真人确认；真人确认通过后才能进入 `ahe-design`。

## 反模式

- 从用户想法直接跳到架构设计
- 把头脑风暴笔记当成已批准规格
- 在规格里直接写任务、里程碑或提交计划
- 把多个独立能力打包成一句“大需求”
- 只写 happy path，不写边界和失败路径
- 除非是硬性外部约束，否则不要使用 class、endpoint、table、framework 这类设计语言
- 把成功标准留成隐含信息
- 把“以后再补”当成默认策略

## 完成条件

只有在存在一份可评审的需求规格草稿，并且它已经准备好交给 `ahe-spec-review` 时，这个 skill 才算完成。
