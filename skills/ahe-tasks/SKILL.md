# AHE 任务拆解

创建任务计划，把已批准设计转化为可执行工作。

## 硬性门禁

在任务计划通过评审并写入批准结论之前，不得开始实现。

如果当前请求尚未经过 `ahe-workflow-starter` 的阶段判断，不直接开始任务规划；应先回到 `ahe-workflow-starter` 完成路由。

## 前置条件

仅在以下条件满足时使用本 skill：

- 需求规格已批准
- 设计已批准

如果设计仍是草稿或还在修订中，请回到 `ahe-design-review` 或 `ahe-design`。

## 目标

任务计划必须让实现过程变得可预期，并且能被冷启动执行。

它至少应明确以下内容：

- 里程碑
- 有序任务
- 依赖关系
- 任务级完成条件
- 验证预期与新鲜证据
- 当前活跃任务选择规则

## 高质量任务计划基线

高质量任务计划不是把设计标题改成任务标题，而是把工作真正拆成可执行单元。

交给 `ahe-tasks-review` 前，计划至少应具备这些特征：

- 每个关键任务都足够小，可以被单任务推进
- 任务能回指到规格 / 设计中的依据，而不是凭感觉新增
- 每个任务都知道会触碰哪些文件或工件
- 每个任务都知道如何验证完成
- 任务计划为 `ahe-test-driven-dev` 准备了测试设计种子，但没有越权替代真人测试设计确认

## 工件与状态要求

默认情况下，任务计划应保存到：

- `docs/tasks/YYYY-MM-DD-<topic>-tasks.md`

若 `AGENTS.md` 为当前项目声明了任务计划路径映射，优先使用映射路径；否则使用默认路径。

在任务计划交评审前，文档中至少应体现：

- 状态字段，例如 `状态: 草稿`
- 主题或范围标识
- 当前活跃任务选择规则
- 可供评审定位的章节结构

若项目尚未形成固定进度记录格式，默认使用：

- `templates/task-progress-template.md`

下游 AHE skill 默认读取 canonical 字段名；如果项目仍沿用通用模板，请在实际进度工件中补齐或显式采用这些 canonical 名称，而不要假定 `Current Task` / `Next Action` 等 generic 字段会被自动兼容。

任务计划交给 `ahe-tasks-review` 后，还应同步：

- `task-progress.md` 中的 Current Stage
- `task-progress.md` 中的 Next Action Or Recommended Skill

## 红旗信号

出现以下想法时，先停下。这通常说明你正在为“跳过有效任务拆解”找理由：

| 想法 | 实际要求 |
|---|---|
| “设计差不多了，先开始实现再补任务” | 任务计划未准备好评审前，不进入实现。 |
| “先列几个大任务，后面再细化” | 任务必须小到可以明确执行和验证。 |
| “依赖关系实现时自然会处理” | 依赖和顺序必须提前写清。 |
| “完成条件先不写，做完就知道了” | 每个任务都要有明确完成定义和验证方式。 |
| “把整块功能放成一个任务更省事” | 大到无法独立验证的任务不能进入计划。 |
| “验证放到功能全部做完再一起测” | 验证预期必须体现在任务级，而不是拖到最后。 |
| “测试设计后面让实现阶段自己想” | 任务计划至少要给出测试设计种子，减少实现阶段补脑。 |

## 工作流

### 1. 阅读已批准输入并提取任务拆解信号

阅读：

- 已批准需求规格
- 已批准设计
- 当前项目上下文（如相关）

优先读取：

- `AGENTS.md` 中与任务计划相关的路径映射、模板覆盖和任务拆解约束
- `task-progress.md` 中的当前阶段记录（如果存在）

至少提取：

- 主要工作流
- 依赖与顺序
- 测试影响
- 风险或不确定区域
- 关键需求 / 设计决策的稳定锚点

如果当前是因为 `ahe-tasks-review` 返回 `需修改` 或 `阻塞` 而重新进入本 skill：

- 先读取评审结论中的发现项、计划薄弱点和修订建议
- 优先修复粒度过大、缺少完成条件、依赖遗漏或追溯不清等关键问题
- 只针对仍然阻塞修订的问题向用户补充确认
- 不要把已确认且未受影响的任务全部重做
- 如果阻塞来源于上游规格 / 设计仍不稳定，先回到 `ahe-workflow-starter` 重新编排，再决定是否继续任务修订或回流上游

### 2. 先做文件 / 工件影响图

在正式列任务前，先明确本轮会涉及哪些工件：

- 会创建哪些文件
- 会修改哪些文件、配置、文档或状态工件
- 会新增或更新哪些测试 / 验证入口

这一步的目标不是把实现写出来，而是锁定任务边界，避免出现“实现某模块”这类脱离工件现实的任务。

### 3. 定义里程碑与追溯关系

把工作分组为能产生明确阶段性成果的里程碑。

每个里程碑应包含：

- 目标
- 包含的任务
- 退出标准
- 对应的需求 / 设计依据

至少对关键需求和关键设计决策建立显式追溯关系，说明它们分别落到哪些里程碑与任务。

### 4. 拆解为可执行任务单元

任务必须足够小，能够被明确执行和验证，并适合单任务推进。

优先使用这类任务单元：

- 为某个单一行为或接口补齐一个可验证闭环
- 为某个高风险路径补齐实现与验证
- 为某个依赖切换、数据迁移或状态更新完成可验证收口

避免使用这类模糊任务项：

- 实现某模块
- 完成功能
- 后面再优化

对每个关键任务，至少明确：

- 任务 ID
- 目标
- 前置依赖
- 会触碰哪些工件
- 验证方式
- 预期证据
- 完成条件
- 首个活跃任务或高风险任务的测试设计种子

`测试设计种子` 至少说明：

- 主要行为
- 关键边界或负向场景
- 哪些点适合在 `ahe-test-driven-dev` 中优先做 fail-first

这一步是在为 `ahe-test-driven-dev` 准备更好的输入。真正的 Red-Green-Refactor 微循环仍发生在单个活跃任务内部，不在这里把它拆成独立顶层任务。

### 5. 强化依赖、验证与当前活跃任务规则

对每个任务，尽量给出：

- 依赖它之前完成的任务或工件
- 推荐的执行命令或验证入口
- 预期结果或新鲜证据

如果命令尚不能唯一确定，至少指出应在哪个测试、脚本、记录或工件中看到验证结果。

并至少为计划补一条规则：

- 如何从任务计划中选定唯一 `Current Active Task`

### 6. 编写任务计划

若 `AGENTS.md` 为当前项目声明了任务计划模板、章节骨架或当前活跃任务规则，优先遵循这些约定。

若 `AGENTS.md` 未提供任务计划模板覆盖，则使用以下默认结构：

```markdown
# <主题> 任务计划

- 状态: 草稿
- 主题: <主题>

## 1. 概述

## 2. 里程碑

## 3. 文件 / 工件影响图

## 4. 需求与设计追溯

## 5. 任务拆解

### T1. <任务名>
- 目标:
- 依赖:
- 触碰工件:
- 测试设计种子:
- 验证方式:
- 预期证据:
- 完成条件:

## 6. 依赖与关键路径

## 7. 完成定义与验证策略

## 8. 当前活跃任务选择规则

## 9. 风险与顺序说明
```

编写要求：

- 不把任务计划写成设计文档副本
- 不把里程碑标题当成真实任务
- 让关键任务具备冷启动可执行性
- 让关键任务能够追溯回规格与设计
- 让每个任务都能回答“做完的证据是什么”

### 7. 评审前自检

交给 `ahe-tasks-review` 前，请确认：

- 不存在大到无法单任务推进的任务
- 关键任务有明确依赖、验证方式和完成条件
- 关键任务能追溯回规格或设计依据
- 风险区域已经体现在任务顺序、验证或显式风险说明中
- 计划已经给出唯一 `Current Active Task` 选择规则
- 对首个活跃任务或高风险任务，测试设计种子已足以帮助 `ahe-test-driven-dev` 进入真人测试设计确认

### 8. 交给 reviewer subagent 执行任务评审

任务计划准备好后，不要在父会话里直接按 `ahe-tasks-review` 的检查清单内联评审。

正确做法是：

1. 将任务计划保存到约定路径
2. 组装最小 tasks review request，只包含：
   - 任务计划路径
   - 相关规格 / 设计路径
   - 本次主题 / topic
   - 做判断所需的最小辅助上下文路径
   - 预期 review 记录路径
3. 启动独立 reviewer subagent，并在该 subagent 中调用 `ahe-tasks-review`
4. 由 reviewer subagent 负责：
   - 读取 `ahe-tasks-review`
   - 写 review 记录
   - 回传结构化摘要
5. 父会话读取 reviewer 返回结果后再继续：
   - 若结论为 `通过`，进入 `ahe-test-driven-dev`
   - 若结论为 `需修改` 或 `阻塞`，携带 reviewer 的关键发现项回到本 skill 修订

若项目尚未形成固定 review 记录格式，默认由 `ahe-tasks-review` 使用：

- `templates/review-record-template.md`

交接前请同步：

- 将任务计划保存到约定路径
- 在 `task-progress.md` 中将 Current Stage 更新为能唯一映射到“任务计划草稿已完成、下一步进入 `ahe-tasks-review`”的规范阶段名
- 在 `task-progress.md` 中将 Next Action Or Recommended Skill 更新为 `ahe-tasks-review`

建议的首个活跃任务应先写在任务计划正文中；只有在 `ahe-tasks-review` 通过并写入批准记录后，再把权威版 `Current Active Task` 写入 `task-progress.md` 或等价状态工件。

输出应使用：

```markdown
任务计划草稿已起草完成，下一步应派发独立 reviewer subagent 执行 `ahe-tasks-review`。

推荐下一步动作或 skill: `ahe-tasks-review`
```

注意：只有 `ahe-tasks-review` 返回 `通过`，且批准结论已写入任务计划或等价评审工件后，才进入 `ahe-test-driven-dev`。

## 反模式

- 把任务计划写成设计文档副本
- 使用大到无法验证的任务
- 漏掉依赖关系
- 漏掉完成条件
- 漏掉触碰工件或验证证据
- 写了任务却无法追溯到规格或设计
- 把验证拖到整个功能结束时才做

## 完成条件

只有在存在一份可评审任务计划，并且已准备好交给 `ahe-tasks-review` 时，这个 skill 才算完成。
