# AHE 设计

创建设计文档，说明已批准规格将“如何”被实现。

## 硬性门禁

在设计经过评审并获批之前，不得拆解任务或编写实现代码。

在 `ahe-design-review` 给出“通过”结论之前，不发起真人批准确认。

如果当前请求尚未经过 `ahe-workflow-starter` 的阶段判断，不直接开始设计工作；应先回到 `ahe-workflow-starter` 完成路由。

## 前置条件

仅在以下条件满足时使用本 skill：

- 已存在需求规格
- 该规格已被明确批准

如果规格仍是草稿、评审虽通过但尚未完成真人确认，或仍在修订中，请回到 `ahe-spec-review` 或 `ahe-specify`。

## 目标

设计文档应在保持设计层抽象的前提下，尽量消除后续实现中的主要猜测空间。

它至少应回答以下问题：

- 设计由哪些关键输入驱动
- 存在哪些主要组件或模块
- 职责如何划分，边界如何约束
- 数据和控制流如何在系统中流动
- 需要哪些关键接口与契约
- 约束与非功能需求如何落到设计上
- 设计如何支撑测试与后续任务规划

## 高质量设计基线

高质量设计不是把规格重写一遍，也不是提前写实现细节，而是把关键设计决策讲清楚。

交给 `ahe-design-review` 前，设计至少应具备这些特征：

- 能追溯到已批准规格中的关键需求
- 不只是列组件清单，而是说明它们如何协作
- 关键方案选择有明确理由、代价和风险
- 接口与契约足够清晰，能支撑任务规划
- 非功能需求与约束真正影响了设计，而不是只在概述里出现
- 对非 trivial 设计，至少有最少必要的结构 / 交互视图帮助评审理解

## 工件与状态要求

默认情况下，设计文档应保存到：

- `docs/designs/YYYY-MM-DD-<topic>-design.md`

若 `AGENTS.md` 为当前项目声明了设计路径映射，优先使用映射路径；否则使用默认路径。

在设计草稿交评审前，文档中至少应体现：

- 状态字段，例如 `状态: 草稿`
- 主题或范围标识
- 可供评审定位的章节结构

若项目尚未形成固定进度记录格式，默认使用：

- `templates/task-progress-template.md`

设计草稿交给 `ahe-design-review` 后，后续真人确认完成时还应同步：

- 设计文档状态
- `task-progress.md` 中的 Current Stage
- `task-progress.md` 中的 Next Action Or Recommended Skill
- review 记录中的 Human Confirmation 与 Evidence Location（由 `ahe-design-review` 负责落盘）

## 红旗信号

出现以下想法时，先停下。这通常说明你正在为“跳过设计推敲”找理由：

| 想法 | 实际要求 |
|---|---|
| “规格已经有了，直接拆任务就行” | 设计未评审前，不进入任务拆解。 |
| “这个方案看起来最顺手，不用比较备选方案” | 至少比较两个可行方案后再定稿。 |
| “先把实现细节写进去，后面省事” | 设计要保持设计层抽象，不写成逐行实现说明。 |
| “非功能需求影响不大，可以先忽略” | 约束和非功能需求必须体现在设计决策里。 |
| “接口先模糊写一下，后面再补” | 关键接口与契约要足够明确，才能支撑任务规划。 |
| “测试以后再考虑” | 设计阶段就要说明测试策略。 |
| “先把任务顺手列出来，这样更完整” | 设计要服务任务规划，但不能直接变成任务计划。 |

## 工作流

### 1. 阅读已批准规格并提取设计驱动因素

优先读取：

- `AGENTS.md` 中与设计相关的路径映射、模板覆盖和团队设计约束
- `task-progress.md` 中的当前阶段记录（如果存在）
- 已批准规格中与当前范围直接相关的部分

至少提取这些设计输入：

- 核心范围与关键行为
- 验收标准
- 约束
- 非功能需求
- 集成点与外部接口
- 关键需求编号或稳定需求条目
- 会影响架构选择的开放问题

如果规格中仍有会阻塞架构判断的未决问题，先判断它属于哪一类：

- 若它会改变范围、验收标准、约束、外部接口、兼容要求或其他需求级输入，不要在设计阶段直接用聊天结论替代已批准规格；应回到 `ahe-workflow-starter`，必要时回流到 `ahe-specify` / `ahe-spec-review` 刷新上游工件后再继续设计
- 只有当它属于实现上下文级澄清，而不改变已批准需求边界时，才可在当前设计轮次中补充确认

### 2. 了解最少但必要的技术上下文

阅读最少但必要的技术上下文：

- 现有架构或项目布局
- 当前框架、依赖和运行时约束
- 已知的部署、集成或兼容限制
- 可能复用的现有模块、边界和接口形态

此时不要提前进入实现规划。

### 3. 提出 2 到 3 个候选方案，并形成结构化决策

在最终选定设计前，至少比较两个可行方案。

对每个候选方案，至少说明：

- 它如何工作
- 适合当前规格的原因
- 主要优点
- 主要缺点
- 对约束和非功能需求的影响
- 关键风险

推荐方案时，不只给出“选哪个”，还要给出结构化决策记录：

- 决策背景
- 被考虑的候选方案
- 选定方案
- 主要收益
- 主要代价
- 风险与缓解思路

如果当前是因为 `ahe-design-review` 返回“需修改”或“阻塞”而重新进入本 skill：

- 先读取评审结论中的发现项、薄弱点和修订建议
- 优先修复阻塞任务规划的问题
- 只针对仍然影响设计定稿的问题向用户补充确认
- 不要把已确认且未受影响的设计内容全部重做

### 4. 组织设计内容，保持 WHAT / HOW / TASKS 边界

写设计文档时，明确区分三层内容：

- 规格层：回答做什么，不在这里重复展开
- 设计层：回答如何实现
- 任务层：回答如何分步实施，属于 `ahe-tasks`

设计文档可以说明“哪些部分已经足以支撑任务规划”，但不要直接下沉成完整任务拆解、DoD 清单或执行步骤。

### 5. 编写设计文档

若 `AGENTS.md` 为当前项目声明了设计模板、章节骨架或记录要求，优先遵循这些约定。

若 `AGENTS.md` 未提供设计模板覆盖，则使用以下默认结构：

```markdown
# <主题> 实现设计

- 状态: 草稿
- 主题: <主题>

## 1. 概述

## 2. 设计驱动因素

## 3. 需求覆盖与追溯

## 4. 候选方案

## 5. 选定方案与关键决策

## 6. 架构视图

## 7. 模块职责与边界

## 8. 数据流、控制流与关键交互

## 9. 接口与契约

## 10. 非功能需求与约束落地

## 11. 测试策略

## 12. 风险、待定问题与任务规划准备度
```

默认应先遵循 `AGENTS.md` 中的路径映射与模板覆盖；若未提供，再使用本 skill 的默认路径和上述章节骨架。

编写要求：

- 区分需求与设计
- 清晰定义边界和职责
- 用追溯视角说明关键需求被哪些模块、流程或接口承接
- 解释为什么所选方案适合该规格
- 说明设计怎样支撑后续任务规划，但不要写成任务计划
- 提供足够支撑任务拆解的细节，但不要写成逐行编码说明

### 6. 最少必要设计视图

对 trivial 设计，可用结构化文字表达清楚。

对非 trivial 设计，至少提供最少必要视图帮助评审理解。优先使用 Mermaid。通常至少包括以下 2 到 3 类中的若干项：

- 逻辑架构视图：说明层次、模块或边界关系
- 组件 / 接口关系视图：说明关键依赖与协作
- 关键流程 / 关键交互视图：说明主路径与关键分支
- 数据视图：当持久化结构、数据边界或核心实体关系会影响设计时补充

不要为了“看起来完整”而堆图；每张图都应服务于理解关键设计决策。

### 7. 强化接口与契约章节

`接口与契约` 章节至少要回答：

- 关键输入 / 输出是什么
- 主要接口边界在哪里
- 错误、异常或兼容性语义如何处理
- 外部依赖或协议有哪些限制
- 为什么当前契约已经足以支撑后续任务规划

不要只写“提供一个接口”或“模块之间通信”，而不说明约束和边界。

### 8. 非功能需求与测试策略不能漂浮

当规格中存在性能、可靠性、安全、隐私、可维护性、兼容性等要求时，设计中要明确：

- 它们如何影响方案选择
- 哪些模块、边界或机制负责满足这些要求
- 后续应如何验证，而不是只写“后面测试即可”

测试策略保持设计层抽象即可，重点说明：

- 设计的可测试性来自哪里
- 关键风险点应由哪些测试层面覆盖
- 有没有必须在设计阶段冻结的测试相关决策

### 9. 评审前自检

交给 `ahe-design-review` 前，请确认：

- 关键需求能追溯到相应模块、流程或接口
- 设计不是规格复述，也不是实现说明
- 至少比较了两个可行方案，并解释了选定理由
- 架构是连贯的，而不是组件清单
- 关键边界和模块职责清晰
- 关键接口与契约足够明确，能支撑后续任务规划
- 约束和非功能需求确实影响了设计决策
- 测试策略体现了设计可测试性和主要验证方向
- 风险和待定问题被清楚标出
- 所有新增行为、接口、数据持久化或运营流程，要么能追溯到已批准规格，要么被显式标为待确认 / 范围外，而不是直接写成既定设计
- 文档已经足以让 `ahe-design-review` 按其检查清单展开评审

### 10. 交给 reviewer subagent 执行设计评审

准备好之后，不要在父会话里直接按 `ahe-design-review` 的检查清单内联评审。

正确做法是：

1. 将设计草稿保存到约定路径
2. 组装最小 design review request，只包含：
   - 设计文档路径
   - 已批准规格路径
   - 本次主题 / topic
   - 做判断所需的最小辅助上下文路径
   - 预期 review 记录路径
3. 启动独立 reviewer subagent，并在该 subagent 中调用 `ahe-design-review`
4. 由 reviewer subagent 负责：
   - 读取 `ahe-design-review`
   - 写 review 记录
   - 回传结构化摘要
5. 父会话读取 reviewer 返回结果后再继续：
   - 若结论为 `通过`，由父会话向真人展示评审结论并发起批准确认
   - 若结论为 `需修改` 或 `阻塞`，携带 reviewer 的关键发现项回到本 skill 修订

不要在 `ahe-design` 阶段请求真人批准设计；真人确认只发生在 `ahe-design-review` 返回“通过”，且该结果已经由 reviewer subagent 回传给父会话之后。

若项目尚未形成固定 review 记录格式，默认由 `ahe-design-review` 使用：

- `templates/review-record-template.md`

交接前请同步：

- 将设计草稿保存到约定路径
- 在 `task-progress.md` 中将 Current Stage 更新为能唯一映射到“设计草稿已完成、下一步进入 `ahe-design-review`”的规范阶段名；不要写成含糊的自然语言状态
- 在 `task-progress.md` 中将 Next Action Or Recommended Skill 更新为 `ahe-design-review`

输出应使用：

```markdown
设计文档草稿已起草完成，下一步应派发独立 reviewer subagent 执行 `ahe-design-review`。

推荐下一步动作或 skill: `ahe-design-review`
```

如果这是基于评审意见修订后的版本，也使用同样交接方式重新派发 reviewer subagent 执行 `ahe-design-review`。

注意：只有 `ahe-design-review` 返回“通过”后，才进入真人确认；真人确认通过后才能进入 `ahe-tasks`。

## 反模式

- 把设计文档写成实现伪代码
- 只是复制需求规格，没有加入设计决策
- 只给一个方案，不讨论权衡
- 没有说明为什么选定当前方案
- 在设计文档里直接开始拆任务
- 只写模块名，不写边界、交互和契约
- 忽略非功能需求和约束

## 完成条件

只有在存在一份可评审的设计文档，并且已准备好交给 `ahe-design-review` 时，这个 skill 才算完成。
