# 嵌入式 SOC AI 推理微服务 Analysis Pack

- 状态: 首轮草稿
- 主题: 嵌入式 SOC 上新增 AI 推理微服务进程
- 更新时间: 2026-04-09
- 来源: 基于用户口述场景的首轮 `se-*` workflow 演练

## 1. 问题陈述

当前要分析的主题是：在嵌入式 SOC 系统上新增一个独立微服务进程，对外提供 AI 推理服务。该服务需要：

- 订阅系统状态
- 接收其他服务通过 TCP 发送的请求
- 与下层定义的交互接口协作
- 在上层考虑 OpenSSL 封装
- 设计并发请求的处理机制
- 通过下层 AI 推理框架执行实际推理

这类需求的难点不在“会不会写代码”，而在于：

- 新人 SE 往往不知道哪些点应该先问清楚
- 很容易把团队口头要求误当成既定设计
- 容易跳过服务边界、安全边界和并发模型的前提判断

## 2. 当前范围

本轮分析必须覆盖：

- 新微服务的职责边界
- 上下游关系与关键交互
- OpenSSL 封装可能承担的职责
- 并发模型候选方案与推荐方向
- DFX 风险
- AR 分解思路
- 参考代码量估算

本轮明确不做：

- 详细字段级协议定义
- 真实业务代码仓中的组件复用确认
- 具体框架 API 映射
- 精确工时评估

## 3. 参与者与系统边界

### 调用方

- 板端其他服务，通过 TCP 请求 AI 推理能力

### 服务方

- 新增的 AI 推理微服务进程

### 下游依赖

- 下层定义的接口层 / 交互组件
- 下层 AI 推理框架
- 系统状态发布方

### 关键边界

- 传输与会话安全边界：TCP / TLS / OpenSSL 封装层
- 协议 / 请求语义边界：请求解析、校验、错误语义
- 调度边界：队列、线程池、并发控制
- 推理边界：下层框架适配与资源保护

## 4. 关键场景

### 场景 1：正常推理请求

1. 其他服务建立 TCP 连接
2. 微服务完成连接接入和安全处理
3. 请求被解析并校验
4. 服务读取当前系统状态快照
5. 请求进入调度队列
6. 推理执行单元调用下层框架
7. 返回成功结果或明确失败结果

### 场景 2：系统状态不允许受理

1. 服务收到推理请求
2. 当前系统状态表明资源不足、模型不可用或服务降级
3. 服务快速拒绝、排队或返回忙碌状态

### 场景 3：高并发 / 资源受限

1. 多个请求同时到达
2. 服务需要对连接数、队列深度和执行并发做限制
3. 超过阈值时需要背压、拒绝或超时处理，而不是无限堆积

### 场景 4：依赖层异常

1. TLS 握手失败、证书无效或连接异常
2. 下层推理框架忙碌、失败或线程不安全
3. 服务需要区分接入层错误、协议层错误和推理层错误

## 5. 约束与依赖

- 运行在嵌入式 SOC 环境，资源受限
- 需要与现有系统状态发布机制集成
- 下层组件会定义部分接口或交互契约
- 需要处理 TCP 并发请求
- 需要对安全边界有明确解释，否则上层封装 OpenSSL 的职责会与下层接口层冲突

## 6. 已确认事实

- 需要新增独立微服务进程对外提供 AI 推理服务
- 该服务需要订阅系统状态
- 请求通过 TCP 到达
- 下层会设计至少一部分交互接口
- 上层存在“封装 OpenSSL”的要求，但原因和边界尚未澄清
- 需要处理并发请求
- 推理执行依赖下层框架接口

## 7. 假设

- 外部调用是同步请求-响应模型，而不是异步作业模型
- OpenSSL 封装主要用于 TLS / 证书 / 会话策略统一，而不是实现业务协议
- 下层推理框架的并发能力有限，不能默认按连接数横向扩线程
- 系统状态会影响请求受理或调度策略
- 当前更关注第一版安全可落地，而不是极限吞吐

## 8. 开放问题

- TCP 契约由谁定义到什么粒度？仅定义载荷格式，还是同时定义传输安全和错误语义？
- OpenSSL 封装到底是平台硬约束，还是团队经验性建议？
- 下层推理框架是否线程安全？允许多少并发推理？
- 状态订阅的刷新频率、一致性要求和影响范围是什么？
- 请求在排队中遇到状态变化时应如何处理？
- 是否需要 mTLS、证书轮换和客户端身份校验？

## 9. 研究问题

- 上层为什么要封装 OpenSSL，它自然应负责哪些职责？
- 在嵌入式 / SOC 环境里，推理服务应采用什么并发模型？
- 连接接入、队列管理和推理执行如何分层最稳妥？
- 如何在资源受限情况下设计背压与过载保护？
- 需要哪些 DFX 设计点，才能让方案足够可评审？

## 10. 仓库调研摘要

本轮演练没有绑定真实的业务代码仓，因此仓库调研结论目前只能记录为“证据缺失”，不能假装已有代码依据。

当前缺少的仓库级证据包括：

- 是否已有 TCP / TLS 适配层可复用
- 是否已有统一状态订阅机制
- 是否已有线程池、任务队列或调度组件
- 是否已有 AI 推理框架适配层
- 现有部署 / 启动 / 监控约束

这意味着：本 Analysis Pack 可以用于首轮方案讨论，但不能替代对真实业务代码仓的二轮调研。

## 11. 外部研究摘要

### OpenSSL 封装边界

基于 OpenSSL 3.6 manpages，对上层封装最有说服力的职责包括：

- `SSL_CTX` 级安全策略与生命周期管理
- 每连接 `SSL` 对象的创建、握手和释放
- 证书校验、信任库加载与校验深度
- 客户端侧 hostname 验证与 SNI 设置
- Session / resumption 策略
- `SSL_get_error()` 相关错误归一化

这说明“封装 OpenSSL”在工程上不是多此一举，它通常意味着把 TLS 策略、证书管理和会话错误处理集中到统一层，而不是散落在业务逻辑里。

### 事件循环与 worker pool

外部资料和多 agent 分析都指向同一原则：

- 事件循环只负责非阻塞 I/O 和快速调度
- 耗时任务不能直接放在 event loop 中
- 队列和执行池必须有明确上限
- 过载时应优先背压 / 早拒绝，而不是无限排队

### 边缘推理服务的排队与资源控制

外部研究的高层启发包括：

- 用有界队列而不是无界队列
- 对请求执行设置显式并发上限
- 监控队列深度、排队时间、执行耗时、拒绝数等核心指标
- 资源受限设备应优先保证可预测退化，而不是追求理论最大吞吐

## 12. 候选方案

### Option A. 同进程同步接入 + 宽 worker pool

- 核心思路: 接入、协议、安全和推理都在单进程内完成，请求直接分发给较宽的 worker pool 并调用下层框架
- 优点:
  - 结构直观
  - 客户端模型简单
  - 框架线程安全时吞吐潜力高
- 代价:
  - 对线程安全和资源余量假设较重
  - 更容易在高并发下出现内存和调度失控
- 风险:
  - 下层框架若并发能力不足，问题会迅速暴露

### Option B. 事件驱动接入 + 有界队列 + 小执行池

- 核心思路: 接入层负责 TCP / TLS / 协议解析和快速校验，任务进入有界队列，由很小的执行池调用下层框架
- 优点:
  - 资源行为更可控
  - 更适合嵌入式设备
  - 更容易做背压、限流和过载处理
  - 不强依赖下层框架支持高并发
- 代价:
  - 队列超时、取消和拒绝语义需要额外设计
  - 峰值吞吐不一定最高
- 风险:
  - 如果队列策略设计不好，会出现队头阻塞

### Option C. Gateway 与推理引擎逻辑进一步解耦

- 核心思路: 接入层专门处理 TCP / TLS / 协议和状态门控，内部再通过 IPC 或内部队列调用独立推理执行单元
- 优点:
  - 隔离性最好
  - 安全边界更清晰
  - 适合下层框架质量不稳定或长期架构演进
- 代价:
  - 复杂度更高
  - 内存和部署成本更大
- 风险:
  - 对第一版嵌入式落地可能过重

## 13. 方案矩阵

| 维度 | Option A | Option B | Option C |
| --- | --- | --- | --- |
| 需求匹配度 | 中 | 高 | 高 |
| 资源可控性 | 低 | 高 | 中 |
| 实现复杂度 | 中 | 中 | 高 |
| 安全边界清晰度 | 中 | 高 | 高 |
| 适合第一版嵌入式落地 | 低 | 高 | 中 |
| 对下层线程安全依赖 | 高 | 中 | 中 |
| 可观测与背压设计空间 | 中 | 高 | 高 |

## 14. 推荐方向

### 推荐方案

推荐优先采用 `Option B`：

- 单进程部署
- 事件驱动接入
- 有界队列
- 很小的推理执行池
- 清晰拆分 `TLS / 安全适配`、`协议接入`、`状态快照`、`调度与背压`、`推理适配`

### 推荐理由

- 它最符合嵌入式 SOC 的资源约束现实
- 它不要求下层推理框架一开始就具备强并发能力
- 它把“并发接入”和“并发推理”解耦，更容易保护底层资源
- 它为后续扩展到更强隔离模型留下了边界

### 为什么不优先选其他方案

- 不优先选 `Option A`，因为它过度依赖下层框架线程安全和资源余量
- 不优先选 `Option C`，因为第一版代价偏高，且当前没有证据表明必须做进程内进一步拆分

## 15. 剩余风险与开放问题

- 没有真实业务代码仓证据，复用度和现有约束尚未确认
- 还缺 TCP 契约、TLS 要求和错误码 ownership
- 下层推理框架并发能力未知
- 状态变化期间的请求语义仍需团队对齐
- 是否需要 mTLS、证书轮换和审计日志仍待确认

## 16. 研究依据

- OpenSSL 3.6 manpages：`SSL_CTX_new`、`SSL_CTX_set_verify`、`SSL_set1_host`、`SSL_CTX_load_verify_locations`、`SSL_do_handshake`、`SSL_get_error`、`SSL_shutdown`、`SSL_CTX_set_session_cache_mode`
- 事件循环与 worker pool 的通用实践：event loop 不做阻塞任务，重活下沉到 bounded workers，并配背压控制
- 边缘推理服务的通用启发：优先显式队列上限、拒绝策略、队列深度与执行延迟监控
