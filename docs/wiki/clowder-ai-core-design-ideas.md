# Clowder AI 核心设计思想

## 一句话

Clowder AI 的本质不是“把多个模型接进一个聊天界面”，而是把 **身份、协作、记忆、纪律、审计、环境隔离** 从零散 prompt 和人工操作里抽出来，做成一个真正的 **Agent 团队协作平台层**。

---

## 1. 产品先回答“人和 Agent 是什么关系”

Clowder 不是先从工具集成开始，而是先定义产品哲学：

- 人不是 router，也不是手工中继器
- Agent 不是一次性工具调用器，而是长期协作团队成员
- 系统目标不是“帮人调模型”，而是“让人带着一支 AI 团队把想法做成产品”

这意味着它的设计起点不是：

- 哪个模型更强
- 哪个 SDK 更方便
- 哪个 prompt 更聪明

而是：

- 人负责愿景、判断和关键决策
- Agent 负责执行、协作、复查和持续积累
- 平台负责把团队协作稳定下来

核心心法：

**先定义协作关系，再定义工具链。**

---

## 2. 明确三层边界：Model / Agent CLI / Platform

Clowder 最关键的设计动作，是把三层责任拆开：

### Model

负责：

- 理解
- 推理
- 生成

不负责：

- 长期记忆
- 团队纪律
- 协作状态

### Agent CLI

负责：

- 工具调用
- 文件操作
- 命令执行

不负责：

- 团队编排
- 审计
- review 制度

### Platform

负责：

- identity
- routing
- threads
- memory
- evidence
- discipline
- audit

核心心法：

**模型决定上限，平台决定下限。**

也就是说，系统可靠性不应建立在“某次模型刚好够聪明”上，而应建立在平台定义的稳定协作结构上。

---

## 3. 平台不是 API 外壳，而是控制面

Clowder 的后端不是普通 CRUD API，而更像一个控制面：

- 管 session / thread / queue
- 管 agent registry 和 provider adapter
- 管 memory / evidence / workflow discipline
- 管消息桥接、回调、终端、项目、工作区等运行态对象

这背后的设计思想是：

- 多 Agent 系统的真相源不能放在聊天上下文里
- 协作状态需要被平台显式持有
- “系统如何工作”必须是可恢复、可观察、可追溯的

核心心法：

**一旦目标是团队级 Agent 协作，后端就会天然演化成控制面。**

---

## 4. 用 Adapter 吃掉模型 / 宿主差异，而不是复制工作流

Clowder 没有为 Claude、GPT、Gemini 各写一套 workflow，而是做了统一适配：

- 上层统一路由
- 中层按 provider 做 agent service
- 下层通过 CLI 子进程、流解析、事件归一化接入不同供应商

这说明它把差异定义成：

- 适配问题

而不是：

- 工作流分叉问题

核心心法：

**共享 workflow 逻辑，隔离宿主差异。**

如果宿主差异扩散进主流程，系统很快会变成：

- Claude 一套逻辑
- GPT 一套逻辑
- Gemini 一套逻辑

那就不是平台，而只是多套脚本并排存在。

---

## 5. 共享契约必须独立成层

Clowder 很强调 shared layer：

- types
- schemas
- registry
- utils
- MCP server

这说明它不希望以下内容散落在各处：

- 状态枚举
- 路由协议
- 工具协议
- 共享术语

核心心法：

**只要概念被多个包、多个端、多个 agent 复用，就应该抽成共享契约层。**

这比“大家约定好字符串写法”要稳健得多。

---

## 6. MCP / 工具平面应独立于具体模型

Clowder 把 MCP 看成平台级工具面，而不是某个模型的附属插件。

因此它的设计不是：

- Claude 有工具，别的模型想办法跟上

而是：

- 平台先拥有独立的工具能力
- 再通过 callback / adapter / bridge 让不同 agent 访问

核心心法：

**工具能力属于平台，不属于某一个模型。**

---

## 7. 治理不是 prompt 附件，而是正式工件系统

Clowder 的 `VISION`、`SOP`、`ROADMAP`、`Feature Docs`、`ADRs` 很关键，因为它们不是装饰性文档，而是在共同规定：

- 角色是什么
- 流程怎么走
- 什么叫完成
- 什么叫放行
- 什么能进入 release

这说明它把治理写成了可读取、可同步、可审视的工件，而不是藏在聊天历史里。

核心心法：

**先把治理写成工件，再让 Agent 读取这些工件。**

这是从“会用 AI”走向“能运营 Agent 系统”的关键跃迁。

---

## 8. 运行环境和隔离纪律本身就是产品能力

Clowder 的一个强信号，是它把 runtime / alpha / worktree 区分成不同环境语义，并且写出明确纪律：

- 哪个环境可以开发
- 哪个环境负责验收
- 哪个环境不能随意重启
- 哪个环境用于 release / sync / provenance

很多团队做 Agent 系统时只谈：

- prompt
- tool call
- review

却不谈：

- 端口
- 进程
- worktree
- runtime 单实例
- 回归验收环境

Clowder 说明：

**环境纪律不是运维补丁，而是协作系统可靠性的一部分。**

---

## 9. Human-in-the-loop 不是可选项，而是结构中心

Clowder 并没有追求“全自动自治团队”，而是明确保留人的中心位置：

- 人定义愿景
- 人做关键审批
- 人确认方向
- 平台维护纪律
- Agent 执行和协作

这和很多“Boss Agent 全权调度一切”的想象不同。

核心心法：

**不是取消人的判断，而是让人的判断更聚焦在真正高价值的地方。**

---

## 10. Clowder 真正可迁移的 6 条设计原则

1. 先定义平台边界，再定义技能和流程。
2. 先定义协作对象，再讨论模型能力。
3. 先收敛共享契约，再适配不同宿主。
4. 先把治理写成工件，再让系统读取治理。
5. 先把运行环境分层，再谈多 Agent 自动化可靠性。
6. 先把平台做成控制面，再让 UI、CLI、Connector 挂上去。

---

## 11. 不该照抄的地方

Clowder 很有参考价值，但不等于所有做法都该直接复制：

- 它的系统边界很宽，复杂度高
- 它对外部 CLI 行为有长期依赖，适配层天然脆弱
- 它的本地多进程脚本化环境接入门槛不低
- 它的公开验证面不一定等于完整系统真实能力面

所以更好的吸收方式不是“复刻产品形态”，而是吸收它的结构思想：

- 平台化
- 契约化
- 治理工件化
- 环境纪律化

---

## 12. 最终总结

如果把 Clowder AI 的核心设计思想压缩成四句话，就是：

1. **多 Agent 协作的真相源必须在平台，不在聊天上下文。**
2. **多模型差异应该被 adapter 吸收，而不是让 workflow 分叉。**
3. **治理、review、门禁、release 语义必须写成系统工件。**
4. **环境隔离和运行纪律本身就是 Agent 平台能力。**

所以，Clowder 最值得学习的不是“它支持很多模型”，而是：

**它把 Agent 团队协作从经验和 prompt，提升成了一套可运行、可治理、可持续演进的系统设计。**
