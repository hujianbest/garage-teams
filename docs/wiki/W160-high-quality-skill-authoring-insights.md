# W160: High-Quality Skill Authoring Insights

- Wiki ID: `W160`
- 状态: 参考
- 日期: 2026-04-15
- 定位: 记录对公开 `skill authoring` 资料的交叉阅读结果，提炼“高质量 skill”在触发、结构、验证与演化上的共识，供 `Garage` 后续设计、编写和评审 skill 时参考；默认作为参考资料，不作为当前主线真相源。
- 关联文档:
  - `docs/wiki/W110-skills-refer.md`
  - `docs/principles/skill-anatomy.md`

这篇文章不试图回答“一个 `SKILL.md` 该有哪些段落”这么窄的问题，而是回答更关键的问题：

**为什么有些 skill 会被稳定触发、低成本加载、在真实任务里反复奏效；而另一些 skill 只是把经验写长，最后变成上下文噪音。**

## 文章来源声明

本文基于 2026-04-15 对公开网页资料的阅读与交叉归纳形成，不是对任何单篇文章的全文翻译，也不是对某一份官方文档的改写版。判断优先级上：

- `Anthropic` 官方文档作为一手依据。
- `Agent Skills` 规范站与社区作者文章作为二手实践补充。
- 文中“对 Garage 的启发”属于本文作者基于上述资料做出的本地化归纳，不等同于外部来源原话。

## 一、结论先行

把这些资料压成一句话：

**高质量 skill 不是“写得更长的提示词”，而是一个可被正确触发、只在需要时加载、能把真实经验稳定复用出来的运行时能力包。**

进一步展开，外部资料反复收敛到下面 6 条：

1. `skill` 的第一设计面不是正文，而是 `name + description` 这组触发接口。
2. `SKILL.md` 的职责是给出执行主链，而不是把所有背景知识一次性塞进上下文。
3. 真正高价值的内容不是通用常识，而是模型本来不知道、但做对任务又离不开的项目经验、边界条件和常见误判。
4. 好的 skill 会根据任务脆弱度控制“自由度”，该给启发时给启发，该给硬步骤时给硬步骤。
5. 复杂 skill 不能只有建议，还要有 checklist、模板、验证环和可执行脚本。
6. skill 应该像软件一样通过真实任务和评测迭代，而不是“写完即完成”。

## 二、什么样的 skill 才算高质量

如果只保留一个判断标准，那就是：

**它是否让 agent 在真实任务中，比“没有这个 skill”时更稳定地做出正确选择。**

这意味着 skill 质量不应只看“文档是否完整”，而要看下面四件事是否同时成立：


| 维度    | 高质量信号            | 低质量信号                 |
| ----- | ---------------- | --------------------- |
| 触发    | 该触发时能触发，不该触发时不乱入 | 只有手动点名才会用，或经常误触发      |
| 上下文成本 | 主体精简，细节按需展开      | 把大段知识一次性塞进 `SKILL.md` |
| 执行稳定性 | 有默认路径、验证点、失败回路   | 只有泛化建议，没有过程约束         |
| 可演化性  | 能根据真实失败案例快速修补    | 依赖作者主观想象，长期不回收反馈      |


从这个角度看，高质量 skill 更像一个小型运行时产品，而不是一份“说明文档”。

## 三、第一设计面其实是触发接口

`Anthropic` 官方文档反复强调，系统启动时预加载的是所有 skill 的元数据，而不是所有 skill 的正文。因此真正决定 skill 会不会被挑中的，是 `name` 和 `description`。

这带来一个很重要的设计转向：

- `description` 不是摘要，而是路由面。
- 它必须同时回答“这项能力做什么”和“什么时候该调用它”。
- 它要包含足够具体的触发词、场景词和任务上下文，而不是笼统写成“帮助处理文档”或“处理数据”。
- 官方还明确要求用第三人称，因为这段文字会被注入系统提示中。

这也是为什么多篇资料都强调命名与描述不应含糊：

- 名称要能一眼看出动作或能力边界，`gerund` 风格通常更稳定。
- 描述要说明触发时机，而不只是复述 skill 名称。
- 对常见但容易漏触发的场景，要把用户可能使用的自然表述写进描述里。

对 skill 作者而言，这意味着一个常见误区必须被纠正：

**不是先把正文写好，再顺手补一个 `description`；而是先把触发边界写准，正文才有机会被读到。**

## 四、正文只负责“执行主链”，不要把它写成资料仓库

官方文档和 `Agent Skills` 规范都给出同一个方向：`SKILL.md` 应该保持精简，并通过 `progressive disclosure` 把深层资料放到独立文件里按需读取。

这背后不是排版偏好，而是上下文经济学：

- 所有正文一旦被加载，就会和会话历史、系统提示、其他上下文一起竞争注意力。
- skill 最值钱的不是“内容多”，而是“把当前任务必需的信息放在最前面”。
- 如果正文同时承担概览、规范、样例库、脚本说明、故障排查和长篇参考手册，模型很容易在不相关信息里分神。

比较健康的结构通常是：

- `SKILL.md` 只保留主链、默认路径、关键 gotchas、必要模板入口。
- `references/` 放大篇幅参考材料。
- `examples/` 或独立示例文件放输入输出样例。
- `scripts/` 放可执行、可验证、可重复使用的确定性工具。
- 文件名本身要具备语义，帮助 agent 快速判断该读什么。

这里还有几个非常一致的反模式：

- 给出太多平级选项，却不指定默认做法。
- 让引用关系层层嵌套，导致 agent 读文件路径时迷路。
- 在正文里解释大量模型本来就知道的常识。
- 用“文件 1 / 文件 2 / doc2”这类弱语义命名降低可发现性。

如果压缩成一句规则，就是：

**把 skill 正文写成“执行导航”，不要写成“资料倾倒”。**

## 五、skill 的真正价值来自真实经验，而不是空泛最佳实践

`Agent Skills` 站点最有价值的一点，是把 skill 的素材来源讲得很直白：好 skill 不该主要从“通用最佳实践文章”里生成，而该从真实任务、真实工件和真实失败中提炼。

这类资料特别适合沉淀成 skill：

- 你在实际协作里反复纠正 agent 的地方。
- 团队内部 runbook、style guide、API 约束、schema、配置和审批要求。
- code review 里高频重复出现的意见。
- issue / incident / patch 里已经暴露过的真实坑。
- 某个任务做对后可复用的输入输出结构与步骤顺序。

这条原则非常关键，因为它直接决定 skill 到底是在教“常识”，还是在注入“专有经验”。

一个 skill 最有价值的段落，往往不是“概述”，而是这些内容：

- `gotchas`
- 不一致命名对照
- 默认工具选择
- 失败后回退路径
- 哪些边界情况不能靠模型自由发挥

换句话说，skill 不是把公开知识再说一遍，而是把“如果没人提醒，模型大概率会在这里犯错”的那部分经验显式化。

## 六、真正能落地的 skill，一定会设计自由度、工作流和验证环

官方资料没有把所有任务都按同一种方式写。相反，它明确提出：要根据任务脆弱度决定 instruction 的刚性。

这实际上是在回答一个常见问题：

**什么时候应该写原则，什么时候应该写死步骤？**

资料给出的共识大致是：

- 多种路径都可能成立、且结果允许一定弹性时，给高自由度的原则与启发。
- 有推荐模式但允许少量变化时，给模板、伪代码或带参数脚本。
- 一旦任务脆弱、顺序敏感、失败代价高，就要给低自由度的明确步骤。

所以高质量 skill 往往不是“只有一段说明”，而会显式提供以下结构件：

- checklist：防止多步任务中漏步骤。
- workflow：让 agent 知道当前处于哪一阶段。
- template：把目标输出结构写具体。
- validation loop：形成“执行 -> 检查 -> 修正 -> 再检查”的闭环。
- plan-validate-execute：先产出中间计划或中间文件，再对照真相源验证，再执行。

`Anthropic` 在 agent 设计文章里进一步把这个思路推高了一层：真正可靠的 agent system 倾向于使用简单、透明、可组合的模式，而不是追求抽象上很花哨的复杂框架。

落到 skill 写作上，这意味着：

- 用清晰的顺序替代含混的“你应该综合考虑”。
- 用可验证的中间产物替代“一步到位产出最终结果”。
- 用工具和脚本的清晰接口替代让模型临时发明执行方式。

## 七、skill 质量的上限，取决于你是否把它当成要迭代的资产

官方最佳实践很明确地提出了 `evaluation-driven development`：先观察没有 skill 时模型会怎样失败，再写最小 skill 解决这些具体失败，然后在真实任务里继续迭代。

这比“闭门写规范”强得多，因为它把 skill 开发从想象题变成了证据题：

1. 先跑代表性任务，不带 skill，记录失败和缺口。
2. 用这些缺口构造评测场景。
3. 建立 baseline。
4. 只写足以解决这些缺口的最小说明。
5. 再跑、再观察、再收敛。

这里有两个很容易被忽略的点：

- 评测的目标不只是看任务是否成功，还要看 skill 是否在正确场景被触发。
- 真实使用中的文件导航、遗漏引用、误读结构，也都是 skill 设计问题的一部分。

因此，skill 的维护不该只看“内容有没有更新”，还应该看：

- 有没有漏触发。
- 有没有误触发。
- agent 是否总在读不该先读的文件。
- 哪些内容从不被访问，说明可能应该删除或降级。
- 哪些规则被反复忽略，说明位置、措辞或结构还不够强。

## 八、对 Garage 的直接启发

如果把这些外部资料翻译成 `Garage` 里可执行的作者原则，我会收束成下面 8 条：

1. 先区分“总是成立的仓库规则”和“只在特定任务下激活的 skill 经验”，不要把两者混写。
2. 把 `name + description` 当成一级产品面来写，因为它本质上是 skill 路由器的输入接口。
3. `SKILL.md` 只保留默认执行主链、关键红旗、必要模板入口和少量最有价值的 gotchas。
4. 长参考、规范附录、样例库和大段 API 细节默认外置到独立文件，按需读取。
5. 对高风险 skill，默认要求 checklist、验证环和中间产物，而不是只给原则。
6. 把团队已经纠正过 agent 的地方写进 skill，而不是重复模型已经懂的公开常识。
7. skill review 不该只看“写得是否完整”，还要看“是否能正确触发、是否过度耗费上下文、是否能支持验证”。
8. skill 应该持续吸收真实任务中的失败案例，像代码一样维护版本质量，而不是写完封存。

更进一步说，`Garage` 如果要构建长期可演化的 skill 体系，最值得吸收的并不是某一种固定目录模板，而是下面这条更深的共识：

**skill 的本质不是静态文档资产，而是面向运行时检索、执行和验证设计过的信息结构。**

这也是为什么“怎么写好一个 skill”的答案，最终会落回系统设计，而不只是文案写作。

## 九、可直接复用的 skill 写作检查表

如果后续要快速评估一个新 skill 是否达到“可用线”，可以先问下面这些问题：

- 只看 `name + description`，系统能知道它何时该触发吗？
- 去掉冗余背景后，`SKILL.md` 还剩下多少真正不可替代的信息？
- 当前 skill 是否明确给出了默认路径，而不是列了一堆可选项？
- 多步任务是否有 checklist 或 workflow？
- 失败后是否有验证环和回退路径？
- 长资料是否已拆到独立文件，并且引用关系清晰可导航？
- skill 是否沉淀了真实项目 gotchas，而不是复述公开常识？
- 有没有用真实任务做过 baseline 和改进后的对照？

如果这些问题里有一半答不上来，通常说明这个 skill 还停留在“文档草稿”，还没有成为“可稳定运行的能力资产”。
