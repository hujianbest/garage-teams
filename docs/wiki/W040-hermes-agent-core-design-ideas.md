## 一句话

Hermes Agent 的本质不是“支持很多模型和很多聊天入口的 AI 助手”，而是把 **长期身份、跨会话状态、统一工具面、多入口连续性、安全护栏和运行韧性** 收束成一个真正的 **personal agent runtime / personal agent OS**。

---

## 1. 先把 Agent 设计成长期存在的主体

Hermes 的起点不是“这一轮怎么回答得更好”，而是“这个 agent 如何长期存在”：

- 它会跨会话保留状态
- 它会在不同入口之间延续同一个身份
- 它会把经验沉淀成 memory 和 skill
- 它会在后续任务里继续使用过去积累的知识

这意味着 Hermes 不是把 agent 当作一次性请求处理器，而是当作一个持续运行、不断积累上下文的实体。

核心心法：

**如果系统目标是长期协作，就不能只优化单轮回答质量，而要优先设计长期连续性。**

---

## 2. 一个核心循环，承载所有入口

Hermes 很关键的结构判断，是让 CLI、gateway、ACP、batch 等不同入口共享同一个核心 agent loop。

这背后的想法是：

- 平台差异应该尽量停留在边缘
- 核心行为语义必须统一
- 工具调用、状态恢复、安全策略、持久化不应该因为入口不同而分叉

所以它不是：

- CLI 一套 agent 逻辑
- Telegram 一套 agent 逻辑
- 编辑器接入再来一套 agent 逻辑

而是：

- 外层入口负责接入
- 核心运行时负责执行

核心心法：

**多入口系统想保持一致性，必须先有高于入口的统一 agent core。**

---

## 3. Prompt 不是文案，而是运行时契约

Hermes 对 prompt 的处理很像在维护一份 runtime contract，而不是在拼接一段好看的系统提示词。

它隐含的设计判断是：

- prompt 需要稳定，不应每轮随意漂移
- prompt 需要可组合，不同能力块应清晰装配
- prompt 需要可扫描，外部上下文文件不能无条件注入
- prompt 需要与缓存策略协同，否则会破坏成本和行为稳定性

这说明 Hermes 眼里，prompt 不只是“告诉模型怎么做”，而是：

- 身份约束
- 执行纪律
- 记忆使用规则
- 工具使用规则
- 上下文注入边界

核心心法：

**Prompt 应该被工程化成稳定契约，而不是散落在各处的自由文本。**

---

## 4. 长期知识必须分层：Memory / Session / Skill

Hermes 很成熟的一点，是它没有把所有长期信息混进一个“记忆系统”。

它实际上拆成了三层：

### Memory

存稳定事实：

- 用户偏好
- 环境约束
- 可长期复用的稳定信息

### Session

存发生过什么：

- 历史对话
- 问题与处理过程
- 工具调用上下文
- 可搜索的会话证据

### Skill

存可复用的方法：

- 工作流
- 程序性知识
- 复杂任务的可重放做法

这三者混在一起时，系统通常会出现：

- memory 垃圾化
- transcript 不可检索
- workflow 经验无法沉淀

核心心法：

**稳定事实、历史证据、可复用方法，必须是三种不同的资产。**

---

## 5. 工具不是零散能力，而是统一控制面

Hermes 的工具系统不是“哪里要用就加一个函数”，而是被当作一个统一控制面来治理。

它体现出来的思想是：

- 工具 schema、handler、availability 需要统一管理
- 工具应该属于 agent 的能力面，而不是某个入口的临时附件
- 平台之间尽量共享同一套核心工具语义
- toolset 是产品能力边界，不只是代码标签

这使 Hermes 能做到：

- CLI 和消息入口默认共享核心工具面
- 插件工具、MCP 工具和内建工具可以并入同一运行时
- 能力治理集中在 registry / toolset，而不是散落在 prompt 和入口代码里

核心心法：

**工具系统应该是平台层能力面，而不是一堆零散函数的集合。**

---

## 6. Gateway 不是消息转发器，而是 Agent 的第二生存空间

Hermes 把 Telegram、Discord、Slack、Email 等入口看成同一个 agent 的不同表面，而不是一组外围 connector。

这背后有几个重要判断：

- 消息入口不应该是弱化版 agent
- 会话、记忆、模型切换、审批、打断、cron、voice 都要在 gateway 侧保留完整语义
- 跨平台连续性本身就是产品价值，而不是接入便利性

所以 Hermes 的 gateway 不是单纯“收消息 -> 调模型 -> 回消息”，而是：

- 维持会话状态
- 维护运行中的 agent
- 接住平台差异
- 保持 agent 在多入口里仍然像同一个人

核心心法：

**当 Agent 真正活在多个入口里时，gateway 就不是桥接层，而是运行时的一部分。**

---

## 7. 扩展性要开放，但开放边界必须克制

Hermes 给了很多扩展点：

- plugins
- MCP
- memory providers
- delegate subagents
- 平台适配器

但它并没有把开放性做成“什么都能无限叠加”。相反，它处处在收边界：

- 外部 memory provider 一次只允许一个
- 子代理不能无限递归
- 子代理被剥离掉共享 memory、跨平台发消息、再次 delegation 等高风险能力
- hook 虽然开放，但仍有明确生命周期语义

这说明 Hermes 的设计观不是：

- 开得越多越高级

而是：

- 可扩展必须建立在可控前提上

核心心法：

**扩展点应该被设计成 seam，但 seam 必须有边界。**

---

## 8. 多 Agent 不是无限自由，而是受控并行

Hermes 支持 delegate/subagent，但它对多 Agent 的理解非常务实：

- 子代理是工作器，不是自治社会
- 子代理适合处理局部任务，不适合拥有所有副作用权限
- 并行度和深度要受控，否则上下文、权限和状态都会失控

这和很多“Boss Agent 无限派生小弟”的想象不同。

Hermes 的判断是：

- 多 Agent 的价值在于降低上下文负担和提高并行效率
- 多 Agent 的风险在于权限扩散、状态污染和行为不可预测

核心心法：

**多 Agent 的关键不是能不能派生，而是谁能做什么、做到哪一层就该停。**

---

## 9. 安全不是补丁，而是运行时协议

Hermes 在安全上的思路很清楚：

- 危险命令检测不是零散规则，而是单一真相源
- 审批不是 UI 提醒，而是阻塞式运行时协议
- gateway 和 CLI 需要共享一致的安全语义
- 并发场景下的审批和会话状态必须真正隔离

这说明它不是把安全看成：

- 文档提醒
- 黑名单补丁
- 出事后再加规则

而是把安全看成 agent 与用户协作的一部分。

核心心法：

**真正有副作用的 agent 系统，安全流程必须进入 runtime，而不能停留在文案层。**

---

## 10. 运维韧性本身就是产品能力

Hermes 很多实现细节都指向一个事实：真实世界里的 agent 不是在干净实验室里运行的。

它需要面对：

- broken stdout / stderr
- 并发写冲突
- 长时间运行的 gateway
- 平台依赖碎裂
- 不同系统环境的能力边界
- 缓存与会话恢复的一致性问题

因此它做了很多看上去“很脏”的工程处理：

- stdout/stderr 兜底
- SQLite WAL + jitter retry
- optional extras 分层
- 明确要求 WSL2，而不是假装原生 Windows 也完全等价

这些都说明 Hermes 很清楚：

系统的可靠性不只来自模型和算法，还来自对真实运行问题的吸收能力。

核心心法：

**能长期运行的 agent runtime，必须把脏活累活当作正式设计对象。**

---

## 11. Hermes 真正可迁移的 6 条设计原则

1. 先定义长期状态模型，再优化单轮回答效果。
2. 先建立统一 agent core，再扩展 CLI、gateway、ACP 等入口。
3. 先分清 memory / session / skill，再谈长期智能。
4. 先把 prompt、toolset、approval 工程化成契约，再谈能力扩张。
5. 先给插件、MCP、subagent 画边界，再开放扩展面。
6. 先解决真实运行问题，再谈架构优雅。

---

## 12. 不该照抄的地方

Hermes 很值得研究，但不等于每个做法都该原样复制：

- 它的核心文件偏大，维护和 review 成本高
- sync/async bridging 复杂，运行时心智负担不低
- import-time registration 扩展灵活，但调试成本也会上升
- 它的产品定位偏长期个人 runtime，不一定适合所有团队级平台场景

因此更适合吸收的是它的结构思想，而不是直接复刻它的产品形态。

应该重点吸收的是：

- 长期状态分层
- 多入口统一核心
- 工具面治理
- 受控扩展
- 安全协议化
- 运维现实主义

---

## 13. 最终总结

如果把 Hermes Agent 的核心设计思想压缩成四句话，就是：

1. **Agent 应该被设计成长期存在的主体，而不是一次性回答器。**
2. **多入口差异应被边缘吸收，核心运行语义必须统一。**
3. **长期知识必须拆成 memory、session、skill 三层，而不是混成一个桶。**
4. **安全、扩展边界和运行韧性，本身就是 agent runtime 的核心能力。**

所以，Hermes 最值得学习的不是“它支持很多平台和很多工具”，而是：

**它把个人 Agent 从一个会调用工具的聊天壳，推进成了一个有长期状态、有统一核心、有边界意识、能在真实环境里持续运行的 runtime 系统。**
