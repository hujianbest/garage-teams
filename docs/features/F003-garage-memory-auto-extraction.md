# F003: Garage Memory — 自动知识提取与经验推荐

- 状态: 草稿
- 主题: 让 Garage 从“会存”走向“会记得、会提醒”
- 日期: 2026-04-18
- 关联:
  - F001（Garage Agent 操作系统）
  - F002（Garage Live）

## 1. 背景与问题陈述

F001 建立了 Garage 的运行时、知识库、经验记录和文件契约；F002 又把 CLI 和真实宿主链路跑通，证明 Garage 已经能在真实工作流中产生 session、artifact 和 experience。

但距离愿景中的“记住你、跟着你、长在你身上”还有一个明显缺口：

**Garage 已经能保存数据，但还不会把一次工作自动提炼成下一次工作可用的记忆。**

当前问题主要体现在：

1. **知识积累仍然偏手动**：session 完成后，技术决策、问题解法和模式识别仍依赖人工整理，摩擦高，容易遗漏。
2. **历史经验难以在下一次任务中主动出现**：系统虽然已有 knowledge / experience 存储，但推荐链路还不完整，用户必须自己去搜。
3. **愿景中的“记忆体”尚未成立**：成长策略把 Stage 2 定义为“工具箱 + 自动知识积累 + 经验检索”，当前仍停留在 Stage 1 向 Stage 2 过渡的中间态。
4. **用户契约要求可审计与用户主导**：即使要自动提取，Garage 也不能绕过用户直接发布知识，更不能让黑箱判断变成唯一真相。

如果这个缺口不补上，Garage 只能是“带存储的工具箱”，还不是“会成长的 Agent 能力基座”。

## 2. 目标与成功标准

### 2.1 核心目标

让 Garage 在一次真实 session 结束后，形成以下闭环：

```
session / artifact / experience
  ↓
自动提取知识候选草稿
  ↓
用户确认 / 编辑 / 拒绝
  ↓
写入知识库并更新经验索引
  ↓
下一次相似任务开始时主动推荐相关经验
```

### 2.2 成功标准

1. **自动提取可用**：一次 session 完成后，系统能自动生成结构化知识候选草稿，而不是只留下原始记录。
2. **用户始终掌舵**：任何知识在进入正式知识库前，都必须经过用户确认、编辑或明确拒绝。
3. **推荐形成闭环**：下一次执行相似任务时，系统能基于历史知识和经验主动给出相关推荐，而不是完全依赖用户手动搜索。
4. **全程可追溯**：每条候选和已发布知识都能回溯到来源 session、artifact 和确认动作。
5. **不破坏现有链路**：即使自动提取被关闭、失败或不准确，F001/F002 的手动知识流和运行时链路仍然可用。

### 2.3 非目标

- 不在本阶段实现自动 skill 生成或自动 agent 组装。
- 不在本阶段引入 Web UI、常驻服务或外部数据库。
- 不追求“零误报”的知识提取算法；第一版以“可用、可确认、可回退”为目标。
- 不把性能优化整体打包到本 spec 中；若查询性能问题需要独立治理，单独立项处理。

## 3. 用户角色与关键场景

### 3.1 主要用户角色

- **Solo Creator**：希望系统自动记住自己的项目决策、问题解法和工作模式，但仍保留最终判断权。
- **Developer**：在连续开发任务中，希望下一次执行时直接看到相似任务的历史经验。

### 3.2 关键场景

1. **自动提取**：`garage run` 完成后，系统自动从 session 与 artifact 中生成知识候选草稿。
2. **草稿确认**：用户查看候选草稿，决定接受、编辑后接受、批量拒绝或延后处理。
3. **知识发布**：被接受的候选进入知识库，并保留来源锚点与确认记录。
4. **相似任务推荐**：下次执行同类 skill、同类问题域或相似模式的任务时，系统默认主动推荐相关知识和经验。
5. **失败降级**：自动提取失败时，session 仍正常完成，系统记录失败原因，不污染知识库。

## 4. 范围

### 4.1 包含

| 功能 | 描述 |
|------|------|
| 自动知识提取 | 在 session 完成后生成知识候选草稿 |
| 候选分类 | 至少支持 decision / pattern / solution / experience summary 四类候选 |
| 用户确认门禁 | 用户可接受、编辑或拒绝候选，未确认前不得正式发布 |
| 来源追溯 | 候选和已发布知识保留 source session / artifact / evidence anchor |
| 冲突提示 | 遇到相似已有知识时，提示用户合并、替换或并存 |
| 经验推荐 | 在相似任务执行前或执行中推荐相关知识与经验 |
| 确认负担控制 | 控制单次候选数量，并支持批量拒绝与延后处理 |
| 失败降级 | 提取失败不影响主 session 完成 |

### 4.2 不包含

- 自动批准或自动发布知识
- 自动 skill 生成、自动 agent 生成
- 复杂机器学习排序系统
- 多用户共享知识治理
- 与外部 SaaS 知识库同步

## 5. 范围外内容

以下内容明确不在当前规格范围内：

- 社区共享、知识市场、跨用户脱敏传播
- 面向团队协作的审批流和权限系统
- 面向百万级知识条目的数据库化存储
- 自动发布博客、自动改写技能、自动升级工作流

## 6. 术语与定义

| 术语 | 定义 |
|------|------|
| **知识候选草稿** | 自动提取产生、但尚未进入正式知识库的候选条目 |
| **已发布知识** | 经用户确认后进入正式知识库的条目 |
| **经验推荐** | 基于历史 knowledge / experience 对当前任务给出的主动提示 |
| **来源锚点** | 能回溯到原始 session、artifact、片段或确认动作的证据定位 |
| **相似任务** | 在 skill、问题域、关键模式、标签或 artifact 类型上与历史记录匹配的任务 |

## 7. 功能需求

### FR-301 自动触发知识提取

- **优先级**: Must
- **来源**: 成长策略 Stage 2；愿景“Agent 要能记住你”
- **需求陈述**: 当一个 session 进入 completed 或 failed 且存在可读取证据时，系统必须自动生成一批知识候选草稿。
- **验收标准**:
  - Given 一个 session 已归档且存在 transcript、artifact 或 experience，When 自动提取流程运行且命中至少一条满足最小质量门槛的知识信号，Then 系统生成至少 1 条结构化候选草稿并为该批次记录执行结果
  - Given 一个 session 没有可提取证据，When 自动提取流程运行，Then 系统不发布知识，并记录“无可提取证据”
  - Given 一个 session 存在可读取证据但所有信号都因重复、信息不足或未达到最小质量门槛而被过滤，When 自动提取流程运行，Then 系统允许产生 0 条候选，但必须记录“已评估但未形成候选”的原因摘要
  - Given 自动提取被关闭，When session 完成，Then 系统跳过提取流程且不影响 session 完成

### FR-302a 第一版必须支持四类候选

- **优先级**: Must
- **来源**: Phase 2 Auto Extractor 设计线索；范围 4.1 候选分类
- **需求陈述**: 当系统生成知识候选草稿时，系统必须能够把候选归类为 decision、pattern、solution 或 experience summary 四类之一。
- **验收标准**:
  - Given 系统生成一个知识候选，When 候选进入待确认队列，Then 该候选的类型必须是 decision、pattern、solution 或 experience summary 之一
  - Given 系统无法把候选归入这四类中的任何一种，When 校验候选，Then 该候选不能进入待确认队列
  - Given 用户查看候选列表，When 列表展示，Then 每条候选都能看到其类型
### FR-302b 候选草稿必须自描述且可追溯

- **优先级**: Must
- **来源**: 用户契约“透明可审计”；设计原则“文件即契约”
- **需求陈述**: 当系统生成知识候选草稿时，每个候选必须包含足够的结构化元信息，使用户和 Agent 都能独立理解它是什么、来自哪里、为何被提出。
- **验收标准**:
  - Given 系统生成一个候选草稿，When 用户查看，Then 该候选至少包含候选类型、标题或主题、摘要、source session 和至少一个来源锚点
  - Given 候选引用了 artifact，When 用户查看，Then 系统能展示该候选关联的 artifact 路径
  - Given 候选缺少必需元信息，When 系统校验，Then 该候选不能进入待确认队列

### FR-303 用户确认是知识发布前置门禁

- **优先级**: Must
- **来源**: 用户契约“你做主”；愿景“关键决策由你做”
- **需求陈述**: 当系统生成知识候选草稿时，系统必须要求用户对每条候选执行接受、编辑后接受或拒绝中的一种动作，之后才能决定是否进入正式知识库。
- **验收标准**:
  - Given 存在待确认候选，When 用户未做确认，Then 该候选保持未发布状态
  - Given 用户选择接受，When 确认完成，Then 系统将该候选发布到正式知识库
  - Given 用户选择编辑后接受，When 用户保存修改，Then 系统按修改后的内容发布并保留确认记录
  - Given 用户选择拒绝，When 确认完成，Then 系统不将该候选写入正式知识库

### FR-303a 第一版必须控制确认负担

- **优先级**: Must
- **来源**: 成长策略“不要牺牲简单性”；ASM-302 风险收敛
- **需求陈述**: 当单次 session 产生候选草稿时，系统必须把第一版用户确认负担限制在小批量范围内，并提供最小化摩擦的批处理选项。
- **验收标准**:
  - Given 单次 session 产生候选草稿，When 候选进入待确认队列，Then 待确认候选数量上限为 5 条
  - Given 自动提取识别到超过 5 条潜在候选，When 系统筛选候选，Then 系统只保留优先级最高的 5 条进入待确认队列，并记录其余候选被截断
  - Given 用户不想逐条处理当前批次，When 用户执行批量拒绝，Then 系统可一次性拒绝该批次中的全部待确认候选
  - Given 用户当前不准备处理候选，When 用户执行延后处理，Then 系统保留该批次为待处理状态且不发布知识

### FR-304 相似知识冲突必须显式处理

- **优先级**: Must
- **来源**: F001 FR-002 知识版本追踪要求；用户契约“透明可审计”
- **需求陈述**: 当系统准备发布一个候选草稿且发现知识库中存在相似的活跃条目时，系统必须显式提示用户处理该关系，而不能静默覆盖已有知识。
- **验收标准**:
  - Given 存在相似活跃知识，When 用户确认候选，Then 系统提供至少“并存”“替换/更新”“放弃发布”三类处理结果中的可选项
  - Given 用户选择替换或更新，When 发布完成，Then 系统保留新旧条目的关系记录
  - Given 不存在相似活跃知识，When 用户确认候选，Then 系统按普通发布流程写入知识库

### FR-305 已发布知识必须进入推荐闭环

- **优先级**: Must
- **来源**: 成长策略 Stage 2；愿景“下次更强”
- **需求陈述**: 当系统开始执行一个相似任务时，系统必须默认主动查询历史 knowledge 与 experience，并返回相关推荐结果。
- **验收标准**:
  - Given 当前任务与历史经验在 skill、问题域、关键模式或标签上匹配，When 系统在任务开始阶段执行主动推荐，Then 返回至少一条按相关性排序的推荐结果
  - Given 没有匹配结果，When 系统在任务开始阶段执行主动推荐，Then 返回空结果而不报错
  - Given 推荐结果存在，When 用户查看，Then 每条推荐都包含标题、类型和来源说明

### FR-306 推荐结果必须解释“为什么推荐”

- **优先级**: Should
- **来源**: 用户契约“透明可审计”
- **需求陈述**: 当系统推荐历史知识或经验时，系统应说明推荐命中的主要原因，避免黑箱式提示。
- **验收标准**:
  - Given 系统返回推荐结果，When 用户查看，Then 系统展示至少一个匹配原因，例如 skill 匹配、标签匹配、问题域匹配或关键模式匹配
  - Given 用户忽略某条推荐，When 当前任务继续执行，Then 系统不强制用户处理该推荐

### FR-307 自动提取失败必须安全降级

- **优先级**: Must
- **来源**: 用户契约“能力不丢失”；F001 错误处理原则
- **需求陈述**: 当自动知识提取流程失败时，系统必须记录失败信息并安全降级，且不得阻塞主 session 的完成与归档。
- **验收标准**:
  - Given 自动提取流程报错，When 错误发生，Then session 仍保持原本的完成结果并完成归档
  - Given 自动提取流程报错，When 错误被记录，Then 系统保存错误摘要、时间和关联 session
  - Given 自动提取流程失败，When 用户查看状态，Then 系统能够明确区分“session 成功”与“提取失败”

## 8. 非功能需求

### NFR-301 全链路可追溯性

- **优先级**: Must
- **来源**: 用户契约“透明可审计”
- **需求陈述**: 当候选草稿被生成、确认、拒绝或发布时，系统必须保留完整的状态变化与来源锚点。
- **验收标准**:
  - Given 任意一个已发布知识条目，When 用户回查，Then 可以定位到来源 session 和至少一个来源 artifact 或 evidence anchor
  - Given 任意一个候选草稿，When 用户回查，Then 可以知道其当前状态是待确认、已接受、已拒绝或已发布

### NFR-302 用户控制优先于自动化

- **优先级**: Must
- **来源**: 用户契约“你做主”；成长约束“不替用户决定成长方向”
- **需求陈述**: 当系统进行自动提取与推荐时，系统必须默认把自动化限定在“建议与草稿”层，而不能越过用户完成发布决策。
- **验收标准**:
  - Given 系统自动提取到候选，When 用户尚未确认，Then 系统不得把该候选标记为正式知识
  - Given 用户关闭自动提取或推荐，When 后续 session 执行，Then 系统尊重该开关

### NFR-303 Stage 2 性能门槛

- **优先级**: Should
- **来源**: F001 NFR-001；当前查询性能风险
- **需求陈述**: 当系统执行候选生成或相关推荐时，系统应在可接受时间内返回结果，并避免在知识规模增长后出现不可接受的退化。
- **验收标准**:
  - Given 单个 session 含不超过 50 个 transcript 片段与 20 个 artifact，When 自动提取运行，Then 90% 的候选生成在 60 秒内完成
  - Given 知识库规模达到 1000 条目，When 执行相关推荐查询，Then 90% 的查询在 5 秒内返回
  - Given 知识库规模从 100 增长到 1000 条目，When 执行同类推荐查询，Then 查询响应时间退化不超过 50%

### NFR-304 与 Phase 1/2 现有能力兼容

- **优先级**: Must
- **来源**: 设计原则“渐进复杂度”；F001 可迁移性
- **需求陈述**: 当自动提取模块不存在、被关闭或升级时，系统必须保持现有知识库、experience 记录和 CLI 基本能力继续可用。
- **验收标准**:
  - Given 自动提取模块不可用，When 用户仍使用现有 knowledge / experience 功能，Then 这些能力正常工作
  - Given 历史知识条目由手动方式创建，When 系统执行推荐，Then 历史条目仍可参与匹配

## 9. 外部接口与依赖

### IFR-301 Session 证据读取接口

- **优先级**: Must
- **来源**: FR-301
- **需求陈述**: 系统必须能够读取 session transcript、artifact 元数据和 experience 记录，作为知识提取的输入证据。
- **验收标准**:
  - Given 一个已完成 session，When 提取流程启动，Then 系统能读取该 session 的基础上下文、artifact 列表和执行结果
  - Given 某类证据缺失，When 提取流程运行，Then 系统对缺失进行显式记录而不是静默假设

### IFR-302 知识确认与发布接口

- **优先级**: Must
- **来源**: FR-303、FR-304
- **需求陈述**: 系统必须通过当前宿主可见的交互界面或命令入口，向用户展示候选草稿并收集确认结果。
- **验收标准**:
  - Given 用户需要确认候选，When 系统展示候选，Then 用户可完成接受、编辑后接受或拒绝操作
  - Given 用户完成确认，When 系统写入结果，Then 发布状态与确认动作一致

### IFR-303 推荐查询接口

- **优先级**: Must
- **来源**: FR-305、FR-306
- **需求陈述**: 系统必须提供可调用的推荐查询能力，并把该能力作为第一版默认主动推荐的底层接口，使运行时或 CLI 能在任务开始前触发相关推荐。
- **验收标准**:
  - Given 当前任务上下文可被表达为 skill、问题域、标签或关键模式，When 推荐查询执行，Then 系统返回结构化推荐结果
  - Given 运行时进入相似任务的开始阶段，When 未显式关闭推荐功能，Then 系统默认触发一次推荐查询
  - Given 用户或配置显式关闭推荐功能，When 查询未触发，Then 主执行链路不被强制阻塞

## 10. 约束

### CON-301 Workspace-first 存储

- **优先级**: Must
- **来源**: 愿景“数据归你”；用户契约“能力不丢失”
- **需求陈述**: 系统必须将候选草稿、确认记录和已发布知识存储在 Garage 仓库可追踪的文件空间内，而不是依赖外部 SaaS 状态。

### CON-302 不得静默自动发布

- **优先级**: Must
- **来源**: 用户契约“你做主”
- **需求陈述**: 系统不得因为置信度、规则命中或宿主默认行为而跳过用户确认，直接发布知识。

### CON-303 契约演进必须可兼容

- **优先级**: Must
- **来源**: 设计原则“文件即契约”
- **需求陈述**: 候选草稿、确认状态和已发布知识的格式变更必须带版本语义，并提供可回读或迁移路径。

### CON-304 第一版不引入外部数据库或常驻服务

- **优先级**: Must
- **来源**: 成长策略“不要牺牲简单性”；当前阶段约束
- **需求陈述**: 当前功能必须建立在现有仓库内文件存储和按需执行模型之上，而不是额外引入外部数据库、消息队列或常驻进程。

## 11. 假设

### ASM-301 Session 证据足够支撑第一版提取

- **优先级**: Should
- **来源**: 当前运行时已有 session / artifact / experience 数据
- **需求陈述**: 假设现有 session transcript、artifact 元数据和 experience 记录已经足够支撑第一版自动提取。
- **失效风险**: 如果证据粒度不足，提取质量会不稳定。
- **缓解措施**: 在设计阶段补充最小必需 evidence contract，并允许“无候选”结果。

### ASM-302 用户愿意审阅小批量候选

- **优先级**: Should
- **来源**: 用户契约“review 系统建议”
- **需求陈述**: 假设用户愿意对每次 session 产生的小批量候选执行轻量确认。
- **失效风险**: 如果候选过多，确认摩擦会使功能失去使用价值。
- **缓解措施**: 第一版把单次候选上限限制为 5 条，并支持批量拒绝或延后处理。

### ASM-303 第一版推荐允许采用启发式匹配

- **优先级**: Should
- **来源**: 成长策略“渐进增强”
- **需求陈述**: 假设 Stage 2 的相关推荐可先基于 skill、标签、问题域和关键模式的启发式匹配，而不依赖复杂模型排序。
- **失效风险**: 如果相关性不足，推荐命中率会偏低。
- **缓解措施**: 保留命中原因和反馈信号，为后续优化提供证据。
