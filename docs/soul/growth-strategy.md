# Garage 成长策略

## 核心思路

Garage 不是一个"做完就完了"的项目。它是一个**跟着用户长**的系统——你用得越多，它越强。

成长不是靠功能堆叠，而是靠三层飞轮：

```
使用 → 积累 → 提炼 → 增强 → 使用（更强）
```

- **使用**：你用 Garage 做 spec、做设计、写代码、写文章
- **积累**：系统自动记录知识、决策、模式
- **提炼**：从积累中识别可复用的 skills、agents、知识模式
- **增强**：新的 skills 和 agents 让下一次使用更高效
- **使用（更强）**：更强的系统让你愿意做更多事

每一层飞轮的转速不需要一样快。前期使用和积累为主，后期提炼和增强跟上。

## 成长阶段

### Stage 1：工具箱（当前）

```
形态：一套结构化的 markdown skills + 项目约定
用户价值：AHE workflow 让开发过程有纪律
系统能力：静态文档，靠宿主（Claude Code）执行
积累方式：手动维护，git 提交
```

触发进入下一阶段的信号：
- 用户日常工作中频繁使用 Garage 的 skills
- 积累了 3+ 个完整的 feature 开发周期经验
- 开始感觉到"每次都要手动告诉 Agent 背景知识"的摩擦

### Stage 2：记忆体

```
形态：工具箱 + 自动知识积累 + 经验检索
用户价值：Agent 记得上次做过什么，能主动推荐经验和模式
系统能力：知识自动提取、结构化存储、按场景检索
积累方式：自动 + 半自动（系统提取，用户确认）
```

核心新增：
- 知识库自动积累（技术决策、问题解法、最佳实践）
- 经验按场景检索（遇到类似问题自动推荐历史解法）
- 会话上下文持续化（跨会话保持项目状态）

触发进入下一阶段的信号：
- 知识库条目 >100
- 识别到 5+ 个可复用模式
- 用户开始期望"系统自动帮我做更多"

### Stage 3：工匠

```
形态：记忆体 + 自动 skill 生成 + Agent 组装
用户价值：系统从你的工作中提炼新 skills，组装专用 agents
系统能力：模式→skill 模板、skill→agent 组合、工作流自动编排
积累方式：全自动（系统提炼，用户审批）
```

核心新增：
- 重复模式自动识别并建议为 skill 模板
- Skills 可组合成专用 agents（如"代码审查 agent"、"博客写作 agent"）
- 工作流编排从手动变成半自动

触发进入下一阶段的信号：
- Skills 数量 >30
- 用户在多个不同领域（开发、写作、研究）都有积累
- 需要跨领域协作（比如"用这次开发的经验写一篇博客"）

### Stage 4：生态

```
形态：工匠 + 跨用户知识共享 + 社区 skills 市场
用户价值：一个人的积累可以惠及社区，社区的优秀 skills 可以直接用
系统能力：Skills 导出/导入、知识脱敏分享、Pack 市场
积累方式：个人 + 社区
```

这个阶段是远景，当前不做详细规划。核心原则是：个人积累优先，社区共享是锦上添花。

## 成长的度量

不设 KPI，但关注这些信号来判断系统是否在健康成长：

| 信号 | 健康表现 | 不健康表现 |
|------|---------|-----------|
| Git 提交频率 | 稳定，每周有实质提交 | 长时间空窗或大量无意义提交 |
| 知识条目增长 | 随使用自然增长 | 停滞不前或爆炸式堆积 |
| Skills 使用率 | 常用 skills 被高频调用 | Skills 写了但从不用 |
| 宿主切换意愿 | 敢于尝试新宿主 | 害怕切换导致能力丢失 |
| 自我成长事件 | 系统能指出"这个模式可以变成 skill" | 所有成长都靠手动维护 |

## 成长的约束

### 不为了成长而成长

每一个新增能力都要回答：这解决了我实际遇到的什么问题？如果答不上来，就不加。

### 不牺牲简单性

Stage 1 的用户体验应该永远可用。如果你只想用最基础的功能，系统不应该因为你"还没积累够"就给你制造摩擦。

### 不制造依赖

成长是赋能，不是绑定。用户随时可以：
- 停止使用 Garage，所有数据仍然可读
- 只用 Garage 的一部分功能，不需要全盘接受
- 把 Garage 的知识导出到其他系统

### 不替用户决定成长方向

系统可以建议"你经常做 X，要不要把它变成一个 skill？"，但不能自动发布、自动生效。所有成长都要经过用户的审视和确认。

## 成长策略与设计原则的关系

这份成长策略是 `docs/soul/design-principles.md` 中五条原则在时间维度上的展开：

| 原则 | 在成长策略中的体现 |
|------|------------------|
| 宿主无关 | 每个 Stage 都能换宿主，不因为成长而绑定 |
| 文件即契约 | Schema 在 Stage 1 就定义好，后续 Stage 只扩展不破坏 |
| 渐进复杂度 | 四个 Stage 就是四层复杂度，每层有明确触发信号 |
| 自描述 | 知识条目、skills、agents 都能被 Agent 独立理解 |
| 约定可发现 | 每个新增约定都有入口可发现，不会淹没在成长中 |

## 成长策略本身的成长

这份策略是活的。每进入一个新 Stage 时，回顾这份文档，根据实际经验修正下一阶段的预期和触发信号。
