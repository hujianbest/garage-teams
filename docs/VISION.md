## 1. 为什么需要这份文档

`Garage` 不只是一个运行时项目，也不只是一个 agent workflow 集合。

它首先是一个产品判断：

**个人创作者真正缺的，不是下一次更聪明的回答，而是一个可以长期协作、持续成长、不断积累方法和默契的 Agent Teams 工作环境。**

因此，这份文档只负责讲清楚：

- 为什么今天的大多数 AI 产品还不够
- `Garage` 想建立的到底是什么工作方式
- 哪些产品哲学和设计公理不能退让

一句话说：

**`docs/VISION.md` 讲“为什么必须做 `Garage`”，`docs/GARAGE.md` 讲“`Garage` 当前是什么产品”。**

## 2. 一句话愿景

`Garage` 想做的，不是另一个更聪明的 AI 聊天工具，也不是一个要求开发者手工集成的底层框架，而是一个辅助创作者的 `Agent Teams` 工作环境，也是一个可持续成长的 `Garage Team runtime`。

## 3. 我们想改变的创作工作方式

今天的大多数 AI 产品，仍然主要停留在这些形态：

- 提供更强的单轮回答
- 把多个模型和工具包在同一个壳里
- 在当前会话里看起来聪明，但很难跨阶段持续推进

但创作者真正缺的，往往是：

- 一个能跨多阶段持续推进的工作环境
- 一支可以分工、交接、复查、沉淀经验的 AI 团队
- 一条从洞察到实现再到表达的连续主线
- 一套不会在每次新会话里重置的长期记忆、可复用方法和运行纪律

如果这些东西不存在，用户就只能不断做人肉路由器：

- 手工切换工具
- 手工复制上下文
- 手工决定下一步
- 手工补齐记忆断层
- 手工把一次性经验重新教给系统

`Garage` 要改变的，正是这种工作方式。

## 4. 为什么不是聊天壳、工具壳或纯开发者框架

`Garage` 不想成为：

- 一个更强的单体聊天助手
- 一个主要靠模型切换和工具清单堆出来的工具壳
- 一个只有开发者才能通过代码集成才能真正使用的基础设施

原因很简单：

- 创作者需要的是可直接进入的工作环境，而不是先写一层集成代码
- 团队协作、长期记忆、review、handoff 和治理，不应该退化成宿主外的一堆手工约定
- 如果系统只能被“集成”，却很难被“直接使用”，它就很难成为真正的产品

因此，`Garage` 的产品优先级应始终是：

1. 先成为一个可直接进入的独立工作环境
2. 再把自己的 agents、skills 和长期能力注入到已有工具中

## 5. Agent Teams 工作环境的产品哲学

### 5.1 领养团队，不是配置工具

`Garage` 的目标不是让用户管理一堆模型和工具开关，而是让用户像拥有一支 AI 团队一样工作。

这意味着产品的一等对象不是：

- model list
- tool list
- integration toggles

而是：

- team
- role
- handoff
- review
- memory
- skill
- shared mission

### 5.2 人负责方向，AI 负责放大

人在 `Garage` 中始终保留：

- 愿景
- 判断
- 审批
- 取舍

AI 团队负责：

- 研究
- 生成
- 执行
- 复查
- 沉淀
- 在治理约束下持续改进自己

### 5.3 结果比回答更重要

`Garage` 不以“单轮看起来很聪明”作为成功标准。

更重要的是：

- 能否持续推进真实工作
- 能否把模糊想法变成结构化结果
- 能否把结果留下来并供下一步消费
- 能否让一次经验变成未来工作的生产力

## 6. 扩展、成长、治理三条主线

### 6.1 可扩展性

`Garage` 必须允许团队持续扩展新能力，而不是被当前能力集合锁死。

这里说的扩展，不只是：

- 增加更多工具
- 增加更多入口

更重要的是：

- 新增不同类型的 agents
- 新增不同领域的团队能力面
- 新增支持代码、文章、产品、音乐、视频等不同创作结果的扩展结构

### 6.2 可成长性

`Garage` 也不能永远停留在“第一天启动时的团队水平”。

它必须允许：

- 从工作结果中形成 `evidence`
- 从 `evidence` 中主动识别 `memory` 与 `skill` 候选
- 在治理约束下持久化长期知识
- 从经验中 patch 旧 skill、改进协作纪律、调整运行策略

### 6.3 治理能力

`Garage` 希望 agent 主动成长，但不希望它无边界地自动固化一切。

因此真正重要的是：

- 它依据什么证据提出成长建议
- 哪些经验可以进入长期资产
- 哪些经验只能先停留在 `evidence`
- 哪些动作必须经过 review、approval 或其他治理动作

换句话说：

**主动成长不是放弃治理，而是在治理下持续成长。**

## 7. 不可退让的设计公理

如果把上面的产品哲学继续压缩，`Garage` 至少有 5 条不可退让的设计公理。

### 公理一：团队先于工具

`Garage` 的最小产品单位不是模型清单、工具清单或插件清单，而是一支 AI 团队。

### 公理二：人定方向，AI 在治理中放大

AI 团队可以主动执行和主动成长，但不能绕开人的判断与治理边界。

### 公理三：扩展与成长并列

系统既要能持续接入新能力，也要能让团队因为做过的事情而变强。

### 公理四：长期连续性先于单轮聪明

`memory / session / skill / evidence` 必须长期分层，而不能混成一个历史桶。

### 公理五：所有成长都必须 `evidence-first`、`workspace-first`、`governance-bounded`

经验要先形成 `evidence`，再变成 `proposal`，再进入 `memory`、`skill` 或 `runtime update`。

## 8. 第一位用户是谁

`Garage` 首先服务的是：

- `solo creator`
- 独立开发者
- 既要想方向、又要真正做出结果的人

这不是因为更复杂场景不重要，而是因为 `Garage` 想先把最核心的个人共创闭环做对，再让系统能力和团队能力一起成长。

## 9. 明确不想做什么

`Garage` 不想成为：

- 另一个“万事都由一个超级 Agent 决定”的单体助手
- 另一个主要靠聊天记忆维持状态的工具壳
- 另一个只有开发者手工集成之后才真正可用的产品
- 另一个把“自动学习”当成黑箱神话、却无法解释知识从哪里来的系统
- 另一个一开始就被重型组织模型或无限 agent society 拖垮的系统

我们更不想为了“看起来会成长”，牺牲边界、治理、可解释性和长期可维护性。

## 11. 一句话总结

`Garage` 的愿景，是让个人创作者不再只是偶尔调用 AI，而是真正拥有一个可以培养自己 `Garage Team` 的工作环境：这支团队会持续扩展能力、持续积累记忆、持续沉淀方法、持续自我改进，并在同一个系统里把想法变成长期持续的结果。
