## 1. 为什么需要这份文档

`Garage` 不只是一个文档仓库，也不只是一个 agent workflow 集合。

它试图回答一个更根本的问题：

**如果个人创作者真的拥有一支会长期协作、持续成长、不断积累知识和方法的 AI 团队，工作方式会发生什么变化。**

因此，这份文档要先讲清楚：

- 我们为什么认为今天的大多数 AI 产品还不够
- `Garage` 想建立的到底是什么工作方式
- 哪些产品哲学必须先于架构和实现被写下来

一句话说：

**`docs/GARAGE.md` 更偏“这是什么系统”，`docs/VISION.md` 更偏“为什么这个系统必须存在”。**

## 2. 一句话愿景

`Garage` 想做的，不是另一个更聪明的 AI 聊天工具，而是一个让个人创作者能够长期带着一支会持续扩展、持续学习、持续自我改进的 AI 创作团队，把想法不断做成产品、内容和结果的 `Creator OS`。

## 3. 我们相信的问题

今天的大多数 AI 产品，仍然主要停留在下面这些形态：

- 给出更强的单轮回答
- 提供若干模型与工具的组合壳层
- 在当前会话里看起来聪明，但难以跨阶段持续推进

但个人创作者真正缺的，往往不是下一次回答，而是下面这些能力：

- 一个能跨多阶段持续推进的工作环境
- 一支可以分工、交接、复查、沉淀经验并逐渐形成协作默契的 AI 团队
- 一条从洞察到实现再到表达的连续主线
- 一套不会在每次新会话里重置的长期记忆、可复用方法和运行纪律

如果这些东西不存在，用户就只能反复做人肉路由器：

- 手工切换工具
- 手工复制上下文
- 手工决定下一步
- 手工补齐记忆断层
- 手工把一次性经验重新教给系统

这就是 `Garage` 想解决的核心问题。

## 4. 车库隐喻到底意味着什么

`Garage` 这个名字不是怀旧，也不是创业神话的装饰。

它表达的是一组非常具体的产品判断：

- 很多重要系统都是从一个小而聚焦的团队起步，而不是从庞大组织起步。
- 起步时团队能力有限、做的事情有限，但这不应该成为系统长期形态。
- 真正好的“车库团队”，不是规模永远很小，而是能不断成长，最终演化成更完整、更强韧、更成熟的组织。
- 对 `Garage` 来说，成长不只意味着“能做更多事”，还意味着“团队本身会因为做过的事而变强”。

因此，`Garage` 的车库隐喻包含两条同等重要的主线：

1. **可扩展性**：团队可以不断引入新能力、新角色、新 pack，而不需要回头推翻核心。
2. **可成长性**：团队会从工作经验里主动沉淀 `memory`、`skill`、协作纪律和运行时改进，而不是永远停留在初始水平。

换句话说，`Garage` 要表达的不是“永远在车库里做一个小玩具”，而是：

**从车库起步，但具备成长成一家真正公司的结构条件。**

## 5. 我们想创造的未来

我们希望个人创作者拥有的，不再只是一个“随叫随到的 AI”，而是一间永不下线的创业车库。

在这间车库里：

- 人始终是作者、方向拥有者和最终判断者
- AI 不只是回答问题，而是以不同角色参与研究、设计、构建、复查、沉淀和自我优化
- 创作不是一次次断裂的请求，而是一条可恢复、可追溯、可长期演化的工作主线
- 团队不仅能执行当前任务，还能从经验中形成知识、方法、纪律和新的能力

这里最关键的不是“多一个记忆功能”，而是：

- 系统会主动发现什么值得留下
- 团队会主动提出哪些经验应晋升为长期资产
- 运行时会随着经验沉淀而变得更可靠、更熟练、更像一个真正的长期团队

## 6. 核心产品哲学

### 6.1 领养团队，不是配置工具

`Garage` 的目标不是让用户管理一堆工具开关，而是让用户像拥有一支 AI 团队一样工作。

这意味着产品的最小单位不只是 model 或 tool，更重要的是：

- role
- handoff
- review
- memory
- skill
- shared mission

### 6.2 人负责方向，AI 负责放大

人不把作者身份交给系统。

人在 `Garage` 中负责：

- 愿景
- 判断
- 审批
- 取舍

AI 团队负责：

- 研究
- 生成
- 执行
- 复查
- 沉淀
- 在治理约束下持续改进自己

### 6.3 结果比回答更重要

`Garage` 不以“单轮看起来很聪明”作为成功标准。

更重要的是：

- 能否持续推进真实工作
- 能否把模糊想法变成结构化结果
- 能否把结果留下来并供下一步消费
- 能否让一次经验变成未来工作的生产力

### 6.4 可扩展性与可成长性是两条并列主线

很多系统只强调扩展性，例如：

- 可以加更多插件
- 可以接更多入口
- 可以做更多领域

但 `Garage` 认为这还不够。

如果团队只能“接更多东西”，却不会因为做过更多事情而变强，那么它依然只是一个静态工具集合。

因此，`Garage` 必须同时具备：

- **扩展能力面**：通过新增 pack、role、artifact mapping 和 runtime capability 承接新工作
- **成长团队本身**：通过 evidence、promotion、skill patching、记忆沉淀、规则调整和运行时改进提升团队成熟度

### 6.5 连续性比一次性聪明更重要

如果系统每次都只能重新开始，它就不是 `Creator OS`，而只是一个高配聊天壳。

因此 `Garage` 必须显式区分并长期维护：

- `memory`
- `session`
- `skill`
- `evidence`

并且明确区分它们，而不是把一切混成一团。

### 6.6 主动成长必须先经过治理

`Garage` 希望 agent 主动成长，但不希望它无边界地自动固化一切。

因此，真正重要的不是“系统能不能自动学”，而是：

- 它依据什么证据提出成长建议
- 它在什么边界内可以主动持久化知识
- 哪些经验可以变成 `memory`
- 哪些经验可以变成 `skill`
- 哪些经验只能先停留在 `evidence`
- 什么时候必须经过 review、approval 或其他治理动作

换句话说：

**主动成长不是放弃治理，而是在治理下持续成长。**

### 6.7 治理不是附属品，而是产品能力

愿景、规则、门禁、评审、审批和归档，不应该只存在于人脑或对话里。

它们应该被写成工件、被系统读取、被流程消费。

如果没有这层治理，所谓“主动成长”只会迅速退化成：

- 记忆污染
- skill 漂移
- prompt 漂移
- 行为不可解释

## 7. 谁是第一位用户

`Garage` 首先服务的是：

- `solo creator`
- 独立开发者
- 既要想方向、又要真正做出结果的人

这不是因为更复杂场景不重要，而是因为 `Garage` 想先把最核心的个人共创闭环做对，再让系统能力和团队能力一起成长。

## 8. 我们明确不想做什么

`Garage` 不想成为：

- 另一个“万事都由一个超级 Agent 决定”的单体助手
- 另一个主要靠聊天记忆维持状态的工具壳
- 另一个只能在单一领域里发挥作用的专用流程器
- 另一个把“自动学习”当成黑箱神话、却无法解释知识从哪里来的系统
- 另一个一开始就被重型组织模型或无限 agent society 拖垮的系统

我们更不想为了“看起来会成长”，牺牲边界、治理、可解释性和长期可维护性。

## 10. 一句话总结

`Garage` 的愿景，是让个人创作者不再只是偶尔调用 AI，而是真正拥有一支会持续扩展能力、持续积累记忆、持续沉淀方法、持续自我改进的 AI 创作团队，并在同一个系统里把洞察变成产品，把想法变成长期持续的结果。
