## Findings (critical → important → minor → nit)

### Critical

**Cr-1 (USER-INPUT) — Cache invalidation hook 与 "每次 ExperienceIndex 写新 record" 不一致 (事实错误)**

- **Spec 位置**: §2.1 D / FR-1404 / §8.3 写明在 `SessionManager._trigger_memory_extraction` 末尾、在 "每次 ExperienceIndex 写新 record 后" invalidate.
- **Code**: `_trigger_memory_extraction` 仅在 `archive_session` 成功归档后调用 (`session_manager.py` ~ L193-197 / L203-277). 末尾仅有 F013-A `SkillMiningHook.run_after_extraction`. 新的 ExperienceRecord 常来自:
  - `garage experience add` → `cli.py` ~ L1715-1741 直接 `experience_index.store`
  - `publisher.py` ~ L136-138
  - `knowledge/integration.py` ~ L222
- 这些路径**都不经过** `_trigger_memory_extraction`.
- **影响**: §8.3 BDD ("第 6 条 record 写入后 invalidate") 在按 Spec 仅改 SessionManager 时**不成立**; RSK-1404 "invalidate hook 由 SessionManager 自动调" 无法覆盖全部写入路径.
- **r2 (USER-INPUT 决策)**: 二选一 (a) 只保证 archive 后失效, 其它路径用户 `--rebuild-cache` 显式触发; (b) 在 ExperienceIndex.store 内加 invalidate hook (轻微改 F004 公共 method 内部行为, 但 method 签名不变); (c) 引入 sibling `WorkflowRecallHook` 在每个 caller 处主动调用 (与 F013-A SkillMiningHook 多 caller 接入同模式).

**Cr-2 (LLM-FIXABLE) — `ExperienceIndex.search(..., domain=)` 与 Spec 的 `problem_domain` 分桶不一致**

- **Spec**: 全文聚类 / CLI 以 `problem_domain` 为主 (例 FR-1401, §1.0 等).
- **Code**: `search()` 的 `domain` 参数过滤的是 `record.domain`, 不是 `record.problem_domain` (`experience_index.py` ~ L77-113). `ExperienceRecord` 同时有 `domain` 与 `problem_domain` (`types/__init__.py` ~ L124-133).
- **影响**: PathRecaller 若调 `search(domain=Y)` 当作用 `problem_domain`, 会**错误分桶**.
- **r2**: PathRecaller 改用 `list_records()` + filter `record.problem_domain` (与 F013-A pattern_detector 同 pattern); 不调 `search(domain=)`. Spec § 调研锚点行也修正.

**Cr-3 (LLM-FIXABLE) — INV-F14-5 "既有 7 步流程" 与 hf-workflow-router 正文不一致**

- **Spec**: INV-F14-5 称 "hf-workflow-router 既有 7 步流程".
- **Code**: `packs/coding/skills/hf-workflow-router/SKILL.md` Workflow 编号为 **1-10** (L47-137: 1 entry / 2 evidence / 3 branch signal / 4 Profile / 4A UI / 5 entry bias / 5A reviewer / 6 handoff / 7 canonical node / 8 review recovery / 9 dispatch / 10 continuous exec). Methodology 表中的 "步骤 7" 指 FSM 路径中一站, 不是 "共 7 步".
- **r2**: INV-F14-5 改 "既有 1-10 步流程不变, F014 在 step 3 (支线信号) 与 step 4 (Profile 决策) 之间插入 step 3.5 (additive)". 摘要 (Spec L4) "step 4 / 6" 仍可对齐 (4=Profile, 6=handoff). 不再说 "7 步".
