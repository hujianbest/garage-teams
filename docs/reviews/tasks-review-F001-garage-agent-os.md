## 3. 发现项

### F-01 [Medium] T14 和 T16 的 Ready When 表述模糊

- **位置**: T14 Ready When "M2+M3 核心=done", T16 Ready When "所有 M2+M3+M4 前置=done"
- **问题**: 使用里程碑名称代替显式 task ID 列表，不同于其他 task 的精确表述（如 T7 的"T5=done AND T6=done"）
- **建议**: 改为显式 task ID 列表：T14 Ready When = "T4=done AND T5=done AND T6=done AND T8=done AND T12=done"
- **阻塞度**: 不阻塞。当前内容可通过上下文推断正确依赖

### F-02 [Low] T22 文档验证缺乏客观量化方法

- **位置**: T22 测试设计种子 "新 Agent 按文档操作 → 5 分钟内理解约定"
- **问题**: "5 分钟内理解约定"无法自动化验证，依赖主观判断
- **建议**: 补充可量化验证项，如"文档中所有路径引用均存在且可访问"、"代码示例可执行且通过"、"AGENTS.md 包含完整的约定映射表"
- **阻塞度**: 不阻塞。T22 是最后一个任务，可在执行时补充

### F-03 [Info] T3 与 T2 的并行执行存在潜在耦合

- **位置**: 并行组 A（T2, T3 并行在 T1 之后）
- **问题**: T3 创建 8 个 YAML 契约文件，如果 T2 最终选择 Python 而非 TypeScript，契约文件中的一些类型定义（如 `string`, `object`）可能需要调整格式。不过，YAML 契约在设计阶段是伪语法（设计文档 10.2 节声明"Phase 1 不可直接机器校验"），实际影响有限
- **建议**: 维持并行，但在 T2 完成后做一次快速校验确认 T3 产出与选型一致
- **阻塞度**: 不阻塞
