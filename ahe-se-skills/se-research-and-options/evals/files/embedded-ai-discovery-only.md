# 嵌入式 SOC AI 推理微服务 Discovery-Only Analysis Pack

- 状态: discovery 完成，待进入 research
- 主题: 嵌入式 SOC 上新增 AI 推理微服务进程
- 更新时间: 2026-04-09

## 1. 问题陈述

当前要分析的主题是：在嵌入式 SOC 系统上新增一个独立微服务进程，对外提供 AI 推理服务。该服务需要：

- 订阅系统状态
- 接收其他服务通过 TCP 发送的请求
- 与下层定义的交互接口协作
- 在上层考虑 OpenSSL 封装
- 设计并发请求的处理机制
- 通过下层 AI 推理框架执行实际推理

## 2. 当前范围

本轮 analysis 需要覆盖：

- 新微服务的职责边界
- 上下游关系与关键交互
- OpenSSL 封装可能承担的职责
- 并发模型候选方案
- 后续 research 需要回答的关键未知点

本轮明确不做：

- 详细字段级协议定义
- 真实业务代码仓中的组件复用确认
- 正式候选方案矩阵与推荐方向
- DFX、AR 和代码量估算的正式收敛
- 具体框架 API 映射
- 精确工时评估

## 3. 参与者与系统边界

- 调用方：板端其他服务，通过 TCP 请求 AI 推理能力
- 服务方：新增的 AI 推理微服务进程
- 下游依赖：下层接口层 / 交互组件、下层 AI 推理框架、系统状态发布方

关键边界：

- 传输与会话安全边界：TCP / TLS / OpenSSL 封装层
- 协议 / 请求语义边界：请求解析、校验、错误语义
- 调度边界：队列、线程池、并发控制
- 推理边界：下层框架适配与资源保护

## 4. 关键场景

### 场景 1：正常推理请求

1. 其他服务建立 TCP 连接
2. 微服务完成连接接入和安全处理
3. 请求被解析并校验
4. 服务读取当前系统状态快照
5. 请求进入调度队列
6. 推理执行单元调用下层框架
7. 返回成功结果或明确失败结果

### 场景 2：系统状态不允许受理

1. 服务收到推理请求
2. 当前系统状态表明资源不足、模型不可用或服务降级
3. 服务快速拒绝、排队或返回忙碌状态

### 场景 3：高并发 / 资源受限

1. 多个请求同时到达
2. 服务需要对连接数、队列深度和执行并发做限制
3. 超过阈值时需要背压、拒绝或超时处理，而不是无限堆积

## 5. 约束与依赖

- 运行在嵌入式 SOC 环境，资源受限
- 需要与现有系统状态发布机制集成
- 下层组件会定义部分接口或交互契约
- 需要处理 TCP 并发请求
- 需要对安全边界有明确解释，否则上层封装 OpenSSL 的职责会与下层接口层冲突

## 6. 已确认事实

- 需要新增独立微服务进程对外提供 AI 推理服务
- 该服务需要订阅系统状态
- 请求通过 TCP 到达
- 下层会设计至少一部分交互接口
- 上层存在“封装 OpenSSL”的要求，但原因和边界尚未澄清
- 需要处理并发请求
- 推理执行依赖下层框架接口

## 7. 假设

- 外部调用是同步请求-响应模型，而不是异步作业模型
- OpenSSL 封装主要用于 TLS / 证书 / 会话策略统一，而不是实现业务协议
- 下层推理框架的并发能力有限，不能默认按连接数横向扩线程
- 系统状态会影响请求受理或调度策略
- 当前更关注第一版安全可落地，而不是极限吞吐

## 8. 开放问题

- TCP 契约由谁定义到什么粒度？仅定义载荷格式，还是同时定义传输安全和错误语义？
- OpenSSL 封装到底是平台硬约束，还是团队经验性建议？
- 下层推理框架是否线程安全？允许多少并发推理？
- 状态订阅的刷新频率、一致性要求和影响范围是什么？
- 请求在排队中遇到状态变化时应如何处理？
- 是否需要 mTLS、证书轮换和客户端身份校验？

## 9. 研究问题

- 上层为什么要封装 OpenSSL，它自然应负责哪些职责？
- 在嵌入式 / SOC 环境里，推理服务应采用什么并发模型？
- 连接接入、队列管理和推理执行如何分层最稳妥？
- 如何在资源受限情况下设计背压与过载保护？
- 需要哪些 DFX 设计点，才能让方案足够可评审？
