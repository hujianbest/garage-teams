# Code Estimation Rubric

## 目的

这里的“参考代码量”不是工时承诺，也不是精确预算，而是帮助方案分析阶段快速判断：

- 变更面大概有多大
- 主要复杂度来自哪里
- 哪些部分会拉高实现和验证成本

## 基本原则

- 一律写范围，不写单点精确数字
- 明确包含什么，不包含什么
- 说明主要不确定性
- 给出置信度

## 估算维度

建议至少从下面几类工件估算：

- 业务逻辑代码
- 接口 / 协议适配代码
- 线程 / 调度 / 并发控制代码
- 配置与部署变更
- 测试代码
- 日志、指标和运维辅助代码
- 文档或脚本补充

## 按 AR 分析工作量

如果方案里已经做了 `AR` 分解，推荐同时给出“每个 AR 的工作量分析”。

这样做的价值是：

- 更容易识别哪个需求切片最重
- 更容易判断先做哪个 AR、哪个 AR 需要预研
- 更容易向团队解释“为什么这个方案工作量大”

每个 AR 至少补 4 项：

- `工作量级别`
- `参考代码量范围`
- `主要工作项`
- `关键不确定性`

推荐写法：

```markdown
### AR-001. 外部服务可通过安全 TCP 通道提交推理请求
- 工作量级别: 中到大
- 参考代码量范围: 220-380 LoC
- 主要工作项:
  - TCP / TLS 接入
  - OpenSSL 封装
  - 协议解析与错误映射
- 关键不确定性:
  - TLS / mTLS 要求
  - 协议 ownership
  - 现有组件复用度
```

注意：

- 每个 AR 的工作量分析是“该 AR 主导的实现贡献”，不一定等于一组完全独立、可直接相加的代码块
- 如果多个 AR 共享基础设施，应在文档里显式说明共享关系
- AR 工作量分析服务于排序、拆解和讨论，不应伪装成精确工时承诺

## 复杂度观察点

以下因素会显著放大代码量和工作量：

- 需要新增进程或服务边界
- 需要适配新协议或安全层
- 需要处理并发、队列、资源隔离或线程安全
- 需要和多个上下游系统打通
- 需要补足大量测试夹具、模拟器或工具脚本
- 需要做兼容、回滚、迁移或双写逻辑

## 推荐写法

```markdown
## 参考代码量估算
- 估算范围: 600-900 LoC
- 包含内容:
  - 服务主流程
  - TCP / TLS 适配层
  - 状态订阅与推理调度
  - 基础日志和指标
  - 单元 / 集成测试
- 不包含内容:
  - 下层框架自身改造
  - 新协议标准化工作
  - 自动化部署脚本大改
- 主要不确定性:
  - 现有通信组件复用程度
  - 下层推理框架线程安全约束
  - TLS 封装边界到底落在哪一层
- 置信度: 中
```

## 快速分档

### 小型

适用：

- 在现有模块里补局部能力
- 改动面集中
- 上下游和协议都稳定

常见范围：

- `100-300 LoC`

### 中型

适用：

- 需要新增 1 到 2 个明确模块
- 有接口适配、并发或状态处理
- 需要补若干测试和运维点

常见范围：

- `300-900 LoC`

### 大型

适用：

- 新增服务边界或大块基础设施
- 需要协议、安全、调度、观测多方面协同
- 验证和联调成本明显高

常见范围：

- `900+ LoC`

## 常见误区

- 把外部框架改造量算进当前服务代码量，却不说明边界
- 只估业务代码，不估测试、日志和配置改动
- 把“代码少”误判成“工作量小”
- 把估算写成单个数字，制造伪确定性

## 最小输出要求

最终至少写出：

- 范围
- 包含 / 不包含
- 主要不确定性
- 置信度
