# Embedded AI Inference Example

## 场景摘要

目标场景：

- 在嵌入式 SOC 系统上新增一个微服务进程
- 对外提供 AI 推理服务
- 需要订阅系统状态
- 接收其他服务通过 TCP 发来的请求
- 下层已有组件会设计交互接口
- 上层需要封装 OpenSSL
- 需要设计线程池处理并发请求
- AI 推理调用下层框架接口

这个例子最适合用来说明：新人 SE 不应该直接跳进方案，而应该先拆清“哪些是事实、哪些是待确认前提、哪些是方案选项”。

## Discovery 阶段应该先问什么

### 问题定义

- 新增这个微服务的业务目标是什么？
- 为什么必须做成独立进程，而不是扩展现有服务？
- 当前谁是直接调用方？返回结果给谁？

### 接口与 ownership

- TCP 协议由谁定义？是本团队、下层组件团队还是跨团队共同制定？
- 上层封装 OpenSSL 的职责边界是什么？只做 TLS 传输安全，还是还包括证书、鉴权、算法适配？
- 下层“设计交互接口”的组件负责到哪一层？是 socket 协议、消息格式，还是更高层的 SDK 契约？

### 并发与资源

- 并发高峰是多少？延迟目标是多少？
- 推理框架是否线程安全？是否支持多实例并行？
- 线程池是为了 I/O 并发、推理任务调度，还是保护底层硬件资源？

### 状态订阅

- 订阅的“系统状态”有哪些？配置、硬件状态、模型状态、健康状态还是安全状态？
- 状态变化时正在处理的请求如何受影响？
- 状态订阅是强一致要求，还是允许短时间滞后？

### 验收口径

- 什么结果算服务可用？
- 哪些错误码、超时、降级行为是可以接受的？
- 是否要求全链路加密、审计日志和指标？

## 对“为什么要封装 OpenSSL”的安全理解

在 discovery 阶段，不要把“要封装 OpenSSL”当作既定正确答案，而要把它当作待验证设计意图。

常见可能原因包括：

- 需要对 TCP 通道增加 TLS 能力
- 需要统一证书、密钥和握手策略
- 需要把安全逻辑与业务逻辑解耦
- 需要屏蔽不同平台或下层接口差异
- 需要把 OpenSSL 使用面收敛到一个可测试、可审计的适配层

也可能并不是这些原因，因此必须继续确认：

- 谁提出这一要求
- 这是否是平台硬约束
- 下层是否已经提供安全能力
- 上层封装是否会和下层接口职责重叠

## 研究阶段的典型问题

### 仓库内问题

- 当前系统是否已有 TCP / TLS 适配层或通用网络组件？
- 是否已有状态订阅、事件总线或回调机制？
- 是否已有线程池、任务队列或调度框架可复用？
- AI 推理框架当前是如何被其他模块调用的？

### 外部问题

- 嵌入式 SOC 上常见的推理服务并发模型是什么？
- TLS 终止应放在业务服务内，还是由更底层统一承接？
- 推理任务调度、超时和资源隔离的常见模式是什么？

## 候选方案示例

### Option A：单进程分层服务

结构：

- `TcpIngress`
- `TlsAdapter`
- `RequestParser`
- `StateSubscriber`
- `InferenceScheduler`
- `InferenceAdapter`
- `Observability`

优点：

- 对资源受限设备更直接
- 进程和运维形态简单
- 容易对照现有框架做最小集成

代价：

- 业务层、网络层和调度层必须严格分层，否则容易耦合
- 一个进程里同时处理网络并发和推理调度，边界设计要求高

### Option B：接入层与推理调度层逻辑解耦

结构：

- 接入层负责 TCP / TLS、协议校验和请求接收
- 调度层负责排队、线程池和推理框架调用
- 两层通过内部队列或明确接口协作

优点：

- 更容易隔离网络压力与推理资源压力
- 并发和降级策略更容易单独设计

代价：

- 组件更多，复杂度更高
- 在资源受限设备上要谨慎控制队列和内存

### Option C：进一步引入独立 broker 或 IPC 中介

优点：

- 解耦最强
- 扩展性更好

代价：

- 对嵌入式场景通常过重
- 引入额外部署和运维复杂度

## 默认推荐方向

如果当前平台资源受限、且不希望引入额外基础设施，通常更值得优先比较：

- `Option A` 的最小分层落地
- `Option B` 的轻量解耦版本

常见推荐结论会是：

- 在一个新微服务进程内保持单进程部署
- 明确拆出 `TLS / 安全适配层`、`请求接入层`、`推理调度层`、`下层推理适配层`
- 使用受控线程池或任务队列保护底层推理框架和硬件资源

但这仍然不是默认事实，必须结合仓库现状和线程安全约束来定。

## DFX 关注点

### 安全

- TLS 握手、证书轮换、连接鉴权
- 非法请求和重放攻击

### 性能

- 请求接入延迟
- 推理执行延迟
- 高并发时的排队深度

### 可靠性

- 下层推理框架失败时如何降级
- 状态变化期间如何处理中间请求

### 可测试性

- 能否 mock 下层推理接口
- 能否注入 TLS、网络和状态事件异常

### 可观测性

- 请求量、成功率、错误码、耗时、队列深度
- 线程池利用率或排队长度

## AR 示例

```markdown
### AR-001. 外部服务能够通过安全 TCP 通道提交推理请求
- 角色: 外部业务服务
- 目标: 提交推理请求并获取结果
- 业务价值: 板端推理能力可以被其他服务稳定复用
- 前提 / 依赖: TCP 协议、TLS 策略和错误码约定已明确
- 验收要点: 能建立连接、发送合法请求、获得成功或明确失败响应
- 风险 / 备注: TLS 握手耗时与连接资源上限是关键风险

### AR-002. 服务能根据系统状态安全地调度推理请求
- 角色: 推理服务自身
- 目标: 基于系统状态决定是否接收、排队、执行或拒绝请求
- 业务价值: 避免在资源受限或状态异常时失控
- 前提 / 依赖: 系统状态源和订阅语义已稳定
- 验收要点: 状态变化时服务行为符合预期的接收 / 降级 / 拒绝策略
- 风险 / 备注: 状态延迟和状态不一致会影响调度正确性
```

## 参考代码量示例

```markdown
## 参考代码量估算
- 估算范围: 600-1100 LoC
- 包含内容:
  - TCP / TLS 适配
  - 请求解析与错误处理
  - 状态订阅与调度逻辑
  - 下层推理适配
  - 基础日志、指标和测试
- 不包含内容:
  - 下层推理框架大改
  - 跨团队协议标准化
  - 专用部署平台改造
- 主要不确定性:
  - 现有网络组件和线程池复用程度
  - 推理框架线程安全和并发约束
  - TLS 封装边界是否已被下层部分覆盖
- 置信度: 中
```

## 这个例子真正想表达什么

不是“应该这么设计”，而是：

- 先问清问题
- 再做仓库和外部研究
- 再比较方案
- 最后才写设计包和工作量估算

这正是 `se-*` workflow 要帮助新人 SE 建立的工作习惯。
