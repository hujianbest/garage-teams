# AR Breakdown Rubric

## 默认理解

在这套 workflow 里，`AR` 默认按类似 User Story 的需求切片理解。

它的目的不是替代技术任务计划，而是把一个大方案切成“业务上可理解、实现上可继续细化、评审时可讨论”的需求单元。

## 一条 AR 至少要回答什么

每条 AR 至少回答下面 6 个问题：

1. 谁需要这个能力？
2. 想达成什么目标？
3. 为什么有价值？
4. 依赖什么前提？
5. 如何粗粒度验收？
6. 主要风险或备注是什么？

## 推荐字段

```markdown
### AR-001. <标题>
- 角色:
- 目标:
- 业务价值:
- 前提 / 依赖:
- 验收要点:
- 风险 / 备注:
- 工作量级别:
- 参考代码量范围:
- 主要工作项:
- 关键不确定性:
```

## 切分原则

优先按这些维度切分：

- 用户角色不同
- 业务场景不同
- 调用路径不同
- 风险和约束不同
- 验收口径不同

不要按这些维度切分：

- 代码目录
- 类名或模块名
- “前端 / 后端 / 测试”这类纯实现分工
- 某个工程师会做什么

## 好的 AR 长什么样

好的 AR 通常具备：

- 一句话就能说清场景
- 与其他 AR 边界相对清晰
- 能关联到 1 组主要风险和 1 组主要验收点
- 既不大到覆盖整个系统，也不小到只剩技术动作

## 不好的 AR 信号

- “实现推理服务线程池”
- “封装 OpenSSL”
- “补齐接口字段”
- “完成模块开发”

这些都是实现任务，不是需求切片。

但这不代表 AR 不能带工作量分析。

正确做法是：

- `AR` 本体继续描述需求切片
- 在同一个 AR 下追加“工作量级别、代码量范围、主要工作项、关键不确定性”

这样既能保持 AR 是需求视角，又能帮助团队讨论实现代价

## 示例

```markdown
### AR-001. 外部服务可通过安全 TCP 通道发起推理请求
- 角色: 外部调用服务
- 目标: 通过受控接口向 AI 推理微服务提交推理请求
- 业务价值: 让外部系统稳定接入板端推理能力
- 前提 / 依赖: 请求协议、鉴权方式和错误码约定已达成一致
- 验收要点: 能建立连接、发送合法请求、收到成功或明确失败响应
- 风险 / 备注: TLS 握手、证书轮换和连接资源上限可能成为关键风险
```

## 与任务拆解的边界

AR 分解之后，后续还可以继续拆成：

- 方案模块
- 接口设计
- 实现任务
- 测试任务

也就是说：

- `AR` 回答“用户 / 系统需要什么切片能力”
- 任务计划回答“团队如何分步实现它”
- AR 工作量分析回答“这个需求切片大概有多重、重在哪”

不要把两层混写。
