# Default DFX Lenses

## 使用原则

这份清单用于帮助新人 SE 在方案阶段主动检查“容易被忽略、但后面一定会追责”的质量维度。

不是每次都要把所有维度展开到同样深度，但至少要回答：

- 当前维度最大的风险是什么
- 当前方案如何应对
- 残余风险是什么
- 后续如何验证

## 1. 安全

关注：

- 身份认证、授权、证书、密钥、加密、输入校验、攻击面

常问问题：

- 谁能调用这个能力？
- 传输是否需要加密？
- 密钥、证书和凭据由谁管理？
- 是否存在伪造请求、重放、越权和数据泄露风险？

## 2. 性能

关注：

- 延迟、吞吐、队列堆积、热点路径、资源利用率

常问问题：

- 峰值并发是多少？
- 最关键的延迟指标是什么？
- 哪些步骤是热点路径？
- 是否需要限流、排队、批处理或缓存？

## 3. 可靠性

关注：

- 超时、重试、熔断、资源隔离、故障恢复、幂等

常问问题：

- 依赖失败时系统怎么退化？
- 请求是否可能重复？
- 是否需要重试、超时和隔离？
- 服务重启或断连后如何恢复？

## 4. 可测试性

关注：

- 单元测试入口、可模拟依赖、可注入错误、可复现实验条件

常问问题：

- 哪些依赖必须能 mock 或 stub？
- 关键失败路径如何验证？
- 是否容易做集成测试和回归测试？
- 哪些行为只能在线上验证，为什么？

## 5. 可维护性

关注：

- 职责边界、模块耦合、配置管理、代码可读性、团队认知负担

常问问题：

- 哪些模块职责最容易混乱？
- 当前方案是否把安全、协议、业务、推理逻辑解耦？
- 配置、开关和版本兼容如何管理？
- 新人是否容易理解和排查问题？

## 6. 可观测性

关注：

- 日志、指标、trace、健康检查、状态事件

常问问题：

- 出问题时先看什么信号？
- 哪些关键指标能区分网络问题、推理问题和依赖问题？
- 是否需要打点请求量、成功率、耗时和队列深度？
- 是否有诊断模式或健康探针？

## 7. 可部署性

关注：

- 依赖打包、版本升级、配置切换、环境差异、回滚

常问问题：

- 这个能力如何部署到目标环境？
- 是否依赖特定硬件、驱动或系统库？
- 升级和回滚会影响哪些上下游？
- 多环境之间有哪些差异？

## 8. 运行安全

关注：

- 资源耗尽、状态污染、推理输出异常、任务失控

常问问题：

- 高负载下是否会拖垮整机或共享资源？
- 是否需要保护下层模型或硬件加速资源？
- 任务堆积时如何避免雪崩？
- 错误输出是否会影响控制链或上游系统决策？

## 最小输出格式

每个维度推荐至少写成下面 4 行：

```markdown
### 安全
- 风险:
- 设计应对:
- 残余风险:
- 验证方式:
```

## 什么时候可以简写

如果某个维度当前明显不关键，可以简写为：

```markdown
### 可部署性
- 当前判断: 本轮不是主要风险维度
- 说明: 没有新增部署形态，沿用现有发布路径
```

但不要直接省略不写，除非已经明确说明“本轮不适用”。
