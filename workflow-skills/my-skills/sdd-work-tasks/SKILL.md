# MDC 任务拆解

创建任务计划，把已批准设计转化为可执行工作。

## 硬性门禁

在任务规划完成并准备好评审之前，不得开始实现。

## 前置条件

仅在以下条件满足时使用本 skill：

- 需求规格已批准
- 设计已批准

如果设计仍是草稿或还在修订中，请回到 `mdc-design-review` 或 `mdc-design`。

## 目标

任务计划必须让实现过程变得可预期。

它应明确以下内容：

- 里程碑
- 有序任务
- 依赖关系
- 任务级完成条件
- 验证预期

## 工作流

### 1. 阅读已批准输入

阅读：

- 已批准需求规格
- 已批准设计
- 当前项目上下文（如相关）

提取：

- 主要工作流
- 依赖与顺序
- 测试影响
- 风险或不确定区域

### 2. 定义里程碑

把工作分组为能产生明确阶段性成果的里程碑。

每个里程碑应包含：

- 目标
- 包含的任务
- 退出标准

### 3. 拆解为任务

任务必须足够小，能够被明确执行和验证。

优先使用这类任务形态：

- 编写失败测试
- 运行并确认失败
- 实现最小行为
- 运行并确认通过
- 重构并重新运行
- 更新状态与记录

避免使用这类模糊任务项：

- 实现某模块
- 完成功能
- 后面再优化

### 4. 记录依赖与完成条件

对每个任务明确：

- 前置条件
- 会触碰哪些工件
- 验证方式
- 完成条件

### 5. 编写任务计划

除非项目要求其他模板，否则使用以下结构：

```markdown
# <Topic> Task Plan

## Overview

## Milestones

## Task Breakdown

## Dependencies

## Definition Of Done

## Verification Strategy

## Risks And Sequencing Notes
```

### 6. 交给评审

任务计划准备好后，交给 `mdc-tasks-review`。

输出应使用：

```markdown
任务计划已起草完成，可进入评审。

Next skill: `mdc-tasks-review`
```

## 反模式

- 把任务计划写成设计文档副本
- 使用大到无法验证的任务
- 漏掉依赖关系
- 漏掉完成条件
- 把验证拖到整个功能结束时才做

## 完成条件

只有在存在一份可评审任务计划，并且已准备好交给 `mdc-tasks-review` 时，这个 skill 才算完成。
