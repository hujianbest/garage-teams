# MDC 需求澄清

创建一份定义“要做什么”的需求规格说明。

## 硬性门禁

在需求规格通过评审并获批之前，不得开始架构设计、任务拆解、脚手架或实现代码编写。

## 目标

需求规格必须足以让后续设计和实现不靠猜测推进。

它应回答以下问题：

- 解决的是什么问题
- 用户是谁
- 哪些内容在范围内
- 哪些内容在范围外
- 存在哪些约束
- 如何验证成功

## 工作流

按以下顺序执行。

### 1. 了解上下文

只阅读理解请求所需的材料：

- 用户提供的需求说明
- 与当前范围相关的现有项目文档
- 如果是增强需求，则阅读现有系统行为

提炼初始假设、约束和未知项。

### 2. 先澄清，再写文档

在起草规格前，先提出聚焦问题。

规则如下：

- 尽量一次只问一个主题
- 只有在能减少来回沟通时，才合并相关问题
- 把模糊表述替换成可衡量描述
- 如果用户未给出范围外内容，要主动询问

至少要澄清这些方面：

1. purpose and users
2. functional scope
3. exclusions
4. constraints and dependencies
5. acceptance criteria

### 3. 起草规格

编写一份把“需求”与“设计决策”分开的规格文档。

除非项目已有固定模板，否则使用以下结构：

```markdown
# <主题> 需求规格说明

## 目标

## 范围

## 用户角色

## 功能需求

## 非功能需求

## 约束

## 范围外内容

## 验收标准

## 开放问题
```

编写建议：

- 功能需求描述可观察行为
- 非功能需求描述可衡量质量目标
- 约束描述硬性限制
- 范围外内容必须显式写出，而不是隐含存在
- 开放问题只能保留那些不会阻塞评审的问题

### 4. 检查规格质量

交付前请确认：

- 需求可测试
- 模糊词已量化或删除
- 需求陈述中没有混入设计选择
- 范围外内容已显式说明
- 核心行为具备验收标准

### 5. 交给评审

草稿准备好后，交给 `mdc-spec-review`。

输出应写成：

```markdown
需求规格已起草完成，可进入评审。

Next skill: `mdc-spec-review`
```

## 反模式

- 从用户想法直接跳到架构设计
- 把头脑风暴笔记当成已批准规格
- 在规格里直接写任务
- 除非是硬性外部约束，否则不要使用 class、endpoint、table、framework 这类设计语言
- 把成功标准留成隐含信息

## 完成条件

只有在存在一份可评审的需求规格草稿，并且它已经准备好交给 `mdc-spec-review` 时，这个 skill 才算完成。
