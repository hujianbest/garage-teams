# MDC 收尾

把当前 MDC 工作周期干净地收尾。

## 目的

这个 skill 的目标是把“已经完成的工作”沉淀为“可持续的项目状态”。

它的存在是为了让下一次会话不必重新摸索：

- 做了什么
- 有哪些证据支持完成
- 对用户来说有哪些变化
- 接下来应该做什么

## 前置条件

只有在当前任务通过完成门禁之后，才能使用这个 skill。

## 工作流

### 1. 更新进度状态

在项目进度记录中补充：

- 已完成任务或范围项
- 日期或会话标记
- 当前状态
- 推荐的下一任务

### 2. 更新发布说明

如果完成的工作影响了用户可见行为，请在发布说明中简要记录：

- 变了什么
- 为什么重要

### 3. 记录证据位置

记录支持完成结论的证据在哪里：

- 缺陷模式排查结果（如有）
- 测试执行结果
- 评审结论
- 追溯性评审结果
- 回归结果
- 完成门禁结果

记录可以简洁，但必须支持后续审计和追溯。

### 4. 准备交接

明确说明：

- 哪些内容已经完成
- 还剩什么
- 哪些受影响工件已经同步刷新
- 如果下一次会话继续，正确的下一步 skill 是什么

## 输出格式

请严格使用以下结构：

```markdown
## 已完成工作

- 已完成项

## 已更新记录

- 已更新记录项

## 证据摘要

- 证据项

## 同步状态

- 已同步工件

## 下一步

`mdc-implement` | `mdc-increment` | `mdc-hotfix` | `workflow-complete`
```

## 反模式

- 宣告完成，却没有更新项目状态
- 把下一任务留成隐含信息
- 对用户可见变化忘记写发布说明
- 明明发生过变更或热修复，却不确认受影响工件是否已同步
- 在收尾过程中混入新的实现工作

## 完成条件

只有在项目状态、发布沟通和交接信息都更新到足以让下一次会话无须猜测地继续时，这个 skill 才算完成。
