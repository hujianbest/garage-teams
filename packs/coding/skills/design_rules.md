# AHE Skills Design Rules

## 1. 一个资产只解决一个问题

- 单个 `SKILL.md`、模板或说明文档应聚焦一个明确目标。
- 如果一个文件需要解释多个独立职责，优先拆分，而不是继续堆叠规则。

## 2. 分清规则、技能、模板和分析

- `AGENTS.md` 负责仓库级约束和工作方式。
- `ahe-coding-skills/` 负责按需激活的知识或操作约定；其中 **AHE workflow skills** 以 `ahe-coding-skills/ahe-*` 扁平目录承载端到端工作流片段。
- `templates/` 负责可复制的文档骨架。
- `docs/` 负责长文分析、研究记录和设计说明。

## 3. 只引用真实存在的路径

- 文档中写出的路径必须与仓库当前结构一致。
- 历史目录已移除时，应直接更新引用，而不是保留“兼容性”假设。
- 不要使用不存在的路径；workflow 能力应指向 `ahe-coding-skills/ahe-*` 下真实目录（见 `ahe-coding-skills/README.md`）。

## 4. 入口必须清晰

- skill 的入口固定为 `ahe-coding-skills/<skill-name>/SKILL.md`（workflow 族为 `ahe-coding-skills/ahe-*/SKILL.md`）。
- 模板应在开头说明适用场景和建议落点。
- 分析文档应说明研究对象和结论范围，避免把分析误当成流程入口。

## 5. 保持 Markdown-first

- 能用 Markdown 清楚表达的知识，不要过早做成脚本。
- 只有当能力需要重复执行、自动校验或跨环境复用时，再落成脚本或工具。
- 若需要 skill 工具链，优先复用 `.cursor/skills/skill-creator/`。

## 6. 证据优先于印象

- 对“已完成”“已验证”“可复用”的表述，优先给出命令、记录或文件证据。
- 模板应鼓励记录输入、结论和下一步，而不是只写主观判断。

## 7. 模板保持通用

- 模板应服务多个任务，而不是绑定单一已废弃流程名。
- 尽量使用通用字段，例如目标、范围、结论、风险、下一步。
- 若模板需要指向存放位置，优先使用当前仓库已存在的目录，例如 `docs/`。

## 8. 优先单人可维护性

- `ahe` 是个人工作台，新增结构应尽量轻量。
- 只有当某类资产会稳定复用时，才为它增加专门目录或复杂流程。
- 不为了“完整性”复制大型框架的全部层次。
