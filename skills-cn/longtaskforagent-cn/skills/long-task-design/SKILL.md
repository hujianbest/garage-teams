# 设计文档生成

以已批准 SRS 为输入。提出实现方案、按小节获得设计批准，产出回答 **HOW** 的设计文档 — SRS 回答 **WHAT**。

<HARD-GATE>
在已向用户展示设计并获得批准之前，不得调用任何实现技能、编写任何代码、搭建任何项目、运行 init_project.py 或采取任何实现动作。此规则适用于**所有**项目，无论其看似多简单。
</HARD-GATE>

## 反模式：「SRS 够细可以直接写代码」

SRS 描述系统**必须做什么**。设计文档描述**如何做**。即使需求非常清晰，实现路径（架构、数据模型、技术栈选择）仍须明确决策与用户批准。跳过设计会导致会话中途纠正与返工。

## 检查清单

你必须为下列每一项创建 TodoWrite 任务并按顺序完成：

1. **阅读已批准 SRS** — 自 `docs/plans/*-srs.md`
2. **探索技术上下文** — 既有代码、框架、部署环境
3. **提出 2–3 种方案** — 含权衡与推荐
4. **按小节批准设计** — 架构、数据模型、API、UI、测试、部署
5. **撰写设计文档** — 保存至 `docs/plans/YYYY-MM-DD-<topic>-design.md` 并提交
6. **转入 ATS** — **必选子技能：** 调用 `long-task:long-task-ats`

**终态为调用 long-task-ats。** 不得调用任何其他实现技能。

## 第 1 步：阅读 SRS、UCD 并提取设计输入

1. 从 `docs/plans/*-srs.md` 读取已批准 SRS
2. 从 `docs/plans/*-ucd.md` 读取已批准 UCD 风格指南（若存在 — 仅 UI 项目有）
3. 提取关键设计驱动：
   - **功能范围** — FR 数量、优先级分布、依赖链
   - **NFR 阈值** — 影响架构的性能、可靠性、可扩展性目标
   - **约束** — 限制技术/方案选择的硬限制
   - **接口需求** — 需对接的外部系统、协议、数据格式
   - **用户画像** — 影响 API/UI 设计决策的技术水平
   - **UCD 风格令牌**（若存在 UCD）— 色板、字体、间距、组件目录 → 影响前端架构与 UI/UX 小节
4. 列出设计推进前须解决的 SRS **Open Questions**
   - 若未决问题影响架构 → 第 2 步前通过 `AskUserQuestion` 询问用户

## 第 2 步：探索技术上下文

1. 探索项目将基于的既有代码 / 仓库
2. 识别 SRS 未载明的技术约束（如单体仓库结构、CI/CD、既有库）
3. 检查设计文档模板：
   - 若用户指定模板路径 → 读取并校验
   - 否则 → 读取 `docs/templates/design-template.md`（本技能默认模板）
   - **校验**：模板须为含至少一个 `## ` 标题的 `.md` 文件

## 第 3 步：提出方案

展示 **2–3 种实现方案**，明确写出权衡：

```markdown
## 方案 A：[名称]
**工作方式**：[1-2 句说明]
**优点**：[bullet list]
**缺点**：[bullet list]
**最适用场景**：[conditions]
**NFR 影响**：[该方案如何影响 SRS 中的 NFR 阈值]
**第三方依赖**：[该方案所需的关键库/框架及版本]

## 方案 B：[名称]
...

## 推荐：方案 [X]
**原因**：[为何它在 SRS 约束与 NFR 前提下最合适]
```

**要点**：每种方案须对照 SRS 约束与 NFR 阈值评估。无法满足「Must」级 NFR 的方案应淘汰。

## 第 4 步：按小节批准

非平凡项目将设计拆成小节，逐节批准：

1. **架构** — 系统组件、逻辑视图、技术栈决策
   - 必须包含 **Logical View**（Mermaid `graph`）：层/包/模块与依赖方向
   - 必须包含 **Component Diagram**（Mermaid `graph`）：运行时组件与交互
   - 必须对照 SRS 约束论证技术栈选择
   - 必须说明如何满足 NFR 阈值
2. **关键特性设计** — 每个关键特性或特性组一章
   - 每章**至少**包含：
     - **类图**（Mermaid `classDiagram`）— 类/模块、属性、方法、关系
     - **一种行为图**：序列图（Mermaid `sequenceDiagram`）或流程图（Mermaid `flowchart`）
   - 复杂特性应包含全部四种视图：类图、序列图、流程图与设计说明
   - 所有图**必须**使用 **Mermaid** — 不用 ASCII 艺术、不引用外部图片
3. **数据模型** — 模式、关系、存储策略
   - 适用处使用 Mermaid ER 图（`erDiagram`）
4. **API / 接口设计** — 端点、契约、协议
   - 须与 SRS 接口需求（IFR-xxx）一致
5. **UI/UX 方法**（若适用）— 版式策略、交互模式
   - 须覆盖 SRS 用户画像
   - 若存在 UCD 文档：须引用 UCD 风格令牌（色、字、间距）与组件目录
   - 前端架构决策（组件库、状态管理、路由）须与 UCD 风格令牌兼容
   - 包含映射：UCD 组件提示 → 具体实现组件
6. **第三方依赖** — **全部**库/框架须写**确切版本号**
   - 须验证依赖间相互兼容
   - 须验证与项目目标运行时版本兼容
   - 每条依赖注明许可证类型
   - 非平凡依赖链须含依赖图（Mermaid）
7. **测试策略** — 仅高层测试方法决策
   - 测试哲学：TDD + 质量门禁（Red → Green → Refactor → Coverage → Mutation）
   - 工具选择：测试框架、覆盖率工具、变异工具（含版本 — 属设计决策）
   - 覆盖率阈值：行 ≥ X%、分支 ≥ Y%、变异 ≥ Z%
   - **边界**：「详细的需求→测试类别映射、NFR 测试方法与跨特性集成场景在 ATS 阶段定义 — 不在此文档。」
8. **部署 / 基础设施**（若适用）— 托管、CI/CD、环境
9. **开发计划** — 里程碑、任务分解、优先级排序
   - 里程碑须有明确退出准则
   - 特性分解为优先级任务（P0–P3）
   - 依赖链（Mermaid `graph`）标明关键路径
   - 含风险评估与缓解措施

每节展示后等待反馈，修改后再进下一节。

**简单项目**（< 5 个特性）：所有小节合并为一次批准，但仍须含所需图与依赖版本。

## 第 5 步：撰写设计文档

将已批准设计保存至 `docs/plans/YYYY-MM-DD-<topic>-design.md`。

### 模板使用

读取第 2 步确定的模板（用户指定或默认 `docs/templates/design-template.md`）：
1. 保留模板标题结构
2. 用已批准内容替换各标题下引导文字
3. 若尚无元数据，在文首补充（`Date`、`Status`、`SRS Reference`、`Template` 路径）
4. 模板未覆盖小节：标 `"[Not applicable]"`
5. 已批准但无对应模板小节的内容：追加为「Additional Notes」

## 第 6 步：转入 ATS

设计文档保存并提交后：

1. 总结 ATS 技能所需关键输入：
   - **来自 SRS**：全部 FR/NFR/IFR 及验收标准
   - **来自 Design**：测试策略、技术栈、架构风险区域
2. **必选子技能：** 调用 `long-task:long-task-ats` 生成验收测试策略（ATS）

## 设计阶段规模

| 项目规模 | 特性数 | 设计深度 |
|---|---|---|
| Tiny | 1–5 | 一段式方案 + 1 次批准；逻辑视图 + 1 个特性图 + 依赖表 + 简化开发计划 |
| Small | 5–20 | 2–3 种方案 + 合并小节批准；逻辑视图 + 关键特性图 + 依赖表 + 里程碑计划 |
| Medium | 20–50 | 完整多小节批准；全部架构视图 + 每特性图 + 完整依赖分析 + 详细开发计划 |
| Large | 50–200+ | 完整多小节批准；每组特性全面图示 + 依赖兼容矩阵 + 分阶段开发与风险登记 |

## 红旗

| 自我合理化 | 正确应对 |
|---|---|
| 「SRS 已经暗示了架构」 | SRS 是 WHAT 不是 HOW。仍须展示选项。 |
| 「只有一种做法」 | 至少展示 2 种方案。显式权衡也有价值。 |
| 「我已知道最佳方案」 | 仍须展示选项，由用户选择 |
| 「用户不耐烦，跳过设计」 | 简短说明价值后高效执行 |
| 「边做边设计」 | 事前设计比中途纠正便宜 |
| 「在这里重新澄清需求」 | 需求属于 SRS。若缺失，记为 Open Question，设计前与用户解决。 |

## 图示要求

所有架构与设计视图**必须**使用 **Mermaid** 语法。这样可：
- 图示与文档一并版本管理（无外部图片文件）
- 在 GitHub、GitLab 与多数 Markdown 查看器中渲染
- 随设计变更同步更新

### 必选图示类型

| 小节 | 图示类型 | Mermaid 语法 | 是否必选 |
|---|---|---|---|
| Architecture Logical View | 分层包图 | `graph TB` | 始终 |
| Architecture Components | 组件交互 | `graph LR` | 始终 |
| Key Feature — Structure | 类图 | `classDiagram` | 每特性 |
| Key Feature — Behavior | 序列图 | `sequenceDiagram` | 每特性（至少一种行为图） |
| Key Feature — Logic | 流程/决策图 | `flowchart TD` | 每特性（至少一种行为图） |
| Data Model | ER 图 | `erDiagram` | 若有持久存储 |
| Dependency Graph | 依赖树 | `graph LR` | 若第三方依赖 > 3 |
| Development Plan | 关键路径 | `graph LR` | 始终 |

### 图示质量检查清单
- [ ] 每图有清晰标题或周围标题
- [ ] 类图含可见性（`+`/`-`/`#`）与关键方法
- [ ] 序列图含主成功路径与至少一条错误路径
- [ ] 流程图含全部分支逻辑的决策节点
- [ ] 无占位图 — 每图反映已批准的实际设计内容

## 第三方依赖规则

1. **必须写确切版本** — 指定 `1.2.3` 或约束范围 `^1.2.0` / `>=1.2,<2.0`；勿用 `latest` 或省略版本
2. **兼容矩阵** — 验证每条依赖与以下兼容：
   - 目标语言/运行时版本（如 Python >= 3.10、Node >= 18）
   - 栈内其他依赖（已知冲突须核查）
3. **许可证审计** — 记录每条依赖许可证；标注可能与项目要求冲突的 copyleft（GPL、AGPL 等）
4. **升级路径** — 注明接近 EOL 或已知迁移风险的依赖

## 开发计划规则

开发计划小节连接设计文档与 Init 阶段。**必须**包含：

1. **里程碑** — 有明确范围与退出准则的阶段
   - M1 恒为「Foundation」（项目骨架、CI、核心抽象）
   - 最后一程恒为「Polish & Release」（NFR 验证、文档、示例）
2. **任务分解** — 特性映射到优先级（P0–P3）并附理由
   - P0：基础 — 被所有其他特性依赖
   - P1：核心价值 — 最小可行特性集
   - P2：扩展 — 重要但不阻塞发布
   - P3：锦上添花 — 时间紧可推迟
3. **依赖链** — Mermaid 图展示特性间阻塞关系
4. **成对特性排序** — 项目同时含后端与前端时，任务分解表须使每个后端特性与其对应前端特性成对。自然开发流：Backend A → Frontend A → Backend B → Frontend B。Init 阶段据此在 `feature-list.json` 中排序特性。
5. **风险登记** — 技术与进度风险及缓解

Init 阶段用本计划填写 `feature-list.json` 的优先级、成对分组与依赖链。

## 集成

**由…调用：** using-long-task（SRS + UCD 已存在、无设计、无 feature-list.json）或 long-task-ucd（第 8 步）  
**需要：** `docs/plans/*-srs.md` 处已批准 SRS；UI 项目可选 `docs/plans/*-ucd.md` 处已批准 UCD  
**链接至：** long-task-ats（设计批准后）  
**产出：** `docs/plans/YYYY-MM-DD-<topic>-design.md`
