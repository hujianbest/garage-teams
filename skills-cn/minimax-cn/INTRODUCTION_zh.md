# MiniMax Skills 介绍文档

## 这套 Skills 是什么

`MiniMax Skills` 是一套面向 AI 编程助手的“领域技能库”。它不是简单的提示词集合，也不是单一应用的源码，而是一组可被 AI coding agent 触发和读取的专业操作手册。每个 skill 都围绕一个明确任务域展开，例如前端开发、全栈架构、移动端开发、Office 文档处理、图像分析、多模态内容生成或 GLSL Shader 开发。

它的核心目标是让 AI 在处理某类任务时，不再只依赖通用模型能力，而是先进入一个结构化、带约束、带流程、带参考资料和脚本入口的“专家模式”。

换句话说，这套仓库做的事情是：

- 把复杂任务拆成可复用的专业技能包。
- 把“应该如何做”写成显式流程、规则和检查清单。
- 把“具体怎么做”落到脚本、参考文档、模板和示例上。
- 让不同 AI 编程工具都能接入同一套技能资产。

## 适合谁看

这份技能库适合以下几类人理解和使用：

- 正在使用 Cursor、Claude Code、Codex、OpenCode 等 AI 编程工具的开发者。
- 想给 AI 增加稳定工作流和领域知识的团队。
- 需要让 AI 可靠处理文档、表格、PPT、多媒体、前端或全栈工程任务的人。
- 想研究“如何把 prompt 升级成可执行 skill”的开发者。

## 一句话理解它的设计思路

可以把它理解为：

“给 AI 编程助手安装一批按领域划分的专业 SOP + 工具箱 + 质量门禁。”

## 仓库整体结构

从当前目录可以看出，这套 skills 的顶层结构很清晰：

- `README.md` / `README_zh.md`
  - 负责介绍整套技能库、技能列表、安装方式和贡献方式。
- `skills/`
  - 真正的技能主体目录。每个子目录就是一个独立 skill。
- 各 skill 内部通常包含：
  - `SKILL.md`：技能主入口，定义触发条件、任务边界、工作流、强约束和导航方式。
  - `references/`：按主题拆分的参考文档，供 AI 按需阅读。
  - `scripts/`：可执行脚本或工具入口，把文档能力落到实际操作上。
  - `templates/` 或 `Samples/`：模板、样例或标准实现。

这意味着它不是“一个很长的总说明文档”，而是“一个总览 + 多个专业技能包”的组织方式。

## 这套 Skills 的主要分类

当前目录下共包含 14 个 skill，大致可以分成 4 组。

### 1. 工程开发类

- `frontend-dev`
  - 面向高质量前端页面与体验设计，覆盖 UI、动画、媒体素材生成、文案和生成艺术。
- `fullstack-dev`
  - 面向全栈系统设计与前后端集成，强调架构先行、接口设计、认证、实时能力和生产加固。
- `android-native-dev`
  - Android 原生开发，聚焦 Kotlin、Jetpack Compose、Material Design 3 和无障碍。
- `ios-application-dev`
  - iOS 应用开发，涵盖 UIKit、SnapKit、SwiftUI 和 Apple HIG。
- `flutter-dev`
  - Flutter 跨平台开发，覆盖状态管理、路由、性能和测试策略。
- `react-native-dev`
  - React Native / Expo 开发，覆盖组件、导航、状态管理、动画、部署和 CI/CD。
- `shader-dev`
  - 实时图形与 GLSL Shader 开发，按技术主题组织了大量 technique 与 reference 文档。

### 2. 文档与办公自动化类

- `minimax-docx`
  - DOCX 的创建、编辑、套模板和验证，强调 OpenXML 的结构正确性与排版质量。
- `minimax-xlsx`
  - Excel/表格的读取、创建、编辑和校验，强调公式优先、零格式损失和 XML 级编辑。
- `minimax-pdf`
  - PDF 的生成、填写和重排，强调文档设计系统和面向打印的输出质量。
- `pptx-generator`
  - PPT 的创建、编辑和读取，支持从零生成和 XML 工作流编辑。

### 3. 多模态与内容生成类

- `minimax-multimodal-toolkit`
  - MiniMax 多模态统一入口，覆盖 TTS、音乐、图片、视频和 FFmpeg 媒体处理。
- `gif-sticker-maker`
  - 把照片转换成带风格和字幕的 GIF 贴纸，偏创意内容生产。

### 4. 感知与分析类

- `vision-analysis`
  - 基于视觉模型对图片做描述、OCR、UI 审查、图表数据提取和目标识别。

## 这套 Skills 不是“文档集合”，而是“可执行的技能规范”

从多个 `SKILL.md` 的设计可以看出，这套 skills 有几个非常鲜明的特点。

### 1. 每个 skill 都有明确的元数据和触发语义

大部分 skill 在开头都使用类似 YAML frontmatter 的结构定义：

- `name`
- `description`
- `license`
- `metadata`
- `triggers`（部分 skill 会有）

这让 AI 工具可以更容易判断：

- 什么时候应该触发这个 skill。
- 这个 skill 处理什么任务。
- 它属于哪个能力域。

例如：

- `fullstack-dev` 会显式写出“什么时候应该触发”和“什么时候不该触发”。
- `minimax-docx` 会把“只要最终产物是正式 Word 文档”都归入它的适用范围。
- `vision-analysis` 会把图片扩展名、OCR、截图分析、UI review 等都纳入触发条件。

这说明它的设计目标不是“给人读”，而是“既给人读，也给 agent 判断和调用”。

### 2. 每个 skill 都强调任务路由

这套仓库的常见模式不是“把所有知识一股脑写进去”，而是先做任务分流，再读具体参考材料。

典型例子：

- `shader-dev` 用 Technique Routing Table 把用户需求映射到具体技术主题。
- `minimax-docx` 用 Pipeline A/B/C 区分创建、编辑、套模板三类任务。
- `minimax-xlsx` 用 READ / CREATE / EDIT / FIX / VALIDATE 进行任务路由。
- `vision-analysis` 用 `describe`、`ocr`、`ui-review`、`chart-data`、`object-detect` 区分分析模式。

这样的设计非常适合 AI：

- 先决定走哪条路径。
- 再只读取当前路径需要的参考文档。
- 避免在一个超长主文档中盲目搜索。

### 3. 每个 skill 都有“强约束”，不是松散建议

这套 skills 的一个突出特点是：大量规则是“必须遵守”，不是“可以参考”。

例如：

- `frontend-dev` 明确规定不要使用占位图 URL、必须先生成真实素材、必须有 loading/empty/error 状态、不能混用某些动画方案。
- `fullstack-dev` 强调 feature-first 结构、typed error、环境变量集中配置、请求日志、健康检查和优雅停机。
- `minimax-docx` 强调 OpenXML 元素顺序、模板套用时的 gate-check、样式污染清理和多 section 页眉页脚保留。
- `minimax-xlsx` 强调禁止 openpyxl round-trip 编辑既有文件、所有计算优先用公式、编辑后必须验证。
- `minimax-multimodal-toolkit` 对输出目录、环境变量、API host、套餐额度、视频分辨率与时长都有明确限制。

这说明这套技能库的核心价值不是“提供灵感”，而是“减少 AI 瞎做和做错的概率”。

### 4. 大量 skill 都把操作落到了脚本和文件结构上

很多 skill 不是停留在提示词层面，而是已经配好了脚本和操作路径。

例如：

- `frontend-dev` 提供图片、视频、音频、TTS 的脚本入口。
- `minimax-docx` 提供 CLI、C# 脚本、文档预览、校验与 diff 流程。
- `minimax-xlsx` 提供 unpack / pack / formula check / insert row / add column 等工具脚本。
- `minimax-multimodal-toolkit` 提供 TTS、音乐、图像、视频和 FFmpeg 脚本。

这意味着 skill 的输出不是“建议你这样做”，而是“告诉 agent 应该走哪条命令路径，并给出操作规则”。

### 5. 参考资料采用“按需加载”的知识库设计

几乎所有较复杂的 skill 都把知识拆成两层：

- 主 skill 文档负责流程、边界、路由和关键规则。
- `references/` 负责深入说明某个子话题。

这种设计有几个好处：

- 主入口不至于失控膨胀。
- AI 可以根据任务只读必要部分。
- 不同能力模块可以独立维护和演进。

从架构上看，这是一种很典型的“轻入口 + 深参考”的知识工程设计。

## 几个代表性 Skill，能帮助快速理解整套库的风格

### `frontend-dev`: 把“前端页面开发”扩展成完整生产流程

这个 skill 并不把前端理解为“写几个组件”，而是把它拆成 5 个协同能力：

- 设计工程
- 动效系统
- 媒体素材生成
- 文案生成
- 生成艺术

它说明 MiniMax Skills 的一个重要方向：不仅关注代码本身，也关注最终交付体验。

### `fullstack-dev`: 强调先做架构决策，再写代码

这个 skill 的显著特点是先要求：

- 收集需求
- 明确架构决策
- 按 checklist 脚手架化
- 实现
- 测试与交付

这体现了该库的另一个设计理念：AI 不应直接开写，而应先做工程决策和风险控制。

### `minimax-docx` 与 `minimax-xlsx`: 面向 Office 文件的工程化技能

这两个 skill 很能体现“文档处理也能工程化”的思路：

- 不只是会读写文件。
- 而是有任务分流、工具链、校验、最小改动原则和结果验证。

它们对“输出文件不能坏、格式不能乱、公式不能丢”这类真实世界需求考虑得非常细。

### `minimax-multimodal-toolkit`: 为多模态任务建立统一入口

这个 skill 把 TTS、音乐、视频、图片和媒体后处理收敛到一个统一入口下，并补充了：

- 环境配置要求
- 输出目录约束
- 套餐额度与模型限制
- prompt 优化规则

这说明它不是“展示模型能做什么”，而是“指导 agent 如何安全、可控地调用模型能力”。

### `shader-dev`: 典型的“主题路由 + 技术百科”设计

`shader-dev` 的规模较大，但结构非常清晰：

- 一个主入口文档
- 多个 technique 文件
- 对应的 reference 深入材料

它适合用来理解这套仓库如何处理复杂知识域：不是写一篇超长总览，而是按任务和技术点拆解成可组合模块。

## 这套 Skills 的共同设计模式

综合多个 skill，可以总结出 MiniMax Skills 的共同设计语言。

### 模式 1：先判断场景，再进入流水线

几乎每个 skill 都会先回答两个问题：

- 当前任务属于哪一类？
- 应该走哪条标准路径？

这让 skill 更像“任务调度器”，而不是纯说明文档。

### 模式 2：把经验固化成规则与禁忌

skill 文档中常见大量：

- MUST
- NEVER
- DO NOT
- REQUIRED
- blocking error

这代表作者希望 AI 避开常见失败模式，而不是只给模糊建议。

### 模式 3：主文档负责决策，参考文档负责细节

这是一种非常适合 agent 的结构：

- 主文档足够短，能快速进入状态。
- 参考文档足够细，能支撑复杂任务。

### 模式 4：能脚本化的地方尽量脚本化

如果一个任务有稳定执行方式，skill 往往会给出脚本入口、命令模板或工具链，而不是只停留在自然语言层面。

### 模式 5：强调结果校验

很多 skill 会要求：

- 构建验证
- 输出预览
- 结构校验
- 公式校验
- diff 检查
- 质量门禁

这说明它非常重视“交付物正确”，而不是“过程看起来合理”。

## 它和普通 Prompt 模板的区别

如果只把它当成 prompt 模板，会低估它的价值。它和普通 prompt 集合的差异主要在这里：

- 它有明确的任务边界，而不是泛泛而谈。
- 它有触发条件，而不是靠用户手工挑选。
- 它有工作流，而不是只给若干建议。
- 它有脚本与模板支撑，而不是只有文字描述。
- 它有质量门禁，而不是生成后就结束。
- 它支持多个 AI 工具接入，而不是绑定单一产品。

所以更准确地说，它是一套“可被 AI 执行的专业操作规范”。

## 这套 Skills 的价值

对于团队或个人而言，这套技能库最直接的价值有 5 点：

- 降低 AI 输出质量波动，让复杂任务更稳定。
- 把专家经验转成可复用资产，而不是藏在个别人脑中。
- 让不同工具共享同一套做事方式。
- 让 AI 不只是会写代码，还会遵循流程、约束和质量标准。
- 对文档、表格、多媒体等“非传统代码任务”提供可操作支持。

## 适合怎么向别人介绍它

如果需要对外快速介绍，可以直接这样说：

> MiniMax Skills 是一套给 AI 编程助手使用的专业技能库。每个 skill 都围绕一个具体任务域，定义了触发条件、标准工作流、强约束、参考资料和可执行脚本，让 AI 能更像一名按规范办事的专业工程师，而不是只会即兴生成内容的聊天模型。

如果要再通俗一点，可以说：

> 它相当于给 AI 安装了一批“专业插件”，让 AI 在做前端、全栈、Office 文档、多模态内容生成等任务时，有固定的方法论、工具链和质量门槛可遵循。

## 推荐的阅读顺序

如果是第一次接触这套仓库，建议按下面顺序看：

1. 先看 `README.md` 或 `README_zh.md`
   - 了解技能矩阵、安装方式和整体定位。
2. 再挑 2 到 3 个代表性 skill 阅读 `SKILL.md`
   - 推荐先看 `fullstack-dev`、`minimax-docx`、`minimax-multimodal-toolkit`。
3. 最后按需进入某个 skill 的 `references/`
   - 观察它如何把复杂主题继续拆分。

这样最容易理解这套 skills 的设计哲学，而不是只看到表面功能。

## 总结

MiniMax Skills 的本质，不是一个“AI 能力展示仓库”，而是一套面向 AI agent 的任务工程化资产。

它的关键设计点在于：

- 用 skill 将任务域模块化。
- 用 frontmatter 和触发规则帮助 agent 选对能力。
- 用标准工作流和强约束控制执行过程。
- 用脚本、模板和参考资料承接落地操作。
- 用校验、门禁和检查清单保障交付质量。

如果要用一句话概括：

`MiniMax Skills = 面向 AI 编程助手的可执行领域知识库 + 标准作业流程 + 工具链约束。`
