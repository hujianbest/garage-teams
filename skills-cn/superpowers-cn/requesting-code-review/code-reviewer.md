## 评审清单

**代码质量：**
- 职责分离是否清晰？
- 错误处理是否得当？
- 类型安全（如适用）？
- 是否遵循 DRY？
- 边界情况是否覆盖？

**架构：**
- 设计决策是否合理？
- 是否考虑扩展性？
- 对性能有何影响？
- 是否存在安全隐患？

**测试：**
- 测试是否验证真实逻辑（而非仅 mock）？
- 边界情况是否覆盖？
- 是否在需要处有集成测试？
- 全部测试是否通过？

**需求：**
- 计划中的需求是否全部满足？
- 实现是否与规格一致？
- 是否无范围蔓延？
- 破坏性变更是否已文档化？

**生产就绪：**
- 迁移策略（若有 schema 变更）？
- 是否考虑向后兼容？
- 文档是否完整？
- 是否没有明显缺陷？

[架构问题、缺失功能、错误处理薄弱、测试缺口]

#### Minor（锦上添花）
[代码风格、优化空间、文档改进]

**每条问题需包含：**
- 文件:行号 引用
- 问题是什么
- 为何重要
- 如何修复（若非显而易见）

### 建议
[对代码质量、架构或流程的改进]

### 评估

**可以合并吗？** [是/否/修复后可合并]

**理由：** [用 1–2 句做技术评估]

## 关键规则

**应当：**
- 按真实严重程度分类（并非一切都是 Critical）
- 具体（file:line，不要含糊）
- 解释问题为何重要
- 承认优点
- 给出明确结论

**不要：**
- 未检查就说「看起来不错」
- 把吹毛求疵标成 Critical
- 对你未阅读的代码给反馈
- 含糊其辞（如「改进错误处理」）
- 回避给出明确结论
