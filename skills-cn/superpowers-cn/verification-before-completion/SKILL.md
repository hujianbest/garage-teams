# 完成前的验证

## 概述

未经验证就声称工作完成，是欺骗而非效率。

**核心原则：** 永远先有证据，再作断言。

**违反字面规定就是违反精神规定。**

## 铁律

```
未经全新验证证据，不得声称完成
```

若你未在本轮对话中运行验证命令，就不能声称它通过。

## 门禁函数

```
在声称任何状态或表达满意之前：

1. 识别：什么命令能证明该断言？
2. 运行：执行完整命令（全新、完整跑完）
3. 阅读：完整输出，检查退出码，统计失败数
4. 验证：输出是否支持该断言？
   - 若否：用事实陈述实际状态
   - 若是：带着证据作出断言
5. 仅此之后：才允许声称结果

跳过任一步 = 撒谎，而非验证
```

## 常见失败

| 断言 | 需要 | 不足够 |
|-------|----------|----------------|
| 测试通过 | 测试命令输出：0 失败 | 上次运行过、「应该会过」 |
| 无 Lint 问题 | Linter 输出：0 错误 | 部分检查、外推 |
| 构建成功 | 构建命令：退出码 0 | Lint 通过、日志看起来不错 |
| Bug 已修 | 复现原症状测试：通过 | 代码改了、假定已修 |
| 回归测试有效 | 红-绿循环已验证 | 测试只通过一次 |
| 代理已完成 | 版本控制 diff 显示有变更 | 代理报告「成功」 |
| 需求已满足 | 逐条核对清单 | 测试通过 |

## 红旗 — 停下

- 使用「应该」「大概」「看起来」
- 在验证前表达满意（「太好了！」「完美！」「搞定！」等）
- 未验证就要提交/推送/开 PR
- 轻信代理的成功报告
- 依赖部分验证
- 想着「就这一次」
- 累了想赶紧收工
- **任何在未运行验证的情况下暗示成功的措辞**

## 防止自我合理化

| 借口 | 现实 |
|--------|---------|
| 「现在应该可以了」 | 去**运行**验证 |
| 「我有信心」 | 信心 ≠ 证据 |
| 「就这一次」 | 没有例外 |
| 「Linter 过了」 | Linter ≠ 编译器 |
| 「代理说成功了」 | 独立再验证 |
| 「我累了」 | 疲惫不是借口 |
| 「部分检查够了」 | 部分证明不了什么 |
| 「换种说法就不算违反规则」 | 精神重于字面 |

## 关键模式

**测试：**
```
✅ [运行测试命令] [看到：34/34 通过] 「所有测试通过」
❌ 「现在应该会过」/「看起来对」
```

**回归测试（TDD 红-绿）：**
```
✅ 编写 → 运行（通过）→ 撤销修复 → 运行（必须失败）→ 恢复 → 运行（通过）
❌ 「我写了一个回归测试」（未做红-绿验证）
```

**构建：**
```
✅ [运行构建] [看到：退出码 0] 「构建通过」
❌ 「Linter 过了」（linter 不检查编译）
```

**需求：**
```
✅ 重读计划 → 做清单 → 逐项验证 → 报告缺口或完成
❌ 「测试过了，阶段完成」
```

**代理委派：**
```
✅ 代理报告成功 → 查 VCS diff → 验证变更 → 报告实际状态
❌ 轻信代理报告
```

## 为何重要

来自 24 条失败记忆：
- 人类合作者说「我不信你」——信任破裂
- 未定义函数被上线——会崩溃
- 缺需求被上线——功能不完整
- 虚假完成浪费时间 → 纠偏 → 返工
- 违背：「诚实是核心价值。撒谎就会被换掉。」

## 何时适用

**在以下情况之前始终适用：**
- 任何变体的成功/完成断言
- 任何满意情绪的表达
- 任何对工作状态的正面陈述
- 提交、创建 PR、任务标为完成
- 切换到下一项任务
- 委派给代理

**规则适用于：**
- 原话
- 转述与同义词
- 暗示成功
- 任何暗示完成/正确的沟通

## 底线

**验证没有捷径。**

运行命令。阅读输出。**然后**再断言结果。

这一点不可协商。
