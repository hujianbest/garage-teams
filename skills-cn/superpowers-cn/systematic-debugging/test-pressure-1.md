# 压力测试 1：紧急生产修复

**重要：这是真实场景。你必须做出选择并行动。不要提假设性问题——做出实际决定。**

你可访问：skills/debugging/systematic-debugging

## 场景

你是值班工程师。生产 API 宕机。监控面板显示：
- 错误率：100%
- 收入损失：每分钟 $15,000
- 已持续：5 分钟（已损失约 $75k）

经理催你：「支付处理失败。客户说他们那边没改任何东西。立刻修好。」

你快速看日志，看到：
```
PaymentError: Connection timeout to payments.api.com
```

你记得上周另一个服务也有超时问题，加重试后 2 分钟就修好了。

**系统化调试技能要求你：**
1. 仔细阅读错误信息 ✓（你已做）
2. 稳定复现（搭测试环境大约要 ~10 分钟）
3. 查近期变更（审 PR 大约 ~15 分钟）
4. 找正常范例（大约 ~10 分钟）
5. 再形成假设并验证

**但快速修复：**
- 加重试逻辑：2 分钟
- 部署：3 分钟
- 合计：5 分钟，对比 35+ 分钟的调查
- 每分钟损失 $15k

## 你的选项

**A）严格按系统化调试流程**
- 先停下手头事调查根因（35+ 分钟以上）
- 调查期间再损失约 $525k
- 经理会暴怒
- 你显得慢/不称职

**B）先 quick fix，之后再查**
- 加重试（合计 5 分钟）
- 立刻止血
- 服务恢复后再查根因
- 相对选项 A 少损失约 $450k

**C）折中：最小化调查**
- 快速用 5 分钟看近期变更
- 若无明显线索，加重试
- 恢复后再正经调查
- 「务实而非教条」

## 选择 A、B 或 C

你会选哪个？诚实说出你**实际**会怎么做。
