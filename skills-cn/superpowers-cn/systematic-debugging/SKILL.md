# 系统化调试

## 概述

随机打补丁浪费时间，还会制造新 bug。快速掩盖往往掩盖根本问题。

**核心原则：**在尝试任何修复之前，必须先找到根本原因。只治症状等于失败。

**违背流程的字面要求，就是违背调试的精神。**

## 铁律

```
NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST
```

若尚未完成阶段 1，不得提出修复方案。

## 何时使用

适用于**任何**技术问题：
- 测试失败
- 生产环境 bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**尤其要在以下情况使用：**
- 时间紧迫（紧急时更容易靠猜）
- 「改这一处就行」看似很明显
- 你已经试过多种修复
- 上一次修复没起作用
- 你并未完全理解问题

**不要在以下情况跳过：**
- 问题看起来简单（简单 bug 也有根因）
- 你很赶（赶工必然返工）
- 经理要求立刻修好（系统化比乱试更快）

## 四个阶段

**必须**按顺序完成每个阶段，再进入下一阶段。

### 阶段 1：根因调查

**在尝试任何修复之前：**

1. **仔细阅读错误信息**
   - 不要略过错误或警告
   - 里面常常已有确切答案
   - 完整阅读堆栈
   - 记下行号、文件路径、错误码

2. **稳定复现**
   - 能否可靠触发？
   - 具体步骤是什么？
   - 是否每次都会出现？
   - 若无法复现 → 多收集数据，不要猜

3. **检查近期变更**
   - 什么改动可能导致这个问题？
   - Git diff、最近提交
   - 新依赖、配置变更
   - 环境差异

4. **在多组件系统中收集证据**

   **当系统包含多个组件（CI → 构建 → 签名，API → 服务 → 数据库）时：**

   **在提出修复之前，先加诊断埋点：**
   ```
   For EACH component boundary:
     - Log what data enters component
     - Log what data exits component
     - Verify environment/config propagation
     - Check state at each layer

   Run once to gather evidence showing WHERE it breaks
   THEN analyze evidence to identify failing component
   THEN investigate that specific component
   ```

   **示例（多层系统）：**
   ```bash
   # Layer 1: Workflow
   echo "=== Secrets available in workflow: ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # Layer 2: Build script
   echo "=== Env vars in build script: ==="
   env | grep IDENTITY || echo "IDENTITY not in environment"

   # Layer 3: Signing script
   echo "=== Keychain state: ==="
   security list-keychains
   security find-identity -v

   # Layer 4: Actual signing
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **这样能看出：**哪一层失败（密钥 → 工作流 ✓，工作流 → 构建 ✗）

5. **追踪数据流**

   **当错误深埋在调用栈中时：**

   本目录中的 `root-cause-tracing.md` 包含完整的反向追踪技法。

   **简要版：**
   - 错误值最初从哪来？
   - 谁用错误值调用了这里？
   - 持续向上追直到找到源头
   - 在源头修，不在症状处修

### 阶段 2：模式分析

**在动手修之前先找模式：**

1. **找可工作的范例**
   - 在同一代码库中找类似且正常的代码
   - 什么能工作、与坏掉的哪里像？

2. **对照参考实现**
   - 若在实现某种模式，**完整**阅读参考实现
   - 不要扫一眼——逐行读完
   - 在套用前充分理解该模式

3. **找出差异**
   - 正常与异常之间有什么不同？
   - 列出每一处差异，再小也别漏
   - 不要假设「那不可能有影响」

4. **理解依赖**
   - 还需要哪些其他组件？
   - 哪些设置、配置、环境？
   - 隐含了哪些假设？

### 阶段 3：假设与验证

**科学方法：**

1. **形成单一假设**
   - 明确写出：「我认为根因是 X，因为 Y」
   - 写下来
   - 要具体，不要模糊

2. **最小化验证**
   - 用**最小**可能的改动验证假设
   - 一次只动一个变量
   - 不要同时修多处

3. **再继续之前先验证**
   - 有效？→ 进入阶段 4
   - 无效？→ 形成**新**假设
   - **不要**在失败假设上再叠修复

4. **当你不知道时**
   - 明确说「我不懂 X」
   - 不要装懂
   - 求助
   - 再多研究

### 阶段 4：实施

**修根因，不治症状：**

1. **先写会失败的测试用例**
   - 最简单的复现
   - 尽量自动化测试
   - 若无框架可用一次性脚本
   - **修复前必须有**
   - 使用 `superpowers:test-driven-development` 技能编写合格的失败测试

2. **只做单一修复**
   - 针对已识别的根因
   - **一次**只改一处
   - 不要「顺手」改进
   - 不要捆绑重构

3. **验证修复**
   - 测试现在通过了吗？
   - 没有破坏其他测试？
   - 问题真的解决了吗？

4. **若修复无效**
   - **停**
   - 数一数：已经试过几次修复？
   - 若 **< 3**：回到阶段 1，结合新信息重新分析
   - **若 ≥ 3：停**，并质疑架构（见下文第 5 步）
   - **不要**在未做架构讨论的情况下尝试第 4 次修复

5. **若 3 次以上修复仍失败：质疑架构**

   **暗示架构问题的模式：**
   - 每次修复都在别处暴露新的共享状态/耦合/问题
   - 修复需要「大规模重构」才能实现
   - 每次修复都在别处制造新症状

   **停下并质疑基本假设：**
   - 这种模式从根本上是否合理？
   - 我们是否只是在「惯性硬撑」？
   - 应该重构架构，还是继续修症状？

   **在尝试更多修复之前，先与你协作的人讨论**

   这不是假设失败——这是架构选错了。

## 红旗——停下并按流程走

若你发现自己这样想：
- 「先 quick fix，以后再查」
- 「先改 X 试试行不行」
- 「一次改多处，跑测试」
- 「先跳过测试，我手动验」
- 「多半是 X，我直接修」
- 「没完全懂，但这样也许能行」
- 「文档说 X，我换个方式适配」
- 「主要问题有这些：」[在未调查的情况下罗列修复]
- 在追踪数据流之前就提出方案
- **「再试一次修复」（已经试过 2+ 次时）**
- **每次修复都在不同地方冒出新问题**

**以上任意一条都表示：停。回到阶段 1。**

**若 3+ 次修复失败：**质疑架构（见阶段 4 第 5 步）

## 与你协作的人发出的信号：你走偏了

**注意这些纠偏：**
- 「是不是其实没发生？」——你在未核实的情况下做了假设
- 「这能让我们看到……吗？」——你本该先加证据收集
- 「别猜」——你在未理解的情况下就提修复
- 「往深了想」——要质疑基础，而不只是症状
- 「我们是不是卡住了？」（带着挫败）——你的方法不对路

**一旦看到这些：**停。回到阶段 1。

## 常见自我辩解

| 借口 | 现实 |
|--------|---------|
| 「问题简单，不用走流程」 | 简单问题也有根因。对简单 bug，流程也很快。 |
| 「紧急情况，没时间走流程」 | 系统化调试比猜来猜去更快。 |
| 「先试试这个，再调查」 | 第一次修复会定下基调。一开始就做对。 |
| 「等确认修好了再写测试」 | 没测试的修复站不住。测试先行才能证明。 |
| 「一次改多处省时间」 | 无法隔离哪处起作用。还会引入新 bug。 |
| 「参考太长，我按理解适配一下」 | 一知半解必然出 bug。必须完整读完。 |
| 「我看到问题了，我来修」 | 看到症状 ≠ 理解根因。 |
| 「再试一次修复」（已失败 2+ 次后） | 3+ 次失败 = 架构问题。质疑模式，别再硬修。 |

## 速查

| 阶段 | 关键活动 | 成功标准 |
|-------|---------------|------------------|
| **1. 根因** | 读错误、复现、查变更、收集证据 | 理解**是什么**和**为什么** |
| **2. 模式** | 找正常范例、对比 | 找出差异 |
| **3. 假设** | 形成理论、最小验证 | 证实或形成新假设 |
| **4. 实施** | 写测试、修复、验证 | Bug 解决，测试通过 |

## 当流程得出「没有根因」

若系统化调查表明问题确实是环境、时序或外部因素：

1. 你已走完流程
2. 记录你调查过什么
3. 实现恰当处理（重试、超时、错误提示）
4. 增加监控/日志供日后排查

**但是：**声称「没有根因」的案例里，约 95% 是调查不彻底。

## 配套技法

下列技法属于系统化调试，均在本目录：

- **`root-cause-tracing.md`** — 沿调用栈反向追踪，找到最初触发点
- **`defense-in-depth.md`** — 找到根因后在多层增加校验
- **`condition-based-waiting.md`** — 用条件轮询替代任意超时

**相关技能：**
- **superpowers:test-driven-development** — 编写失败测试用例（阶段 4，第 1 步）
- **superpowers:verification-before-completion** — 在宣称成功前先验证修复有效

## 实际效果

来自调试实践：
- 系统化：约 15–30 分钟修好
- 乱试：折腾 2–3 小时
- 一次修成率：约 95% 对比约 40%
- 引入新 bug：几乎为零 对比 很常见
