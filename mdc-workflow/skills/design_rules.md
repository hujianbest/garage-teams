# MDC Skills Design Rules

## 1. 路由优先于执行

- 所有 workflow 场景先经过 `mdc-workflow-starter`，再决定当前阶段、profile 与下一步 skill。
- `mdc-*` skills 应被视为状态机节点，而不是可自由挑选的能力列表。
- 下游 skill 负责完成本节点职责，不负责自行决定后续编排。
- 路由完成后，若下一步不是暂停点，应按连续执行原则立刻进入下一节点，而不是停下来等待额外确认。

## 2. 阶段顺序必须显式受控

- 主链顺序固定为：规格 -> 设计 -> 任务 -> 实现 -> 质量链 -> 完成门禁 -> 收尾。
- 规格未批准，不进入设计；设计未批准，不进入任务；任务未批准，不进入实现。
- `review` 的“通过”只表示具备进入确认或下游的条件，不等于自动批准。
- 规格与设计都保留“评审通过”和“真人确认通过”两个层次，不能合并。

## 3. Workflow Profile 用来调密度，不用来降门禁

- `full`、`standard`、`lightweight` 只决定当前需要经过哪些节点，不降低节点内部要求。
- Profile 必须由入口 skill 根据信号与工件状态决定，不能由用户或下游 skill 自行声称。
- 当信号冲突或证据不足时，选择更重的 profile。
- Profile 允许升级，不允许降级。

## 4. 工件驱动，不依赖聊天记忆

- 阶段状态、批准结论、评审结果、验证证据应落在仓库工件中，而不是停留在对话上下文里。
- 路由和门禁判断只读取最少但必要的内容，避免在阶段判断前做大范围探索。
- 若状态、批准或证据不清楚，默认按“未批准”或“证据不足”处理。
- 逻辑工件与实际文件路径分离；团队项目的具体映射由 `AGENTS.md` 提供。

## 5. 强门禁优先于“灵活处理”

- 目录级规则应使用“必须 / 不得 / 仅在...时允许”这类强约束语言，而不是模糊建议。
- 每个关键节点都应能明确回答：前置条件是什么、禁止跳过什么、完成后唯一下一步是什么。
- 当证据冲突时，优先回到更保守的上游节点，而不是向下游猜测推进。
- 红旗信号和反模式不是补充说明，而是防止 agent 为跳步找理由的核心设计。

## 6. 每一层只做本层的事

- 规格只定义“做什么、为什么、如何算成功”，不提前做设计决策。
- 设计回答“如何实现”，但保持在设计层抽象，不写成伪代码或任务清单。
- 任务计划负责把已批准设计翻译成可执行工作，不复制规格或设计正文。
- 实现阶段只在已批准任务约束下推进当前活跃任务，不顺手扩大范围。

## 7. 设计必须强调边界、职责与权衡

- 设计文档至少应覆盖模块边界、职责划分、接口契约、数据流与控制流、约束、非功能需求与测试策略。
- 设计定稿前至少比较两个可行方案，并给出取舍理由。
- 非功能需求、技术约束和集成边界必须进入设计决策，不能留到实现时再碰运气。
- 设计的目标是消除后续实现中的主要猜测空间，而不是提前写实现细节。

## 8. 任务拆解以“可独立验证”作为最小粒度

- 任务必须足够小，能够被明确执行、明确验证、明确关闭。
- 优先按 TDD 节奏拆任务：失败测试、确认失败、最小实现、确认通过、重构、状态回写。
- 每个任务都需要前置条件、依赖关系、验证方式和完成定义。
- 任一时刻只允许一个 `Current Active Task`，避免实现阶段并行漂移。

## 9. TDD 是实现流内部的硬约束

- TDD 不是实现后的补救动作，而是 `mdc-test-driven-dev` 与 `mdc-hotfix` 的内建路径。
- 没有失败测试，不写生产代码；若先写了实现，应删除并回到 TDD 起点。
- 在进入 TDD 之前，应先给出测试用例设计，并取得真人确认。
- 除记录在案的明确例外外，不得因为“任务很小”“修复很急”就绕过 TDD。

## 10. 质量链和完成门禁都要求 fresh evidence

- 实现完成后，应按既定顺序进入缺陷模式排查、测试评审、代码评审、追溯评审、回归门禁和完成门禁。
- 评审结论不能替代实际验证；“看起来没问题”也不能替代命令输出。
- 任何“已完成”“已修复”“测试通过”的结论，都必须由当前代码状态上的最新证据支撑。
- `mdc-completion-gate` 的职责是批准或拒绝完成结论，并给出唯一下一步，而不是做形式化收尾。

## 11. 变更与热修复是受控支线，不是捷径

- 需求变更必须先做影响分析，再决定回写规格、设计、任务、验证策略与状态记录。
- 热修复必须先复现，再做最小安全修复，再重新验证并回到后续门禁。
- 支线可以压缩路径，但不能破坏主链的可追溯性和证据闭环。
- 任何变更或热修复完成后，都应留下同步记录，说明影响范围、已更新工件和唯一下一步。

## 12. Skill 设计本身也应保持轻量和可维护

- 单个 `SKILL.md` 应聚焦触发条件、硬门禁、工作流、交接与完成条件，不堆积大段样例。
- 大型检查表、记录模板、参考材料优先下沉到 `references/` 或模板文件。
- skill 的描述要明确“何时使用”，避免把完整流程塞进 `description`。
- 项目相关的路径映射、审批别名、团队规范应放在 `AGENTS.md`，不要在多个 skill 中重复硬编码。
- skill之间不要相互依赖，skill之间的执行顺序通过外部调度

## 13. 新增或重构 skill 时的自检问题

- 这个 skill 是否只承担一个清晰节点职责？
- 它是否明确声明了前置条件、禁止跳过的门禁和完成条件？
- 它是否依赖工件证据，而不是依赖会话记忆？
- 它是否给出了唯一下一步或明确的回退方向？
- 它是否和现有 profile、主链 / 支线、评审 / 门禁顺序保持一致？

如果以上任一问题无法清晰回答，通常说明这个 skill 仍然混入了过多职责，或者破坏了 `mdc-workflow` 的核心约束。
