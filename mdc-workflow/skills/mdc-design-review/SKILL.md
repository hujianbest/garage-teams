# MDC 设计评审

评审设计文档，并判断它是否可以成为任务规划阶段的已批准输入。

在完整 MDC workflow 中，它通常用于任务规划前的设计质量判断。

## 硬性门禁

在设计通过评审并经过真人确认之前，不得进入 `mdc-tasks`。

## 目的

这个 skill 用来防止过早拆解任务。

如果设计不清晰，任务规划就会变成猜测，或者过早下沉到实现细节。

## 评审范围

请基于以下内容评审设计：

- 已批准的需求规格
- 主要约束与非功能需求
- 项目现有技术上下文（如果已知）

评审时不要开始写任务或代码。

## 真人确认规则

`mdc-design-review` 给出的“通过”，只表示设计已经达到可确认状态，不等于自动批准。

只有结论为 `通过` 时，才进入真人确认。

如果结论为 `需修改` 或 `阻塞`，不要发起真人批准确认；应基于评审结果直接回到 `mdc-design`，与用户确认必要修订后继续完善设计。

在进入 `mdc-tasks` 之前，必须完成以下动作：

1. 向真人展示评审结论、主要设计风险和需要重点确认的决策
2. 邀请真人确认“该设计是否批准进入任务规划阶段”
3. 如果真人提出修改意见，继续和真人对话，回到设计内容上迭代修订
4. 只有在真人明确确认通过后，设计才算已批准

## 记录要求

如果当前项目需要留痕，评审完成后，默认将本次结论写入：

- `docs/reviews/design-review-<topic>.md`

如 `AGENTS.md` 为当前项目声明了等价路径，按其中映射路径保存。

若项目尚未形成固定 review 记录格式，默认使用：

- `skills/mdc-workflow/templates/review-record-template.md`

如果当前任务正在使用 `task-progress.md` 驱动 workflow，且真人确认通过，还应同步更新：

- 设计文档中的状态字段
- `task-progress.md` 中的当前阶段与推荐下一步动作或 skill

若项目尚未形成固定进度记录格式，默认使用：

- `skills/mdc-workflow/templates/task-progress-template.md`

## 与入口 skill 的结构化衔接

本 skill 的结论会被 `mdc-workflow-starter` 用来恢复编排。

因此除人类可读的评审结论外，还应显式交接：

- `nextNode`：`设计真人确认` 或 `mdc-design`
- `blockingReasons`：为什么当前不能进入下游
- `requiredReads`：本轮评审实际依赖了哪些设计、规格和状态证据
- `expectedWrites`：design review 记录、设计状态、`task-progress.md` 等写入目标
- `writesScope`：默认是当前 `change workspace`；真人确认完成并收口后，设计最终版才会提升为 baseline

## 检查清单

### 1. 需求覆盖

- 设计是否覆盖了规格中的关键需求？
- 主要行为是否映射到了组件、模块或接口？

### 2. 架构一致性

- 职责和边界是否清晰？
- 选定方案是否被解释清楚，而不仅仅是提了名字？
- 模块之间的交互是否容易理解？

### 3. 约束适配

- 设计是否体现了已声明约束？
- 是否考虑了非功能需求和集成点？

### 4. 接口准备度

- 关键契约是否足够明确，可以支撑任务规划？
- 主要数据流和控制流是否清晰？

### 5. 测试准备度

- 设计是否包含设计层面的测试策略？
- 该设计是否可测试，而不依赖隐藏假设？

## 输出格式

请严格使用以下结构：

```markdown
## 结论

通过 | 需修改 | 阻塞

## 发现项

- [严重级别] 问题

## 薄弱或缺失的设计点

- 条目

## 下一步

`等待真人确认` | `回到设计修订`

## 阻塞原因

- 无 | 阻塞原因

## 记录位置

- `docs/reviews/design-review-<topic>.md` 或映射路径

## 结构化交接

- currentWorkspace: <当前 workspace>
- nextNode: `设计真人确认` | `mdc-design`
- blockingReasons: 无 | <阻塞原因>
- requiredReads: <本轮实际读取的最少证据>
- expectedWrites: <design review 路径>, <设计状态更新>, `task-progress.md`
- writesScope: `change workspace`

## 交接说明

- `等待真人确认`：仅当结论为 `通过`
- `回到设计修订`：仅当结论为 `需修改` 或 `阻塞`，并携带评审修订清单继续修订
```

## 判定规则

只有在以下条件全部满足时，才返回 `通过`：

- 可以追溯到需求规格
- 清楚说明了连贯的实现方案
- 边界和接口足够清晰，可以进入任务规划
- 覆盖了主要约束和测试影响

以下情况返回 `需修改`：

- 核心设计可用，但还不完整
- 某些模块、流程或接口仍然描述不足
- 测试方案偏弱，但可修补

以下情况返回 `阻塞`：

- 设计无法清晰支撑需求规格
- 关键架构决策仍然缺失
- 存在矛盾或未解决的技术风险，无法继续规划

## 批准处理

如果设计通过，请明确说明它已经可以提交给真人确认。

只有在真人明确确认后，它才能成为任务规划阶段的已批准输入。

真人确认完成后，请显式同步：

- 设计文档状态为已批准
- `task-progress.md` 的 Current Stage
- `task-progress.md` 中记录的推荐下一步动作或 skill
- review 记录中的 Human Confirmation 与 Evidence Location

如果未通过，请把它退回设计修订，并附上具体修订点。

退回时请显式做到：

- 不等待真人批准确认
- 把发现项转换为用户可确认的修订行动
- 让下一轮设计修订直接基于这些修订点继续，而不是从零重新推导完整方案

## 反模式

- 因为“实现时再说”就直接通过
- 把设计评审变成底层编码建议
- 接受只是复述需求、没有设计决策的文档
- 忽略缺失的接口或模块边界
- 设计评审刚返回“通过”就直接进入 `mdc-tasks`
- 不和真人确认，就把设计当成已批准输入

## 完成条件

只有在给出明确结论、支持真人确认或退回修订的说明，并且选定唯一下一步之后，这个 skill 才算完成。
