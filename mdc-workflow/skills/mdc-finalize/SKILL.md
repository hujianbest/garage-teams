# MDC 收尾

把当前 MDC 工作周期干净地收尾。

## 目的

这个 skill 的目标是把“已经完成的工作”沉淀为“可持续的项目状态”。

它的存在是为了让下一次会话不必重新摸索：

- 做了什么
- 有哪些证据支持完成
- 对用户来说有哪些变化
- 接下来应该做什么

## 前置条件

只有在当前任务通过完成门禁之后，才能使用这个 skill。

默认应结合以下状态工件一起完成收尾：

- `task-progress.md`
- `RELEASE_NOTES.md`
- `docs/reviews/`
- `docs/verification/`

## Artifact 收口原则

`mdc-finalize` 不只是“补记录”，还负责把当前 `change workspace` 的结果收口成两类产物：

- 回写到 `baseline artifacts` 的长期事实
- 进入 `archive` 的闭环留痕

在执行收尾前，先明确：

- 当前完成的是哪个 `change workspace`
- 哪些规格、设计或稳定规范已经因为本次 change 变成新的 baseline
- 哪些 review、verification、发布与收尾记录应保留在 archive 中

如果无法回答这些问题，就说明收尾仍不完整。

## 红旗信号

出现以下想法时，先停下。这通常说明你正在为“跳过收尾”找理由：

| 想法 | 实际要求 |
|---|---|
| “已经做完了，不用再写记录” | 收尾的目标就是把完成状态沉淀为可继续的项目状态。 |
| “这些证据我自己知道在哪，不用写出来” | 证据位置必须显式记录，便于后续追溯。 |
| “下一步别人自己看代码就知道了” | 交接信息必须明确说明下一步和剩余工作。 |
| “发布说明以后再补也没关系” | 只要影响用户可见行为，就应在收尾时同步记录。 |
| “顺手再改一点实现再一起收尾” | 收尾阶段不再混入新的实现工作。 |
| “状态先不更新，后面想起来再说” | 进度状态、同步状态和交接信息必须在当前收尾中完成。 |

## 工作流

### 1. 更新进度状态

在项目进度记录中补充：

- 已完成任务或范围项
- 日期或会话标记
- 当前状态
- 推荐的下一任务
- 当前阶段
- 下一步动作或推荐 skill
- 已批准工件摘要

### 2. 决定 baseline 回写与 archive 留痕

明确区分：

- 哪些结果要成为新的长期事实，回写到 `baseline artifacts`
- 哪些结果只作为本次 change 的闭环证据，进入 `archive`

典型判断：

- 已批准且会影响后续工作的 spec / design 最终版本，应回写为 baseline
- review、verification、finalization summary、当次发布沟通，通常进入 archive

### 3. 更新发布说明

如果完成的工作影响了用户可见行为，请在发布说明中简要记录：

- 变了什么
- 为什么重要

### 4. 记录证据位置

记录支持完成结论的证据在哪里：

- 缺陷模式排查结果（如有）
- 测试执行结果
- 评审结论
- 追溯性评审结果
- 回归结果
- 完成门禁结果

记录可以简洁，但必须支持后续审计和追溯。

如果项目已经将 review / verification 结果落盘，应优先记录这些文件路径，而不是只写口头摘要。

### 5. 准备交接

明确说明：

- 哪些内容已经完成
- 还剩什么
- 哪些受影响工件已经同步刷新
- 如果下一次会话继续，推荐的下一步动作或 skill 是什么

### 6. 维护状态收口

确保以下内容已同步：

- `task-progress.md` 中的 Current Stage
- `task-progress.md` 中的 Current Active Task
- `task-progress.md` 中的 Next Action Or Recommended Skill
- `task-progress.md` 中的 Session Log
- 与本轮相关的证据索引

若当前主链任务全部完成，应在状态记录中明确标记工作流完成，而不是只在对话中说“结束了”。

## 输出格式

请严格使用以下结构：

```markdown
## 已完成工作

- 已完成项

## 已更新记录

- 已更新记录项

## Baseline 回写

- 已回写的 baseline artifacts

## Archive 记录

- 已归档的记录
- archive 位置

## 证据摘要

- 证据项

## 证据位置

- 路径

## 同步状态

- 已同步工件

## 下一步

`进入下一任务实现` | `进入变更处理` | `进入热修复处理` | `工作流完成`
```

## 反模式

- 宣告完成，却没有更新项目状态
- 把下一任务留成隐含信息
- 对用户可见变化忘记写发布说明
- 明明发生过变更或热修复，却不确认受影响工件是否已同步
- 在收尾过程中混入新的实现工作
- 没有记录证据位置，就直接结束当前会话
- 没有更新 `task-progress.md` 中的阶段和下一步，却声称已经完成收尾

## 完成条件

只有在项目状态、发布沟通和交接信息都更新到足以让下一次会话无须猜测地继续时，这个 skill 才算完成。
