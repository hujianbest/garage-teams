# MDC 测试评审

评审当前任务的测试。

在完整 MDC workflow 中，它通常用于正式评审链中的测试质量判断。

## 目的

这个 skill 用于判断测试是否真的在验证行为，而不只是给任务表面上加了一层浅覆盖。

## 适用时机

优先用于以下场景：

- 需要判断当前测试是否足以支撑对实现结果的信任
- 当前任务已经完成一轮最小实现并补充了测试
- 准备从缺陷模式排查进入正式评审链
- 需要判断现有测试是否足以支撑后续代码评审和完成判定

## 输入

阅读以下最少必要信息：

- 当前任务新增或变更的测试
- 当前实现与测试目标
- 与这些测试相关的需求、设计或任务背景，以及 fail-first 证据（如有）

如果当前上下文无法提供完整规格或设计，至少应提供测试目标、当前实现范围和可用的失败证据或替代说明。

## 记录要求

如果当前项目需要留痕，评审完成后，默认将本次结论写入：

- `docs/reviews/test-review-<task>.md`

如 `AGENTS.md` 为当前项目声明了等价路径，按其中映射路径保存。

若项目尚未形成固定 review 记录格式，默认使用：

- `skills/mdc-workflow/templates/review-record-template.md`

如果当前任务正在使用 `task-progress.md` 驱动 workflow，应同步更新：

- `task-progress.md` 中的测试评审状态
- 当前任务是否需要回到实现修订，还是可以进入后续质量检查
- 当前阻塞原因或待补测试缺口（如有）

若项目尚未形成固定进度记录格式，默认使用：

- `skills/mdc-workflow/templates/task-progress-template.md`

## 检查清单

### 1. 失败优先纪律

- 在实现前，是否先出现了失败测试？
- 这个失败是否准确代表了缺失行为？

### 2. 行为价值

- 测试验证的是行为，而不是实现细节吗？
- 断言是否有意义？

### 3. 覆盖形态

- 在适用场景下，是否至少覆盖了基本异常路径或边界路径？
- 重要任务行为是否都被体现出来？

### 4. 测试质量风险

- 过度使用 mock
- 断言过弱
- 命名模糊
- 测试与实现细节耦合过紧

## 输出格式

请严格使用以下结构：

```markdown
## 结论

通过 | 需修改 | 阻塞

## 发现项

- [严重级别] 问题

## 测试质量缺口

- 条目

## 下一步

`进入实现质量或后续验证` | `回到实现修订` | `补齐阻塞条件后重试当前评审`

## 记录位置

- `docs/reviews/test-review-<task>.md` 或映射路径
```

## 判定规则

只有当这些测试对当前任务真正有验证价值时，才返回 `通过`。

如果测试质量太弱，尚不足以支撑对任务结果的信任，则返回 `需修改`。

如果缺少关键测试上下文、无法判断 fail-first 证据，或当前测试资产本身不可读取，则返回 `阻塞`。

## 反模式

- 因为“有测试”就直接通过
- 忽略没有失败优先信号的问题
- 把浅层断言当成足够证据

## 完成条件

评审结论可以输出在当前对话、评审评论或临时评审记录中，但仍必须给出明确结论、测试质量缺口和唯一下一步。

只有在给出明确结论、记录位置（或等价输出位置）和唯一下一步之后，这个 skill 才算完成。
