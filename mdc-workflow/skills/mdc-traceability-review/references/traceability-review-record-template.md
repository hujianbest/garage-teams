# MDC 可追溯性评审记录模板

这个模板用于记录一次 `mdc-traceability-review` 的评审结果。

目标不是做厚重的 RTM 表格，而是以最小但清晰的方式确认：

- 需求和设计是否仍然被实现正确承接
- 任务与实现是否仍然对得上
- 测试与验证证据是否支持当前结论
- 是否存在未记录漂移、断链或需要回写的工件

## 使用原则

- 只记录当前评审范围内的关键链路，不追求把所有细枝末节都写全。
- 重点关注“被声称完成的内容”是否真的能一路追到规格、设计、任务、实现和验证。
- 对小改动可保持精简；对接口变更、用户可见行为、复杂逻辑和热修复应更严格。
- 如果发现偏离，不要只记“有问题”，要明确缺口、风险和回写动作。

## 建议使用时机

- 代码评审通过后、进入回归门禁前
- 热修复后，确认修复没有与现有规格/设计脱节
- 增量变更后，确认需求、设计、任务和实现链路仍完整
- 收尾前，做一次轻量收口检查

## 评审提示

做记录时，优先回答以下问题：

- 当前实现真的满足被批准的需求和设计吗？
- 本次实现是否超出了当前任务原本范围？
- 新增或变更测试是否覆盖了被声称完成的行为？
- 有没有“代码已经变了，但文档、任务或验证记录没跟上”的情况？
- 是否出现 undocumented behavior、orphan code、无测试支撑的结论？

## 常见问题类型

可以重点检查以下几类断链：

- 规格已改，但设计未更新
- 设计已改，但任务计划未更新
- 任务范围扩大，但未记录
- 代码实现已偏离原设计，但没有回写说明
- 测试只验证了局部行为，不足以支撑“完成”结论
- 热修复改动改变了行为，但发布说明和状态记录未更新

## 示例骨架

```markdown
## 评审范围

- topic / 任务：用户登录失败重试限制
- 相关需求：登录失败超过 5 次后短时锁定
- 相关设计：认证模块重试计数与锁定策略
- 相关任务：增加失败计数、锁定判断、提示信息、测试覆盖
- 相关实现：AuthService / LoginHandler
- 相关测试 / 验证：单元测试、接口测试、失败场景验证

## 结论

需修改

## 链路检查

### 1. 规格 -> 设计

- 是否对齐：是
- 说明：设计覆盖了锁定策略与恢复条件。

### 2. 设计 -> 任务

- 是否对齐：部分对齐
- 说明：任务中未显式写出恢复时间窗口验证。

### 3. 任务 -> 实现

- 是否对齐：部分对齐
- 说明：实现包含锁定逻辑，但还加入了额外告警上报，任务中未记录。

### 4. 实现 -> 测试 / 验证

- 是否对齐：否
- 说明：测试覆盖了锁定触发，但未覆盖恢复窗口和重复失败边界。

## 追溯缺口

- 任务计划未显式覆盖恢复窗口验证
- 告警上报属于额外行为，未记录到任务或设计

## 漂移风险

- 当前“已完成”结论缺少对恢复窗口行为的测试支持
- 额外告警行为可能造成设计与实现不一致

## 需要回写或同步的工件

- 工件：任务计划
  - 原因：缺少恢复窗口验证项
  - 建议动作：补任务与 DoD
- 工件：设计文档
  - 原因：新增告警行为未记录
  - 建议动作：明确是否纳入正式设计

## 下一步

`mdc-test-driven-dev`
```
