# 独立版 skill 质量审查评测

这个目录包含独立版 `skill-reviewer` 的评测 prompts。

## 目的

这些评测用于验证该 skill 是否真正做到：

- 区分单文件审查与整包审查，不在输入不足时假装完成 full-package review
- 审查 skill 的触发质量、正文边界、包结构、评测准备度和共享准备度
- 判断 `evals` 是否真的在验证正文规则，而不是奖励错误行为
- 区分“可继续迭代”和“适合共享或发布”
- 保持 audit-only 边界，不在 review 过程中直接重写 skill
- 在用户施加“直接改写”压力时，仍先守住 review-first 结论
- 按 skill 类型判断 `README`、`references`、`scripts` 是否必要，而不是机械扣分
- 区分“文档看起来完整”和“已经有实证验证证据”这两种成熟度

## 建议评分关注点

1. 是否明确写出本轮已审范围与未评估项。
2. 是否把 `description` 当作触发字段来审，而不是只点评文风。
3. 是否能发现 `README`、`SKILL.md`、`evals` 之间的对齐问题。
4. 是否会检查命名、frontmatter 和渐进披露这类基础 skill 结构要求。
5. 是否区分核心质量问题与共享或发布准备度问题。
6. 是否在没有 `scripts/` 或 `references/` 时先判断必要性，而不是自动判缺陷。
7. 是否保持 audit-only，不把审查输出偷偷变成重写稿。
8. 在用户要求“直接改好”时，是否仍然先做正式审查。
9. 是否区分“有 evals 设计”和“真的有验证证据”。
10. 是否能抓到会被 `quick_validate` 这类脚本直接判 invalid 的结构问题。
11. 是否能识别正向验证证据，而不只是会说“缺少证据”。
