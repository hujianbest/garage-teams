# 个人博客写作

从主题到成稿，按用户个人风格撰写技术博客或思考随笔，输出 Markdown 文件到 Garage 仓库。

## When to Use

**适用：**
- 写技术博客（编程心得、工具评测、架构思考）
- 写思考随笔（行业观察、个人成长、方法论总结）
- 续写已有博客系列（如"写代码的一点心得"系列）
- 用户给出主题/素材/链接要求写博客

**不适用 → reroute：**
- 公众号长文（卡兹克风格）→ `khazix-writer`
- 深度研究报告（PDF）→ `hv-analysis`
- 短内容（小红书、推特、朋友圈）→ 通用能力即可
- 纯调研不含写作 → `web` 工具直接搜索

**与相邻 skill 的区别：**
- `khazix-writer`：固定卡兹克文风，公众号格式，无 frontmatter
- `hv-analysis`：横纵双轴研究，输出 PDF 报告，万字起步
- 本 skill：用户个人风格，Markdown + frontmatter，技术+思考均可

## Workflow

### 1. 确定主题

用户的典型输入是**灵感驱动、碎片化的**——讲了一段经历或见闻，说了想写什么，但未必有完整结构。这一步的核心工作是：把碎片整理成清晰的写作意图。

**先接收，再补全：**

用户可能给的东西：
- 触发灵感的背景故事（"最近在搞XX，发现一个有意思的事"）
- 想写的方向或观点（"我觉得大家都在忽略一个问题"）
- 散乱的要点或情绪（"就是觉得不对劲，但又说不清楚"）
- 一个链接、一段对话、一个踩坑经历

**需要帮用户理清的问题：**

如果用户已经说得足够清楚，不要机械追问，直接进入下一步。如果信息有缺口，补问以下关键项（不要一次全问，挑最缺的问）：
- **核心观点**：用一句话说，这篇文章想表达什么？
- **个人经历**：有没有自己的故事/案例可以作为主轴？
- **对立面**：有没有不同的声音或做法？这个对比能让文章更立体
- **系列关系**：是否属于已有系列？需要衔接前文吗？
- **目标读者**：写给谁看的？（默认：有编程经验的开发者）

**输出**：把用户的碎片输入整理成一句清晰的主题陈述，确认后再进入调研。

### 2. 调研素材

如果主题需要外部素材：
- 用 `web` 搜索收集引用、数据、案例
- 用 `browser` 抓取用户提供的链接内容
- 将调研结果整理为要点，不要直接拼凑进文章

**调研方法（WSL 环境）：**
- Brave Search API 需要 API key（可能未配置）
- DuckDuckGo HTML 版可作为搜索替代：`curl "https://html.duckduckgo.com/html/?q=..." `
- DDG 结果解析：`class="result__a"` 取链接，`class="result__snippet"` 取摘要，URL 需从 `uddg=` 参数解码
- 抓取文章内容用 curl + 去 HTML 标签，但很多网站有 JS 反爬（OpenAI、Medium、InfoQ 等）
- 网络需要代理：`--proxy "http://127.0.0.1:9567"`（WSL 镜像模式，Windows 代理）
- 调研结果整理为结构化摘要供写作参考，不要把原始搜索结果直接堆给用户

### 3. 写大纲

根据内容类型选择文章原型：

| 原型 | 适合 | 结构重心 |
|------|------|----------|
| 经验分享型 | 编程心得、踩坑总结 | 个人经历→提炼规律→给建议 |
| 方法论型 | 实践方法、工作流 | 理论→案例→可执行步骤 |
| 观点论述型 | 行业观察、技术判断 | 现象→分析→立场 |
| 系列续写型 | 延续已有系列 | 承接前文→新维度→预告后文 |

确定章节结构（3-5 个大章节，带编号子节），输出大纲给用户确认后再写。

### 4. 初稿

严格遵循用户的**写作风格**（见下方 Style Guide）。

关键要求：
- **从个人经验出发**：用第一人称"我"，从真实经历切入
- **每个观点配案例**：抽象概念必须有具体场景支撑
- **中英自然混用**：技术术语保留英文，括号补充说明
- **加粗关键原则**：每篇 20-30 处粗体标记核心概念
- **引用权威**：适当引用书籍/名人/论文，但自然融入
- **联系 AI 时代**：适当关联当下 AI 辅助开发的实践

### 5. 润色

对初稿执行三级检查并修改：

**L1 硬性规则：**
- 无 AI 套话（"综上所述"、"值得注意的是"、"不难发现"、"首先…其次…最后"）
- 无空洞形容词堆砌（"赋能"、"抓手"、"打造闭环"）
- 无教科书式开头（"在当今…的时代"、"随着…的发展"）
- 每个提到的工具有具体名称，非空泛表述

**L2 风格一致性：**
- 第一人称叙事，有"过来人"语气
- 有自嘲/谦逊表达
- 有隐喻/类比（至少 3 处）
- 有对仗/对比结构（至少 2 处）
- 段落有节奏：长短交替，关键句独立成段
- 每个大章节结尾有格言式总结
- **加粗精简**：初稿往往加粗过多（100+处）。精简到 20-30 处，只保留格言式洞察和核心原则。去掉：结构标签（"第一代/第二代"）、术语名（"Harness Engineering"）、列表头（"分解问题/判断质量"）、描述性语句。用 `sed` 脚本批量去加粗比 patch 逐个操作更可靠

**加粗精简的技术陷阱：**
- `read_file` 有缓存机制——如果文件被 `write_file` 写坏后再 `git checkout --` 恢复，`read_file` 可能仍返回缓存的坏内容。解法：用带 `offset` 参数的调用强制重新读取，或直接用 `terminal("grep ...")` 操作
- 用 `execute_code` + `write_file` 做批量文本替换风险极高（一次替换错误可能清空所有加粗）。**优先用 `sed -i` 脚本批量处理**，出错后 `git checkout --` 即可恢复

**L3 内容质量：**
- 每个核心观点有案例支撑
- 引用自然融入，不像"下面我来科普"
- 理论→实践→哲学的递进感
- 二元对立/多角度分析，不是单向输出

**L4 Humanizer-zh 去AI味检查（推荐）：**

L1-L3 通过后，用 `humanizer-zh` skill 对全文做一轮扫描，重点关注：

- **"不是…而是"对仗过量**：全文控制在 5-6 处以内，只保留格言级的核心对比。超出的改写为其他表达（"谈不上什么X，就是Y"、"X不行，Y也不行，得Z"、"先A再B"等）
- **AI高频词**："洞察"→"发现"、"格局"→换具体词、"此外"→删掉或改连接
- **破折号密度**：一行内不超过 2 个——
- **三段式列举**：连续三个顿号分隔的项目可以保留（中文正常用法），但"X、Y和Z"式的刻意三件套要打破
- **扫描方法**：用 `execute_code` 跑正则扫描，按类别统计，报告给用户后批量修改

### 5.5 评分报告修订（可选）

如果通过子代理或外部评审拿到了量化评分报告（如 humanizer-zh 的 5 维度评分），按以下流程修订：

**1. 创建优先级 TODO：**
- 🔴 高优先：结构性问题（章节合并、核心段落重写、关键句式修改）
- 🟡 中优先：风格密度调整（某类修辞过多、类比降频、填充句删除）
- 🟢 低优先：小幅度精简（bullet list 压缩、去重观点、措辞微调）

**2. 批量指标修订用 `execute_code`，不用逐个 patch：**
- 破折号/类比/句式等"数量指标"类修改，先统计全文现状，再用 Python 脚本批量替换
- **先激进减到目标以下，再手动恢复关键保留项**——比逐个判断"这个要不要删"快得多
- 替换后立即验证数量，如果低于目标就恢复部分

**3. 结构性合并直接用 patch：**
- 小节合并（如 2.4→2.3）：把被合并节的内容压缩为 1-2 段，拼接到目标节末尾，删除原节标题
- 观点去重：保留首次出现的完整表达，后续重复处只保留结论句

**4. 每轮修订后统计关键指标：**
```bash
# 字数
python3 -c "import re; text=open('FILE').read(); cn=len(re.findall(r'[\u4e00-\u9fff]',text)); en=len(re.findall(r'[a-zA-Z]+',text)); print(f'~{cn+en}字')"
# 破折号
grep -o '——' FILE | wc -l
# "不是…而是"
grep -o '不是.*而是' FILE | wc -l
```

**经验教训：**
- 破折号密度：6000-8000字文章建议 25-35 个（约每 250 字一个），太少会失去节奏感
- 摄影类比/隐喻：全文保留 3-5 处核心类比即可，过多会让主题模糊
- 填充悬念句（"这个问题我想了很久"、"有一天突然想通了"、"说了这么多"、"洋洋洒洒写了这么多"）：全部删除，直接进入内容
- 评分报告的"精炼度"维度通常暗示字数可砍 20-30%，优先砍重复观点和过度展开的类比

**V3→V4 迭代的关键教训（精炼度仍是最大短板时）：**

当破折号和结构已优化但精炼度分数仍没提升时，问题出在**语义层面的重复**而非表面修辞。这类修改更大胆：

1. **整段删除而非修修补补**：如果一个小节的核心观点已在其他地方表达过（如"不是凭感觉拼的"已在全文主旨中隐含），直接删整个小节，不要压缩重写
2. **小节合并**：§4.4"跟着你一起长"只有一句金句有价值→删除小节标题，把金句附到4.3末尾
3. **总结段只保留增量信息**：总结段的 bullet list（回顾系列）是有信息量的，但总结段后面的"回到焦虑→重述论点→摄影类比收尾"几乎都是前面说过的。砍掉后半截，只留一句收束
4. **核心论点去重要狠**：同一个论点（如"约束很重要"、"方法论没过时"）出现4-6次是正常的写作节奏，但精炼度评分会因此扣分。保留首次提出+数据支撑+落地实践三处，其余全删
5. **批量修改用 execute_code 而非逐个 patch**：当修改项超过8条时，用 Python 脚本一次性替换所有内容，避免行号偏移和上下文不匹配的问题
6. **替换后必须全文通读**：批量替换容易产生重复段落（如"我做的不是发明新轮子"出现两次）、断裂的过渡、或丢失的章节编号重排。替换后从头到尾读一遍

**V1→V4 全流程迭代经验（第5篇博客实战）：**

| 版本 | 分数 | 核心改动 | 教训 |
|------|------|----------|------|
| V1 初稿 | — | 5800字 | 初稿不必追求完美，先出骨架 |
| V2 | 37 | 扩展到7600字，丰富案例 | 字数膨胀不等于质量提升 |
| V3 | 36 | 破折号73→29，章节合并 | 表面修辞优化对精炼度帮助有限 |
| V4 定稿 | 35 | 删§5.1整段、删§4.4整段，字数7200→6700 | 整段删除比修修补补更有效，但精炼度仍卡在6 |
| V5 融合 | 33 | 借鉴卡兹克版亮点融合进V4 | 风格融合不解决精炼度，反而引入新问题 |
| V5+对齐 | — | 骑马→摄影器材、成就拆散、术语中文化 | 风格一致性验证是定稿前必做步骤 |

**核心结论：**
- **精炼度是最难突破的维度**：V2→V4 三轮迭代都没突破6分。根因是核心论点以不同变体重复出现（8-10次），需要结构级合并才能解决，但改动太大
- **用户判断"不改了"就停**：当迭代收益递减（每轮-1分）且改动越来越大时，用户选择V4定稿是合理的决策。不要无限迭代
- **评分用子代理做**：delegate_task + glm-5.1，传入5维度×10分框架，效果稳定
- **read_file 缓存陷阱**：`read_file` 返回带行号的内容，用 Python `open()` 直接读才不会匹配失败。文件被写坏后 `git checkout --` 恢复，但 `read_file` 可能返回缓存旧内容

### 6. 生成 frontmatter

```yaml
---
title: "文章标题"
date: YYYY-MM-DD
tags: [tag1, tag2]
series: "系列名称"  # 如果属于系列
series_index: N     # 系列中的第几篇
draft: true         # 初稿默认 draft
---
```

### 7. 写入文件

将 frontmatter + 正文写入 Garage 仓库的 `.garage/blog/` 目录。

文件命名：`YYYY-MM-DD-标题slug.md`

路径示例：`/mnt/e/workspace/Garage/.garage/blog/2026-04-17-ai-coding-workflow.md`

如果目录不存在，先创建。

**写完后把文件发给用户审阅**（飞书发文件）：
1. 读取 `~/.hermes/.env` 获取 `FEISHU_APP_ID` 和 `FEISHU_APP_SECRET`
2. 调用飞书 API 获取 tenant_access_token：`POST https://open.feishu.cn/open-apis/auth/v3/tenant_access_token/internal`
3. 上传文件：`POST https://open.feishu.cn/open-apis/im/v1/files`（`file_type=stream`）
4. 发送文件消息：`POST https://open.feishu.cn/open-apis/im/v1/messages?receive_id_type=chat_id`（`msg_type=file`）
5. 飞书 API 不需要代理，国内直连

## Output Contract

- **写什么**：完整 Markdown 文件（frontmatter + 正文）
- **写哪**：`/mnt/e/workspace/Garage/.garage/blog/` 目录
- **命名**：`YYYY-MM-DD-标题slug.md`

## 风格融合写法（Style Fusion）

当用户想借鉴另一种写作风格（如 khazix-writer）的亮点来改进自己的定稿时，用「风格融合」而非整体重写。

用户喜欢用不同写作 skill 生成同一篇文章的多个版本进行对比。同一主题不同风格的文件用后缀区分（如 `-khazix`），放在同一目录。

**适用场景：**
- 用户说「借鉴卡兹克版的亮点融合进 V4」
- 用户想对比两种风格后取长补短
- 已有定稿（V4），另有风格对比版（如 khazix版），想融合但不破坏原文

**操作步骤：**

1. **读取两个版本**，识别对比版的亮点（通常是前 100-200 行最好）
2. **提炼可借鉴元素清单**（具体到写法技巧，不是抽象评价）：
   - 情绪标点（。。。/？？？/= =）及其克制使用频率
   - 短句独立成段的断裂式节奏
   - 自嘲口吻的具体句式
   - 「」引号 vs ""引号的使用效果
   - 段落呼吸感的制造方式
3. **以定稿为主干**，逐章将借鉴元素融入（不是重写，是点缀）
4. **写入独立文件**（如 `-v5.md`），不覆盖原定稿
5. **总结列明**：借鉴了什么、保留了什么、文件在哪

**关键约束：**
- 保留定稿的章节结构和技术内容不变
- 借鉴是「点缀」不是「替换」——情绪标点全文控制在 3-5 处
- 避免借鉴版的问题（如公众号腔、预告式过渡、论点重复）
- 字数应与定稿接近，不要因融合而膨胀

**风格融合的实际教训（V5 实验）：**

V5 融合版评分 33/50，比 V4 定稿（35/50）反而低了 2 分。原因：
- V5 在 V4 骨架上贴了卡兹克的脸，但骨架里的水没挤掉反而多了
- 融合引入的亮点（情绪标点、自嘲）本身是好的，但**不解决精炼度问题**
- 精炼度从 V4 的 6 分降到 5 分——因为核心论点重复次数不减反增

结论：**风格融合解决的是「真实性/信任度」维度，不解决「精炼度」。如果定稿的精炼度已经卡在瓶颈，融合不会提升总分，反而可能引入新问题。**

## 风格对比验证（Style Consistency Check）

当博客属于已有系列时，初稿完成后应与前文做风格一致性对比。这是 L2 风格检查的补充步骤，特别适合系列续写。

**方法：** 用子代理抓取前文内容，逐维度对比一致性。

**10 个对比维度：**

| 维度 | 检查要点 |
|------|----------|
| 开头风格 | 是否延续系列惯用开场（自嘲/轶事/他人视角）？ |
| 自嘲谦逊度 | 自嘲是否"同一味道"——不回避失败、带点苦笑？ |
| 成就引用 | 是否"顺带一提"而非"集中检阅"？不要在同一段亮出多个资历 |
| 隐喻类比 | 是否只用自己真正懂的领域做类比？不要用个人经验外的隐喻 |
| 三层递进 | 理论→实践→哲学的结构是否保持？ |
| 格言式总结 | 金句风格是否一致——短句、反转、实用主义底色？ |
| 中英混用 | 英文术语密度是否与前文一致？首次出现中文注释，后续用中文 |
| 章节结构 | 编号+描述性副标题的模式是否一致？ |
| 交叉引用 | "之前聊过的"式回扣是否自然、贯穿全文？ |
| 结尾风格 | 是否延续"一句话留白"而非过度总结？ |

**评分判断标准：**
- ✅ 一致（6/10以上通过）→ 可定稿
- ⚠️ 部分一致（3-5个维度有偏差）→ 按优先级修改
- ❌ 偏离（6个以上维度不一致）→ 需要结构性重写

**最关键的 3 个 AI 痕迹检测点：**
1. **综述式科普段落**：前文从不用数据堆砌论证，如果某段像"文献综述"就是 AI 腔
2. **隐喻领域不匹配**：前文只用作者真实的个人爱好/经验领域做类比（如摄影、建筑），跨领域隐喻是AI选的
3. **结尾过度总结**：前文结尾的精髓是"一句话定格"，信息量过大的结尾是AI腔

**发现偏差后的修改原则：**
- 隐喻替换：换成作者真实经验领域的类比
- 成就拆散：集中出现的资历引用分散到不同段落自然带出
- 术语中文化：首次出现加中文注释，后续只用中文简称
- 结尾一刀收：保留最强金句，删除后续解释
- 用词去重复：同义词替换过度使用的词（如"驾驭"→"用好"、"核心"→换具体词）

**实际验证案例（第5篇 vs 前4篇）：**

对齐修改实例（17处替换，从33分偏差回升）：
- ❌ "AI 是一匹千里马，但你得会骑" → ✅ "AI 是一台顶级相机，但构图得你来"（换回作者真实爱好领域）
- ❌ "被缰绳拴住了" → ✅ "被框在了正确的轨道里"（去掉不属于自己的隐喻）
- ❌ "驾驭 AI 的缰绳" → ✅ "用好 AI 的基本功"（去马术化，换朴素表达）
- ❌ "在华为做了五年工程实践，在 ICT 软件大会上分享过"（一段亮两个资历）→ ✅ 拆散到不同段落，且去掉与论证无关的资历
- ❌ "Spec-Driven Development（SDD）" → ✅ "规格驱动开发（Spec-Driven Development，简称 SDD）"
- ❌ "用 ADR 记录架构决策" → ✅ "用架构决策记录（ADR）来记录架构决策"

## Red Flags

- 写着写着变成论文语气 → 回到对话式第一人称
- 案例不够硬开始编造 → 标注"此处需用户补充真实案例"
- 全是观点没有案例 → 每个观点必须配具体场景
- 段落越来越长像教科书 → 拆短，加粗独立成段
- 偏离用户个人风格变成通用 AI 输出 → 重读 Style Guide
- 加粗超过 40 处 → 太多了，精简到只保留格言式核心原则
- 第5章（实践展示）变成框架文档/功能列表 → 回到"方法论的设计故事"视角

## 用户对内容方向的典型反馈模式

- **"不是想讲 X，想讲 Y"**：说明偏离了用户的核心意图。不要试图修补，直接按 Y 重写对应章节
- **"这一版更差"**：说明改动方向完全错了。用 `git checkout --` 回退到上一版，从那个版本重新改，不要在错误基础上继续改
- **"先在之前的版本上改"**：用户明确指示回退基准。立即恢复指定版本再改
- **"标题/小标题还可以改一改，我们先围绕标题改一轮"**：先列所有标题给用户看（带编号），等反馈后再批量改。不要自作主张改完再问

## 写技术框架/方法论的注意事项

当文章涉及介绍自己做的框架、工具或工作流时，**不要列功能清单或分层架构**。那是 README 干的事，不是博客该写的。

**核心原则：讲"在什么方法论指引下设计出来的"和"怎么把经典方法封装进框架"，而不是讲框架本身的约束机制、结构细节。**

**正确做法：讲方法论的设计故事。**
- 在什么经典方法论的指引下设计的？（SDD、TDD、Fagan 审查、PMBOK……）
- 每个阶段选了什么方法？为什么选这个方法？（方法跟着职责走）
- 怎么把方法"装"进框架里的？（封装思路：方法不是操作手册，是决策框架）
- 踩了哪些坑？教训是什么？

**素材来源优先级：**
- README 的方法论表格/分层图 > 设计原则文档（anatomy）> 代码
- README 里有"哪些方法分配给了哪些节点"的完整映射，最适合博客引用
- anatomy 文件偏"怎么写 Skill 的结构约束"，是给 skill 作者看的，不是博客读者要的
- **用 anatomy 文件理解设计哲学，但写博客时讲的是方法论选型和封装思路，不是 anatomy 细节**

**反面模式：**
- ❌ 列六层方法论（意图层、执行层、验证层……），每层一段解释——这是文档，不是文章
- ❌ 讲 Skill 的 anatomy 细节（description 是分类器、骨架结构、evals 机制）——这是给 skill 作者看的
- ❌ "这不是我发明的，是 XX 的智慧"后面直接贴架构图——像在做技术分享 PPT
- ✅ "hf-specify 用了 EARS 句式和 BDD，因为这些是需求工程的方法，不是架构设计的方法"——讲方法选型的逻辑
- ✅ "方法论不是操作手册，是决策框架——你要告诉 AI 的不是步骤，而是在这个节点哪些纪律必须被遵守"——讲封装思路

## Style Guide（用户个人风格）

以下风格特征提取自用户"写代码的一点心得"系列（4 篇），写作时必须贯穿：

### 语气
- **"过来人"的第一人称**：像一个有经验的同事在茶歇时聊天。"说实话，当时拿金码奖的时候，我啥高级算法也不懂"
- **谦逊但不卑微**：频繁自嘲（"买书如山倒，读书如抽丝"、"技术菜鸟"），但观点坚定
- **有感染力**：用情感化表达（"屎山代码"、"数字考古"），让人感到作者的热情

### 结构
- **引言用个人轶事开场**：一段对话、一个回忆、一个意外发现
- **编号章节 + 编号子节**：大章用编号（1、2、3…），子节也带编号（1.1、1.2…），方便用户审阅时按编号指修改
- **标题文字要有态度**：虽然带编号，但标题文字要像聊天时起的绰号，不要像目录
- **每个子节 = 一个具体要点 + 案例支撑**
- **章节结尾格言式总结**：朗朗上口的口号，如"重构不是技术，是一种态度"

### 标题风格（重要）
- **小标题必须是自然的、有态度的**，不能像 AI 生成的结构性标题
- ✅ 好："充了钱，然后呢？"、"等一下……"、"越约束，越自由"、"穿别人的鞋，总不太合脚"、"是AI不行吗"、"熟悉的配方"
- ❌ 差："充值使我更强大？"、"顿悟时刻"、"约束的悖论"、"别人的框架用起来别扭"——这些要么太"标题党"，要么太 AI 腔
- **自检方法**：如果一组小标题读起来像教科书目录，就太死了；如果读起来像朋友圈标题，就对了

### 标题修改工作流
1. 先列所有标题给用户审阅（用编号方便指认），等用户逐条反馈
2. 用户确认后批量修改，然后 commit + push 方便用户在 GitHub 上查看
3. **不要猜标题**——用户对标题的品味很具体，让用户说了算

### 系列内引用
- 提到前文内容时，用随意的"之前聊过/之前提过"，**不要**写"我在第X篇里聊过"——太刻意、太 AI
- 总结部分可以例外，那里逐篇回顾是合理的结构需要

### 节奏
- **短段落为主**（1-3 句），关键句独立成段
- **长短交替**：连续 3 段以上句式长度相近 = 节奏呆板，必须调整
- **空行制造呼吸感**
- **加粗标记关键原则**（每篇 20-30 处）

### 论述手法
- **二元对立**：新手 vs 专家、抽象 vs 实现、理论 vs 实践——用对比推进论述
- **隐喻丰富**：盖房子、城市改造、医生诊断、武术——把软件概念映射到生活
- **引用自然融入**：Martin Fowler、Uncle Bob、Kent Beck、吴军——"聊着聊着想到的"，不是"下面我来引用"
- **从理论到实践再到哲学**：每个主题都经历这三层递进
- **括号旁白**：用 `()` 补充说明或自嘲（"当然，也可能是我太菜了"）

### 语言
- **中英混用**：技术术语保留英文（Big Ball of Mud, Trade-off, Entropy, YAGNI）
- **对仗句式**："时间换空间"、"代码轻，数据重"
- **口语化但不随意**：自然但不失专业感

### 系列特征
- 开头链接系列前文
- 结尾预告后文
- 统一标签（如 #代码心得）
- 前文概念在后文中反复出现并深化
- "做好每一件小事"作为贯穿主题
