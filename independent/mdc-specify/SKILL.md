# MDC 需求澄清与规范起草

把模糊想法逐步澄清成一份可用于后续设计的规范文档。

## 硬性门禁

在规范文档足够清楚之前，不要开始：

- 架构设计
- 技术方案细化
- 任务拆解
- 代码实现

如果用户把接口、表结构、类、框架、重试策略等实现细节混进需求描述，先提炼背后的业务意图，再把这些内容留给设计阶段处理。

## 目标

最终文档必须回答至少这些问题：

- 为什么要做这件事
- 谁会使用它
- 哪些内容在范围内
- 哪些内容明确不做
- 有哪些业务规则、约束和依赖
- 成功长什么样
- 还有哪些开放问题会阻塞后续设计

## 快速工作模式

| 情况 | 当前回合应做什么 | 不要做 |
|---|---|---|
| 需求极度模糊 | 先回放已知信息，暴露阻塞点，只问一个高价值问题 | 不要补全 spec 或验收标准 |
| 需求过大、含多个子系统 | 先给 2-3 种范围切分方式并推荐先写哪一份 | 不要直接写一份大而全的总规范 |
| 用户提到旧草稿但未给正文或路径 | 先索取草稿内容或路径，并说明它仍是草稿输入 | 不要凭空重写一份替代稿 |
| 用户指定路径/模板并要求直接起草 | 严格遵循该路径/模板，必要时用工作假设与开放问题起草 | 不要退回默认路径或默认章节 |
| 功能名已知但细节不足 | 仅当核心业务对象、目标、主要用户、主流程、范围边界已最小稳定时，才可以起草受约束的草稿；仍需保留 `[待确认]`、开放问题和 `仍需继续澄清` | 不要假装已经 `可进入设计` |

优先级规则：

- 只要命中“需求极度模糊”“旧草稿未提供正文/路径”这类阻塞条件，先进入澄清模式
- 只有在最小事实已经稳定时，才允许进入“受约束的草稿”模式

## 工作方式

借鉴 `brainstorming` 的方式推进，但收尾是规范文档而不是设计文档。

### 1. 先理解上下文

先读完成当前需求所必需的最小上下文：

- 用户当前描述
- 现有产品或流程说明
- 与当前范围直接相关的文档
- 用户或项目已经给出的路径、模板、章节要求

如果用户提供了已有草稿，把它当作待完善输入，而不是当作已定稿结论。

### 2. 先澄清，再动笔

像做需求访谈一样推进：

- 一次只问一个主题
- 一次回复里只推进一个问题块
- 优先问会影响范围和验收的关键问题
- 能用选择题时，优先给用户可选项
- 选择题尽量控制在 2-4 个选项
- 不要一口气抛出整张问卷
- 如果当前回合以提问收尾，不要在结尾再附加第二组独立问题

至少澄清这些方面：

1. 目标与用户
2. 功能范围
3. 非目标 / 范围外内容
4. 约束、依赖与外部边界
5. 验收标准
6. 边界情况与负路径

如果需求明显过大，先不要把所有子系统混在同一份规范里。先帮助用户拆出 2-3 种范围切分方式，说明取舍，并推荐本轮先写哪一份。

### 3. 提到旧草稿时，先做 intake 再补稿

如果用户说“在旧草稿上补完”“不要从头来过”：

- 先拿到草稿正文或明确路径，再开始补稿
- 在未看到草稿正文前，只能做 intake/triage：说明它仍属草稿、列出待检查项、索取文档
- 不要因为用户提到草稿，就假装你已经看过它
- 不要凭空写一份替代稿，再声称那是在“现有基础上补完”
- 真正补稿时，优先保留现有结构和已知业务事实，只补缺口并标出阻塞项

### 4. 必要时给出 2-3 种范围方案

当需求有多种合理切法时，先给出 2-3 种方案，再推荐一个：

- 最小可落地范围
- 业务完整范围
- 集成优先或风险优先范围

说明每种方案的取舍，再请用户确认采用哪一种，然后继续写规范。

### 5. 先对齐理解，再起草文档

在真正写文档前，先用简短摘要回放当前理解，至少覆盖：

- 目标
- 用户
- 范围
- 非目标
- 关键约束
- 成功标准

如果这些点还不稳定，就继续澄清，不要急着起草。

如果业务对象、主要用户或主行为仍未知，也不要输出“已完成”的规范草稿。

### 6. 起草规范文档

若用户或项目已经指定文档路径、命名方式或模板，优先遵循。

若没有指定，默认保存到：

- `docs/specs/YYYY-MM-DD-<topic>-spec.md`

若没有现成模板，使用以下最小结构：

```markdown
# <主题> 规范文档

## 背景与目标

## 用户与使用场景

## 范围

## 非目标

## 功能要求

## 非功能要求

## 约束与依赖

## 验收标准

## 边界情况与负路径

## 开放问题
```

写作要求：

- 需求写成可观察行为，不写实现方式
- 非功能要求尽量量化
- 范围外内容必须显式写出
- 开放问题要标清是否阻塞后续设计
- 若用户没给答案，不要擅自补全成既定事实
- 用户明确写出的业务对象、接收者、触发条件，默认按原意保留；未知部分再列为开放问题
- 如果用户指定了路径和章节骨架，规范正文必须严格遵循；补充说明放在正文外，不要在正文里偷偷恢复默认模板
- 如果当前只能起草“受约束的草稿”，要把工作假设、`[待确认]` 和阻塞项写清楚；这种情况下通常仍应给出 `仍需继续澄清`

## 轻量自检

起草完成后，用一轮轻量自检判断文档是否真的可进入设计：

- 目标是否清楚
- 范围和非目标是否都写明
- 核心行为是否可观察、可测试
- 模糊词是否被量化、替换或删除
- 是否混入了接口、表结构、类、框架等设计决策
- 验收标准是否覆盖主流程
- 是否至少识别了一类负路径或边界情况
- 是否还有阻塞设计的开放问题
- 如果这轮只是 intake/triage 或范围切分，是否错误地把它包装成“草稿已完成”

如果仍有阻塞项，不要假装已经准备好进入设计。明确告诉用户还需要补齐什么。

## 输出格式

不是每一轮都应该输出“需求规范草稿已完成”。

### A. 仍在澄清阶段

如果核心业务对象、主要用户、主流程、范围边界，或旧草稿正文/路径仍未拿到，使用：

```markdown
当前仍在需求澄清阶段。

- 已确认:
  - <已知事实>
- 仍待确认:
  - <阻塞项>
- 下一问题:
  - <只保留一个主题的问题>
```

此模式下：

- 不要写 `可进入设计`
- 不要假装规范草稿已完成
- 不要输出 completion block

### B. 草稿已形成

只有在以下条件同时满足时，才使用 completion block：

- 本轮已实际起草、补全或实质更新了一份规范草稿
- 文档路径已明确
- 阻塞项已被识别并写明
- 已完成轻量自检

完成时输出应包含：

```markdown
需求规范草稿已完成。

- 文档路径: `<spec-path>`
- 当前判断: `可进入设计` | `仍需继续澄清`
- 仍待确认:
  - <若无则写“无”>

如需，我可以继续根据这份规范进入设计阶段。
```

只有在阻塞性开放问题已消除，且核心范围具备可验证验收标准时，才能判断为 `可进入设计`。

## 反模式

- 用户刚说一个想法，就直接给架构方案
- 把头脑风暴笔记当成规范文档
- 范围内写得很多，但完全不写非目标
- 验收标准留成隐含理解
- 用 endpoint、table、class、queue、retry policy 之类词汇偷带设计
- 因为“功能很简单”就跳过澄清
- 一次把十几个问题都丢给用户

## 完成条件

如果当前回合仍处在澄清模式，正确停在“当前仍在需求澄清阶段”并等待更多输入，是允许的中间状态。

这不算 skill 已全部完成，但属于正确输出；不要为了“看起来完成了”而伪造占位草稿。

只有在以下条件满足时，这个 skill 才算完成：

- 关键需求已被澄清到足以写成规范
- 规范文档草稿已落盘
- 已完成一轮轻量自检
- 已明确给出 `可进入设计` 或 `仍需继续澄清` 的判断
