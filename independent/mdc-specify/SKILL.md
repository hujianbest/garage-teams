# MDC 需求澄清与规范起草

把模糊想法逐步澄清成一份可用于后续设计的规范文档。

## 硬性门禁

在规范文档足够清楚之前，不要开始：

- 架构设计
- 技术方案细化
- 任务拆解
- 代码实现

如果用户把接口、表结构、类、框架、重试策略等实现细节混进需求描述，先提炼背后的业务意图，再把这些内容留给设计阶段处理。

## 目标

最终文档必须回答至少这些问题：

- 为什么要做这件事
- 谁会使用它
- 哪些内容在范围内
- 哪些内容明确不做
- 有哪些业务规则、约束和依赖
- 成功长什么样
- 还有哪些开放问题会阻塞后续设计

## 工作方式

借鉴 `brainstorming` 的方式推进，但收尾是规范文档而不是设计文档。

### 1. 先理解上下文

先读完成当前需求所必需的最小上下文：

- 用户当前描述
- 现有产品或流程说明
- 与当前范围直接相关的文档
- 用户或项目已经给出的路径、模板、章节要求

如果用户提供了已有草稿，把它当作待完善输入，而不是当作已定稿结论。

### 2. 先澄清，再动笔

像做需求访谈一样推进：

- 一次只问一个主题
- 优先问会影响范围和验收的关键问题
- 能用选择题时，优先给用户可选项
- 不要一口气抛出整张问卷

至少澄清这些方面：

1. 目标与用户
2. 功能范围
3. 非目标 / 范围外内容
4. 约束、依赖与外部边界
5. 验收标准
6. 边界情况与负路径

如果需求明显过大，先不要把所有子系统混在同一份规范里。先帮助用户拆出 2-3 种范围切分方式，说明取舍，并推荐本轮先写哪一份。

### 3. 必要时给出 2-3 种范围方案

当需求有多种合理切法时，先给出 2-3 种方案，再推荐一个：

- 最小可落地范围
- 业务完整范围
- 集成优先或风险优先范围

说明每种方案的取舍，再请用户确认采用哪一种，然后继续写规范。

### 4. 先对齐理解，再起草文档

在真正写文档前，先用简短摘要回放当前理解，至少覆盖：

- 目标
- 用户
- 范围
- 非目标
- 关键约束
- 成功标准

如果这些点还不稳定，就继续澄清，不要急着起草。

### 5. 起草规范文档

若用户或项目已经指定文档路径、命名方式或模板，优先遵循。

若没有指定，默认保存到：

- `docs/specs/YYYY-MM-DD-<topic>-spec.md`

若没有现成模板，使用以下最小结构：

```markdown
# <主题> 规范文档

## 背景与目标

## 用户与使用场景

## 范围

## 非目标

## 功能要求

## 非功能要求

## 约束与依赖

## 验收标准

## 边界情况与负路径

## 开放问题
```

写作要求：

- 需求写成可观察行为，不写实现方式
- 非功能要求尽量量化
- 范围外内容必须显式写出
- 开放问题要标清是否阻塞后续设计
- 若用户没给答案，不要擅自补全成既定事实

## 轻量自检

起草完成后，用一轮轻量自检判断文档是否真的可进入设计：

- 目标是否清楚
- 范围和非目标是否都写明
- 核心行为是否可观察、可测试
- 模糊词是否被量化、替换或删除
- 是否混入了接口、表结构、类、框架等设计决策
- 验收标准是否覆盖主流程
- 是否至少识别了一类负路径或边界情况
- 是否还有阻塞设计的开放问题

如果仍有阻塞项，不要假装已经准备好进入设计。明确告诉用户还需要补齐什么。

## 输出格式

完成时输出应包含：

```markdown
需求规范草稿已完成。

- 文档路径: `<spec-path>`
- 当前判断: `可进入设计` | `仍需继续澄清`
- 仍待确认:
  - <若无则写“无”>

如需，我可以继续根据这份规范进入设计阶段。
```

只有在阻塞性开放问题已消除，且核心范围具备可验证验收标准时，才能判断为 `可进入设计`。

## 反模式

- 用户刚说一个想法，就直接给架构方案
- 把头脑风暴笔记当成规范文档
- 范围内写得很多，但完全不写非目标
- 验收标准留成隐含理解
- 用 endpoint、table、class、queue、retry policy 之类词汇偷带设计
- 因为“功能很简单”就跳过澄清
- 一次把十几个问题都丢给用户

## 完成条件

只有在以下条件满足时，这个 skill 才算完成：

- 关键需求已被澄清到足以写成规范
- 规范文档草稿已落盘
- 已完成一轮轻量自检
- 已明确给出 `可进入设计` 或 `仍需继续澄清` 的判断
